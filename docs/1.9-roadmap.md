## What 1.8 shipped (so future readers know what's done)

The 1.8 release was a foundation refresh for design — no new features, all infrastructure work. Everything below already exists and works in the 1.8 codebase.

### Color system (foundation)

- **Brand color:** Muted Teal `#2A6B7C` light / `#2A7585` dark
- **Strict semantic contract** (each color has exactly one job):
  - `Color.Semantic.green` = positive dollar AMOUNT user receives or saves (refunds, tax savings, family gain). NOT strategic conclusions, eligibility, rates, or balances. *(Original strict rule said "refunds only"; relaxed to current rule per cross-validated review during TaxPlanningView migration. See spec §2.)*
  - `Color.Semantic.amber` = action required only (deadlines, missing input, IRMAA proximity). NEVER applied to dollar amounts themselves — only to deadline text and pills.
  - `Color.Semantic.red` = error / blocking only (form validation, crossed cliffs). NEVER for "tax owed" or negative amounts.
  - No yellow.
- **Three non-overlapping namespaces:** `Color.UI.*` / `Color.Semantic.*` / `Color.Chart.*`. Hard rule: `Color.Chart.*` never overlaps `Color.Semantic.*`.

### Component library

- `BrandButton` — 6 style variants (primary, secondary, tertiaryUtility, tertiaryForward, destructiveSecondary, destructivePrimary)
- `MetricCard` — top-stripe card with informational/actionRequired/error categories
- `InfoButton` — filled brand-teal `info.circle.fill` at 16pt, 24pt hit target
- `Badge` — refund/due/error/neutral inline status labels
- All in `RetireSmartIRA/Theme/Components/`

### Migrated views (15+ files)

DashboardView, TaxPlanningView (Scenarios), RMDCalculatorView, LegacyImpactView, SocialSecurityPlannerView (+ 5 sub-views), AccountsView, IncomeSourcesView, QuarterlyTaxView, SettingsView, RothConversionView, ScenarioChartsView, StateComparisonView, GuideView, ClickwrapView, SourcesReferencesView. All use tokens; zero raw `.foregroundStyle(.green)` etc. literals remain.

### Tests

- 21 WCAG AA contrast assertion tests (light + dark, every text/surface and semantic combination)
- ~15 component behavior XCTest tests
- Token unit tests (12+)
- All 670+ tests pass (637 pre-existing + 33+ new)

### What 1.8 did NOT include (deferred to 1.9)

- Snapshot testing infrastructure (deferred due to Swift Testing framework integration issues)
- Tooltip discoverability deeper UX work (visual fix only in 1.8)
- MetricCard component swaps in DashboardView/TaxPlanningView (token migration only; component swap declined per per-file judgment)
- Legacy Impact "3869%" headline rework (research-backed, deferred to 1.9 by user decision)
- Any new features (ACA, Medicare, contributions, AGI dashboard)

## The 5 spawned 1.9 tasks

All 5 are parked as Claude Code spawn-task chips. Each chip's prompt is self-contained — a fresh session can pick up any chip and have full context.

### Task 1 — Rework Legacy Impact ROI display

**Source of truth:** `docs/beta-feedback/2026-04-25-legacy-roi-research.md` (full research + §7 cross-validated synthesis)

Replace the misleading "3869%" headline in Legacy Impact with industry-standard metrics. Research is done across 3 independent sources (own agent + ChatGPT + Gemini) — all converged on the same recommendation. Implementation:

- New headline: absolute dollar gain + PV in today's dollars
- Three-metric scorecard: break-even year + annualized return + Vanguard BETR-style implied break-even tax rate
- Optional: "Tax Wedge" bar chart visualization
- Drop the apologetic "Exceptionally high" disclaimer; replace with educational tooltip

Methodology + display layer change only — underlying after-tax inheritance math is already correct.

**Estimated scope:** 1-2 weeks.

### Task 2 — Re-add snapshot testing

Bring back `swift-snapshot-testing` (or equivalent native Swift Testing approach if Apple has shipped it). Was deferred from 1.8 due to linker errors with Swift Testing framework integration in versions 1.18+. By the time 1.9 starts, the SPM ecosystem should have stabilized.

- Component snapshot tests (BrandButton, MetricCard, Badge, InfoButton) — original test code in 1.8 plan body
- Screen-level snapshot tests for ~9 top-level views in light + dark mode
- Use the `DemoProfile` (already exists, DEBUG only) as test fixture

**Estimated scope:** 1-2 days.

### Task 3 — MetricCard component sweep

Audit Dashboard / TaxPlanning / RMD / Legacy / Quarterly / SS / Accounts for ad-hoc card patterns that should be MetricCard. Decline-to-swap calls were correct for 1.8 (lower risk) but left technical debt: ad-hoc card patterns sit alongside the canonical component.

**Estimated scope:** 3-5 days.

### Task 4 — Tooltip discoverability UX initiative

Solve the deeper "users miss most info buttons" problem flagged by Ron Park. The 1.8 visual fix (filled brand-teal info button at 16pt) helped but didn't solve discovery. This is research + content audit + light implementation:

- Audit every tooltip; score by value
- Promote top 5-10 high-value tooltips to inline microcopy (always visible)
- Build a lightweight first-run tour
- Add tap telemetry on InfoButton

**Estimated scope:** 1-2 weeks.

### Task 5 — 1.9 features bundle (the big one)

The headline content of 1.9. Four features that all share AGI as the central planning lever:

1. **ACA Marketplace subsidy modeling** — biggest gap for pre-Medicare retirees
2. **Medicare plan-type awareness** — Original Medicare vs Medicare Advantage (different base premiums; same IRMAA)
3. **Pre-tax contribution levers** — 401(k), Traditional IRA, HSA contributions (CA + NJ tax HSA gains, watch state-specific handling)
4. **Reduce-AGI dashboard section** — group existing AGI-lowering levers (QCD, charity, stock donations) with new contribution levers under one headline

**Estimated scope:** 4-6 weeks. By far the largest 1.9 task. Possibly worth splitting into 1.9.0 (ACA + Medicare) and 1.9.1 (contributions + Reduce-AGI dashboard) if shipping cadence matters.

## Recommended execution order

This isn't strictly enforced — chips can be started in any order — but this sequence minimizes rework and gives each task the maximum benefit from preceding work:

1. **Re-add snapshot testing** (small, gives the rest visual regression cover)
2. **MetricCard component sweep** (small, finishes 1.8 cleanup; benefits from snapshot tests)
3. **Rework Legacy Impact ROI** (medium, well-researched, contained scope)
4. **Tooltip discoverability** (parallel-able with #5; mostly UX work)
5. **1.9 features bundle** (the big one — should be where most of 1.9's calendar time goes)

Total runway: ~8-12 weeks from "1.8 has merged to main" to "1.9 ready to submit."

## Cross-task coordination

- **Snapshot testing first** so all subsequent tasks get visual regression coverage
- **MetricCard sweep before features bundle** — when adding new dashboard sections in #5, use MetricCard from the start; don't accumulate new ad-hoc card patterns to clean up later
- **Tooltip discoverability + features bundle parallel** — the ACA cliff warnings and IRMAA proximity warnings should ship as inline microcopy from the start (per the tooltip task's promotion strategy), not as tooltip-only callouts
- **Legacy Impact rework is independent** of the others — can run anytime after snapshot testing lands
- **Demo profile carries forward** — `RetireSmartIRA/DemoProfile.swift` (DEBUG only, `-DemoProfile` launch arg) provides reproducible test fixtures and screenshot data for the entire 1.9 cycle

## Explicitly deferred to 1.10 / 2.0 (NOT 1.9 scope)

These came up during 1.8 brainstorm or beta feedback but are intentionally NOT in 1.9 scope. Don't get pulled into them.

- **Multi-year Roth optimization engine** (Boldin-style year-by-year optimal conversion ladder) — separate 2.0 feature requiring new planning engine
- **Brokerage account modeling** (info-only OR tax-modeled) — meaningful feature, separate task
- **HSA accounts as full account type** — only the contribution lever is in 1.9; full HSA balance modeling is a separate task
- **History / year-snapshots** (frozen historical plan data, "2025 Plan / 2026 Plan / 2027 Plan" tabs) — strong differentiator vs Boldin but separate 2.0 feature
- **Rental income / sale of property / crypto** — separate income types, deferred per Ron Park feedback
- **Multi-year ACA optimization** — 1.9 ships single-year ACA cliff awareness; multi-year ACA strategy is its own initiative
