# RetireSmartIRA 2.0 Scope

**Status:** Draft, awaiting approval
**Date:** 2026-05-01
**Strategic frame:** 1.9 features + 2.0 features ship as a **single App Store release tagged 2.0**. No public 1.9 release — internal `1.9-internal` checkpoint only. The version bump and release event happen once, with 1.9 + 2.0 work combined.

**Why this framing:** indie apps get only a handful of marketable release moments per year. Combining 1.9 (healthcare cost planning: ACA + Medicare + contribution levers + Reduce-AGI dashboard) with 2.0 (multi-year optimization + plan history + brokerage modeling) into a single 2.0 release captures editorial attention, press cycles, and re-engagement from existing users in a way two sequential releases would not.

---

## What 2.0 contains (in addition to 1.9)

### Headline features (drive the marketable narrative)

**1. Multi-year Roth optimization engine**

Boldin-style year-by-year optimal Roth conversion ladder. Computes the optimal conversion amount per year given:
- Tax bracket trajectory (current → RMD age → planning horizon)
- Market return assumptions (optimistic / average / pessimistic)
- IRMAA cliff awareness (don't optimize one year into a worse Medicare bracket)
- ACA cliff awareness (don't optimize one year into lost subsidies — pre-Medicare only)
- Withdrawal-order interactions (defer brokerage gains, accelerate Trad IRA, etc.)

This is the **headline 2.0 feature**. Justifies the version bump. Differentiates from competitors who only do single-year planning.

Architectural notes:
- New planning engine (separate from current single-year tax engine)
- Reads existing single-year tax engine's outputs as building blocks
- Returns a per-year recommendation: amount + reason + projected lifetime savings
- Should integrate with the new 1.9 Reduce-AGI dashboard (the multi-year recommendations show up there)

**2. Plan history / year snapshots**

Frozen historical plan data. UI shows tabs or a year-picker: "2025 Plan / 2026 Plan / 2027 Plan." Each year captures the state of the plan as of that year (income, accounts, scenario decisions). Once a year is "closed," its data freezes — future edits create the next year's plan instead.

This is a **strong differentiator vs Boldin**, which doesn't preserve historical plans well. Also serves a real user need (per Ron Park: "look back at my 2025 and 2026 plans when I start on 2027").

Architectural notes:
- Anchored on the `planYear` field already added in 1.7.2 (Ron fix #5)
- New persistence layer for per-year frozen snapshots
- New "Start planning {next year}" workflow that bumps `planYear` and freezes the prior year
- Per-year tax-config lookup so historical plans use historical tax rules

**3. ACA-as-Roth-constraint-engine** (Input 2's strategic insight)

Connects ACA modeling to Roth conversion guidance. Roth conversion suggestions become ACA-aware: *"Convert $48K this year — fills 24% bracket WITHOUT triggering ACA subsidy loss."* The cliff becomes part of the optimization fabric, not just a passive warning.

Small in code volume but high in strategic value. Connects 1.9's ACA work to 2.0's multi-year engine.

Architectural notes:
- Builds on 1.9's ACA cliff calculation (already exists by 2.0 start)
- Roth conversion suggestion logic gains a new constraint: "stay below ACA cliff if pre-Medicare and ACA enabled"
- Multi-year engine respects this constraint year-by-year

### Strong inclusions (extend 1.9's foundation)

**4. HSA full account modeling**

1.9 added HSA contribution levers. 2.0 adds full account modeling: balance tracking, distributions, growth projection, state-specific gain treatment for CA + NJ.

Why include: completes the HSA story. Without it, 1.9's HSA lever is conceptually orphaned (you can model contributions but not the resulting account balance over time).

Scope: medium. Extends existing `IRAAccount` model (or adds a new `HSAAccount` type), extends `AccountsManager`, adds CA/NJ-specific gain logic.

### Probable inclusions (smaller, complete the picture)

**5. Multi-year ACA optimization**

1.9 ships single-year ACA cliff awareness. 2.0 extends to multi-year: "Convert $48K this year and $52K next year — keeps you below the cliff both years and avoids triggering CSR loss in 2027."

Builds on multi-year Roth engine (#1) + ACA-as-Roth-constraint (#3). Mostly an extension of those, not a new system.

## What's NOT in 2.0 (defer to 2.1+ or later)

**Brokerage account modeling**
Taxable brokerage accounts (cap gains LT/ST, basis tracking, step-up at death, wash sale awareness, state-specific cap gains treatment). Significant scope — could be a 4-week feature on its own. Deferred to 2.1 to keep the 2.0 release focused on the multi-year planning narrative. **Withdrawal-order strategy modeling depends on this** and is also deferred to 2.1.

**Rental income / sale of property / crypto income types**
Deferred per Ron Park feedback (not blocking his current use). These are specialized tax treatments (rental QBI / depreciation; §121 home-sale exclusion; crypto cost-basis tracking) each significant on their own. Better as their own 2.1 / 2.2 sub-releases.

**App name rethink**
Ron noted: "App name undersells the breadth." True, but a name change requires careful App Store rebrand strategy (rating reset cost, search ranking impact). Separate strategic decision, not engineering. Track as a pre-3.0 consideration.

**Form 2210 annotation flow for Q4 Roth conversions**
Mentioned in Ron Park feedback as deferred to a Safe Harbor polish pass. Small, low-priority. 2.1 candidate.

**iOS / iPadOS snapshot test coverage**
Pass 1 + Pass 2 of snapshot testing land macOS-only. iOS coverage is a separate task. 2.1 candidate.

## Release plan

**Build sequence:**
1. **1.9 features bundle** (current brainstorm in flight): ACA + Medicare + contribution levers + Reduce-AGI dashboard. Tag internal `1.9-internal` checkpoint when done. **Do NOT submit to App Store.**
2. **2.0 features bundle**: multi-year Roth engine, plan history, ACA-as-Roth-constraint, HSA full account, brokerage accounts, withdrawal-order, multi-year ACA. Each gets its own brainstorm + spec + plan + execution.
3. **2.0 App Store submission**: combined 1.9 + 2.0 work. Single submission. Single review. Single release event.

**Estimated combined timeline:**
- 1.9 features: ~2 weeks (per earlier calibration, AI-assisted pace)
- 2.0 features: ~3-5 weeks (multi-year engine + plan history + ACA-Roth constraint + HSA full account; brokerage deferred to 2.1)
- Combined: **~5-7 weeks total** before App Store submission

**Marketing prep happens during build:**
- Screenshots showing multi-year Roth ladder, plan history, ACA cliff visualization
- Blog post / press release narrative: "RetireSmartIRA 2.0: now plans multi-year Roth strategies, healthcare cost optimization, and historical plan tracking"
- Beta tester (Ron + others) feedback before submission

## Approval gate

This scope is intentionally tighter than "everything deferred ever." Three criteria for inclusion:
1. **Marketable headline value** — feature appears in the 2.0 launch narrative
2. **Architectural fit with 1.9** — feature naturally extends rather than competes with 1.9 work
3. **Reasonable scope** — feature ships in 4-8 weeks combined with the others, not 6 months alone

If any item above doesn't meet all three, it's deferred. If you want to add an item, it should meet all three.
