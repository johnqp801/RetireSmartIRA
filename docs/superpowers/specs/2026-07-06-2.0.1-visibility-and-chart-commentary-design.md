# 2.0.1 — Senior Bonus Visibility + Chart Commentary — Design

**Date:** 2026-07-06
**Branch:** `2.0.1/visibility-and-commentary` (off `main`, which already carries the OBBBA senior-bonus itemization fix `f88966b`)
**Release:** 2.0.1 / build 59 (both platforms)

---

## Context

2.0.1 exists to ship the already-merged OBBBA senior-bonus itemization correctness fix (a 65+ itemizer under the MAGI phaseout was silently losing the $6,000/$12,000 deduction). Two low-risk, presentation-only enhancements ride along:

1. **Senior Bonus card visibility** — the explanatory "OBBBA Senior Bonus (age 65+)" card currently renders only on the Scenarios tab, buried under Charitable Contributions. Surface it on the Tax Summary tab where a 65+ user reviews their tax picture.
2. **Chart commentary** — an on-demand, tap-to-reveal popover that explains each of the five Multi-Year charts in plain language, generated deterministically from the chart's own model.

Neither part touches the tax/retirement engine. This is UI-only work plus the fix already on `main`.

## Non-goals (explicit scope boundaries)

- **No Multi-Year Senior Bonus card.** The card shows a current-year dollar amount; on a decades-long multi-year tab that is an impedance mismatch that needs its own design. Deferred.
- **No engine / tax-math changes.** The senior-bonus dollar figure is already correct on every tab; this is purely about explaining and surfacing it.
- **No LLM-generated text.** Commentary is deterministic and offline.
- **No always-visible caption line.** Commentary is opt-in via an ⓘ affordance (keeps an already-long tab uncluttered and leaves room for "how to read this chart" framing).
- Version bump / release-notes / archive-and-submit are release steps, handled separately from this feature work.

## Part 1 — Senior Bonus card on Tax Summary

**What:** Render `SeniorBonusDeductionCard()` on `DashboardView` (Tax Summary, ContentView tag 6), placed immediately after the Deduction section (~`DashboardView.swift:475`).

**Why it's safe:**
- `SeniorBonusDeductionCard` is a fully self-contained view. It reads shared `DataManager` state (`currentAge`, `filingStatus`, `enableSpouse`, `spouseCurrentAge`, `seniorBonusDeductionAmount`) and self-hides via `isApplicable` when nobody is 65+ (no placeholder when hidden).
- Tax Summary is a current-year view, and the card's amount is the current-year `seniorBonusDeductionAmount` — exact fit, no caveat needed.
- Zero new state, zero engine calls, no changes to the card itself.

**Placement detail:** the Deduction method/amount is displayed around `DashboardView.swift:439–475`. The card goes just below that block so the bonus is explained adjacent to the deduction it affects.

**Testing:**
- The card's internal applicability logic already exists. Add a lightweight presence check so the Dashboard placement can't silently regress (assert `SeniorBonusDeductionCard` is reachable in the Dashboard hierarchy for a 65+ profile, absent for under-65). Follow whatever view-presence testing pattern the existing suite uses; if none exists for Dashboard, assert on the card's own `isApplicable` gating via its accessibility identifier `seniorBonusDeductionCard` in the smallest feasible harness.

## Part 2 — Chart commentary (tap-to-reveal popover, all 5 charts)

Three pieces:

### 2a. `ChartInfoButton` (new reusable view, own file)

A small SwiftUI view: an ⓘ (`info.circle`) button that presents a `.popover` containing a title and a plain-language body. Built once, reused by all five chart views.

```
ChartInfoButton(title: String, body: String)
```

- Renders as a `.borderless` button with `info.circle` at `.caption`/`.footnote` weight, tinted secondary so it reads as a quiet affordance next to the chart title.
- Tapping toggles a `@State private var showing` that drives `.popover(isPresented:)`.
- Popover content: `Text(title).font(.headline)` + `Text(body).font(.callout)`, padded, `presentationCompactAdaptation(.popover)` so it behaves on iPhone (popover, not full-sheet) where appropriate.
- Accessibility: label "Explain this chart", the `body` as the accessibility value/hint.

### 2b. `commentary` on each chart model

Add a computed commentary to each of the five pure value-type models:

- `BalancesChart`
- `TaxImpactChart`
- `ConversionLadderChart`
- `ThresholdMapChart`
- `HeirFrontierChart`

Shape: a small struct returned by each model.

```
struct ChartCommentary: Equatable, Sendable {
    let title: String      // e.g. "Account balances over time"
    let body: String       // plain-language, adapts to the user's numbers
}
```

Each model exposes `var commentary: ChartCommentary`. The text is derived from the model's own data — the same philosophy as `PlanSummary.headline` and `PlanComparison.headline(units:)`:
- adapts to the user's actual numbers (a Roth-heavy user gets different phrasing than a taxable-light one),
- degrades gracefully on empty/degenerate inputs (empty path, no conversions, no IRMAA),
- no alarmist framing, no em dashes, matches existing on-tab voice.

Commentary content per chart (final wording refined during implementation, reviewed against real numbers):
- **Balances** — what the three lines are (Traditional / Roth / Taxable), that the shaded band (when shown) is growth-assumption sensitivity, not probability.
- **Tax Impact** — plan vs. doing-nothing yearly tax; where the plan front-loads tax to save later.
- **Conversion Ladder** — the modeled per-year conversion amounts and why they stop before the next threshold.
- **Threshold Map** — what the IRMAA/ACA and bracket lines mean and how the plan's income sits relative to them.
- **Heir Frontier** — the owner-tax vs. heir-tax trade-off and what moving along the curve costs/saves.

### 2c. Wire ⓘ into each `*ChartView` header

Each chart view currently starts its header with `Text("<title>").font(.headline)`. Wrap that title in an `HStack { Text(...); ChartInfoButton(title: model.commentary.title, body: model.commentary.body) }`. Five view edits: `BalancesChartView`, `TaxImpactChartView`, `ConversionLadderChartView`, `ThresholdMapChartView`, `HeirFrontierChartView`.

### Testing (Part 2)

The commentary functions are where auto-generated financial text can be subtly wrong, so this is where the test effort concentrates:
- Unit test each `commentary` for representative model states: empty path, conversions present vs. none, IRMAA present vs. absent, band present vs. absent, legacy/heir on vs. off.
- Assert the produced `title`/`body` contains the right claims and never the wrong ones (e.g., don't claim IRMAA when there is none; don't claim conversions when the path has none).
- `ChartInfoButton` is a trivial presentational view; covered indirectly. No dedicated snapshot test required.

## Architecture / isolation

- `ChartInfoButton` — one purpose (present an explanation popover), one well-defined interface (`title`, `body`), no dependencies beyond SwiftUI. Understandable and testable in isolation.
- `ChartCommentary` + per-model `commentary` — pure functions of the model. No view or engine dependency. Each is independently testable.
- Chart views gain one line in their header and are otherwise unchanged.
- `DashboardView` gains one card call. No new state.

## Data flow

Unchanged. Chart models are already built from `activePath` in `MultiYearPlanView`; `commentary` reads only fields those models already hold. The Senior Bonus card already sources everything from `DataManager`.

## Error handling / edge cases

- Under-65 profile: Senior Bonus card self-hides (existing behavior), no empty space.
- Empty / degenerate Multi-Year path: chart views already guard rendering; `commentary` returns a safe generic sentence for empty input.
- iPhone popover sizing: `presentationCompactAdaptation(.popover)` keeps the ⓘ a popover rather than a full-screen sheet.

## Release framing

Neutral, forward-looking per project rules (no "bug fix"/"honesty" framing). Release notes drafted separately with 2–3 wording options before submission. Build 59 both platforms.
