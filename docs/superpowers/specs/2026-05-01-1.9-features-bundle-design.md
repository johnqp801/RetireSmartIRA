## 1. Overview

This spec covers the four features comprising 1.9's headline "healthcare cost planning + AGI reduction" theme:

1. **ACA Marketplace subsidy modeling** — pre-Medicare users (age <65) see how AGI affects their ACA subsidies; cliff awareness for the post-2025 400% FPL threshold
2. **Medicare plan-type awareness** — users on Medicare see total household cost (Part B + Part D + Medigap or MA + IRMAA), per spouse, with mixed-household support
3. **Pre-tax contribution levers** — 401(k), Traditional IRA, HSA contribution sliders that reduce AGI; CA + NJ state-specific HSA handling
4. **Reduce-AGI dashboard section** — unified view that ties levers (existing + new) to their AGI impact and downstream consequences (taxes, IRMAA, ACA)

All four features share **AGI as the central planning lever**. The 1.9 features bundle is the foundation for 2.0's multi-year Roth optimization engine — 2.0 builds optimization on top of these single-year mechanics.

**Critical context:** the IRA-era enhanced ACA subsidies expired at end of 2025. The 400% FPL cliff is back for 2026. A poorly-timed Roth conversion crossing the cliff can trigger $10K-$20K in lost subsidies — the single largest financial landmine for the 60-64 pre-Medicare audience.

### Scope discipline

**In scope (1.9 features bundle):**
- ACA single-year subsidy modeling with cliff awareness + national-average Silver plan default + override
- Medicare plan-type cost display (Original vs Advantage), per-spouse, with default premium overrides
- Six pre-tax contribution levers (your+spouse × 401k/IRA/HSA), with limit + age + Medicare-eligibility validation
- Reduce-AGI dashboard section with cost-spike chart and cross-feature warnings
- IRMAA 2-year lookback awareness in scenario warnings
- Mixed-household handling for couples in different Medicare-status combinations
- Strongly-typed AGI variants (FederalAGI, ACAMAGI, IRMAAMAGI)
- Cross-feature `ScenarioWarningEngine` that surfaces ACA cliff + IRMAA tier crossings simultaneously

**Out of scope (deferred):**
- Multi-year Roth optimization (2.0 headline)
- Plan history / year snapshots (2.0)
- HSA full account modeling — balance tracking, growth, distributions (2.0; 1.9 only models contribution levers)
- Multi-year ACA optimization (2.0)
- Brokerage account modeling (2.1 per scope doc)
- Withdrawal-order strategy (2.1)
- ACA Original-vs-MA plan comparison tool (Medicare option B/C territory; 2.0)
- Form 1095-A reconciliation (tax-prep concern, not planning)
- Foreign earned income exclusion (audience mismatch)
- Medicaid eligibility (different problem space)
- iOS / iPadOS snapshot tests (2.1)

**Estimated scope:** ~1.5-2 weeks at AI-assisted pace, single PR on `1.9/features-bundle` branch.

## 2. Decisions log

Each decision below was made during the 2026-05-01 brainstorm. Captured here so a future agent picking up this spec doesn't re-litigate.

| # | Decision | Rationale |
|---|---|---|
| 1 | **All 4 features in one spec, ship as part of combined 2.0 release** | Per `docs/2.0-scope.md` strategic decision: single splashy App Store release vs incremental 1.9 + 2.0. 1.9 features locked-in via internal `1.9-internal` checkpoint, no public 1.9 release. |
| 2 | **ACA scope: simplified APTC with national-average benchmark + override + honest framing + cliff visualization + Roth-conversion warning** | Pure cliff-awareness (option A) is too weak for a 60-64 audience facing $10-20K subsidy losses. Full national APTC (option C) has data-maintenance burden indie app can't sustain. Hybrid B with optional local override gets directional dollar signal without ongoing data work. |
| 3 | **Medicare scope: display-only (option A) with per-spouse plan type and overrides** | Audience is sophisticated planners who already chose their plan, not consumers needing comparison. The Original-vs-MA comparison card (option B) is consumer-education for a once-in-a-lifetime decision. Plan-type cost optimization belongs in 2.0's multi-year engine. |
| 4 | **AGI/MAGI strongly typed** (`FederalAGI`, `ACAMAGI`, `IRMAAMAGI`) | Three AGI variants exist in the app's logic. Even if equal today, they will diverge. Strong typing prevents silent bugs and clarifies UI ("this number is used for X"). |
| 5 | **Data-driven ACA cliff** | Don't hardcode "400% FPL" in engine. Derive from applicable-figure table where subsidy=0. ACA policy is unstable (Congress could restore enhanced subsidies); JSON update should adapt the engine without code change. |
| 6 | **Per-spouse Medicare plan type** | Mixed households (one on Medicare, one pre-Medicare) require independent per-spouse plan-type fields. Joint MAGI still drives IRMAA, but each spouse has their own plan-type/cost. |
| 7 | **IRMAA 2-year lookback explicit** | The current code's "today's IRMAA from today's MAGI" is technically wrong (IRMAA is paid 2 years after the income year). Show both: `currentYearIRMAA` (today's premium, approximated from current MAGI as a proxy) + `projectedIRMAA` (premium 2 years out from this year's scenario decisions). Visible note explains the lookback. |
| 8 | **HSA combined-limit-only enforcement** (no per-spouse cap) | Original "cap each spouse at family/2 with override" creates artificial constraint + escape hatch. Cleaner: enforce only the joint family cap. Header shows "Combined HSA: $X / $8,550 family limit." IRS accepts any split. |
| 9 | **HSA gated on `medicarePlanType == .preMedicare`** | Medicare enrollment legally precludes HSA contributions (HDHP requirement). Simpler than month-by-month proration. Edge cases get tooltip. |
| 10 | **Catch-up contributions auto-applied based on age** | Age ≥50 → standard catch-up. Age 60-63 (only) → SECURE 2.0 super catch-up for 401(k). Age ≥64 → drops back to standard catch-up. Age ≥55 → HSA catch-up. User sees the higher limit but can override. |
| 11 | **Limit validation soft (amber warning, not blocking)** | Users may model future-year limits or have special circumstances. Warning informs without preventing entry. |
| 12 | **State-specific HSA via new `StateTaxConfig` property** | CA + NJ tax HSA contributions/gains as ordinary income. Federal AGI reduction still applies, but state AGI doesn't. New property: `hsaContributionsTaxableForState: Bool`. |
| 13 | **Cost-spike chart: two stacked panels with explicit timing** | Single Y-axis mixing this-year costs (taxes, ACA) with 2-years-out costs (IRMAA) creates panic-misinterpretation risk ("Do I owe $12K this April?"). Stacked panels with timing labels prevent confusion. |
| 14 | **`ScenarioWarningEngine` as new file** | Cross-feature warning logic (ACA cliff + IRMAA crossing + bracket crossing + widow risk) needs centralized engine, not scattered per-feature warning code. Top-3 by `dollarImpactPerYear` displayed; "+N more" for the rest. |
| 15 | **Pre-Medicare projection band for ages 63-64** | The 2-year IRMAA lookback means today's decisions affect Medicare costs at 65+ start. Surface this proactively in the dashboard, not only when sliders are touched. |

### 3.2 — Strongly-typed AGI variants

```swift
struct FederalAGI {
    let value: Double
}

struct ACAMAGI {
    let value: Double
    // = FederalAGI + tax-exempt interest + non-taxable Social Security
}

struct IRMAAMAGI {
    let value: Double
    // = FederalAGI + tax-exempt interest
}

extension DataManager {
    var federalAGI: FederalAGI {
        FederalAGI(value: scenarioGrossIncome - totalAboveTheLineDeductions)
    }
    var acaMAGI: ACAMAGI {
        ACAMAGI(value: federalAGI.value + taxExemptInterest + nonTaxableSocialSecurity)
    }
    var irmaaMAGI: IRMAAMAGI {
        IRMAAMAGI(value: federalAGI.value + taxExemptInterest)
    }
}
```

`ACASubsidyEngine` takes `ACAMAGI` parameter (not `Double`). `MedicareCostEngine` takes `IRMAAMAGI`. Compile-time prevents passing the wrong type. Existing IRMAA-using code is migrated to use the typed wrapper.

### 3.3 — AGI formula refactor

**Today:**
```swift
estimatedAGI = scenarioGrossIncome
```

**After 1.9:**
```swift
totalAboveTheLineDeductions =
    yourTraditional401kContribution + spouseTraditional401kContribution +
    yourTraditionalIRAContribution + spouseTraditionalIRAContribution +
    yourHSAContribution + spouseHSAContribution

federalAGI.value = scenarioGrossIncome - totalAboveTheLineDeductions
```

Existing scenario levers (Roth conversion, extra withdrawal, QCD, charitable donations) continue operating on `scenarioGrossIncome` — they're not above-the-line deductions in tax law. New contribution levers operate as above-the-line deductions on top of that.

## 4. Per-feature scope

### 4.1 — ACA Marketplace subsidy modeling

**Engine: `ACASubsidyEngine.swift`**

```swift
enum ACASubsidyEngine {
    static func calculateSubsidy(
        acaMAGI: ACAMAGI,
        householdSize: Int,
        benchmarkSilverPlanAnnualPremium: Double,
        primaryAge: Int,
        spouseAge: Int?,
        config: TaxYearConfig
    ) -> ACASubsidyResult
}
```

**Logic:**
1. Compute FPL for `householdSize` (with AK/HI multipliers if applicable).
2. Compute `fplPercent = acaMAGI.value / fplAmount × 100`.
3. Look up `applicable_figure` for `fplPercent` (interpolate between table rows).
4. Detect cliff: `cliffPoint = applicableFigures.first { $0.applicableFigure >= 1.0 }`. If `fplPercent >= cliffPoint.fplPercent`, subsidy is 0; `isOverCliff = true`.
5. Else: `expectedContribution = acaMAGI.value × applicableFigure`; `annualPremiumAssistance = max(0, benchmarkSilverPlan - expectedContribution)`.
6. `dollarsToCliff = cliffPoint.fplPercent × fplAmount / 100 - acaMAGI.value` (positive if below).

**Override field (Settings tab):** "Local benchmark Silver plan monthly premium" — converts to annual for engine input. Default uses national average from JSON.

**Cliff visualization:** banner in Reduce-AGI dashboard when `dollarsToCliff < 5000` with format "Approaching ACA subsidy cliff: $X to threshold." Plus integrated into cost-spike chart.

**Roth-conversion ACA-cliff warning:** inline amber band on Roth conversion slider when proposed conversion would push MAGI across cliff. Format: *"Crosses ACA cliff at MAGI $84,200. Estimated $14,400/yr subsidy lost."* (Specific numbers from current scenario state.)

**Gating:** ACA section appears only if (`enableACAModeling == true` AND at least one spouse `<65`). Auto-hides with explanatory banner when both spouses age into Medicare.

### 4.2 — Medicare plan-type awareness

**Engine: `MedicareCostEngine.swift`**

```swift
enum MedicareCostEngine {
    static func computeCostForSpouse(
        planType: MedicarePlanType,
        irmaaMAGI: IRMAAMAGI,
        partBOverride: Double?,
        partDOverride: Double?,
        medigapOverride: Double?,
        advantageOverride: Double?,
        filingStatus: FilingStatus,
        config: TaxYearConfig
    ) -> MedicareCostBreakdown
}
```

**Logic:**
1. If `planType == .preMedicare`, return zero cost (with `irmaaTier = -1`).
2. Compute IRMAA tier and surcharges from `irmaaMAGI` and `filingStatus` (uses existing `TaxCalculationEngine.calculateIRMAA` path with typed input).
3. `partB = partBOverride ?? config.medicare.partBStandardMonthly` + IRMAA Part B surcharge.
4. `partD = partDOverride ?? config.medicare.partDAvgMonthly` + IRMAA Part D surcharge.
5. If `planType == .originalMedicare`: include `medigap = medigapOverride ?? config.medicare.medigapAvgMonthly`.
6. If `planType == .medicareAdvantage`: include `advantagePremium = advantageOverride ?? config.medicare.advantageAvgMonthly` (typically $0 for many MA plans, with Part B + Part D folded in or partially separate; default value is conservative average).
7. Sum to monthly total.

**Mixed household:** the engine is called separately for each spouse with the same `irmaaMAGI` (joint, since IRMAA brackets use joint MAGI for MFJ filers). Per-spouse `MedicareCostBreakdown` returned independently. Household total is sum.

**IRMAA 2-year lookback:**
- `currentYearIRMAA` — uses today's `irmaaMAGI` as approximation for what user is paying now (since we don't track 2024 MAGI). Tooltip: "IRMAA premiums are officially based on income from 2 years ago. This estimate uses your current income as a proxy."
- `projectedIRMAA` — uses today's scenario-decision-influenced `irmaaMAGI` to compute what user will pay in 2 years.
- UI labels: "Current Medicare premium (estimated)" vs "Projected 2028 premium impact." Visible info banner at section top: *"Medicare IRMAA premiums are based on your income from 2 years prior. Decisions you make today affect your premiums in 2 years."*

**Display on Tax Summary tab (DashboardView):**
- Existing IRMAA tier chart unchanged.
- New: total household Medicare cost broken down per spouse, with IRMAA contribution highlighted.
- For pre-Medicare users 63-64: projection band shows expected costs at age 65 based on current scenario.

**Limit computation:**

```swift
extension DataManager {
    func four01kLimit(for owner: Owner) -> Double {
        let age = owner == .primary ? currentAge : spouseCurrentAge
        let limits = config.contribution_limits_2026.fourOhOneK
        if age < 50 { return limits.base }
        if age >= 60 && age <= 63 { return limits.base + limits.catchup_age_60_to_63 }
        return limits.base + limits.catchup_age_50_to_59  // covers 50-59 and 64+
    }
    
    func iraLimit(for owner: Owner) -> Double {
        let age = owner == .primary ? currentAge : spouseCurrentAge
        let limits = config.contribution_limits_2026.ira
        return age >= 50 ? limits.base + limits.catchup_age_50_plus : limits.base
    }
    
    func hsaCombinedLimit() -> Double {
        // Family limit applies if either spouse has family HDHP coverage.
        let limits = config.contribution_limits_2026.hsa
        let baseCap = enableSpouse ? limits.family : limits.self_only
        let catchupYou = currentAge >= 55 ? limits.catchup_age_55_plus : 0
        let catchupSpouse = (enableSpouse && spouseCurrentAge >= 55) ? limits.catchup_age_55_plus : 0
        return baseCap + catchupYou + catchupSpouse
    }
}
```

## 5. Cross-feature: `ScenarioWarningEngine`

**Engine: `ScenarioWarningEngine.swift`**

```swift
enum ScenarioWarningEngine {
    static func warningsFor(
        scenarioState: ScenarioState,
        federalAGI: FederalAGI,
        acaMAGI: ACAMAGI,
        irmaaMAGI: IRMAAMAGI,
        primaryAge: Int,
        spouseAge: Int?,
        filingStatus: FilingStatus,
        enableACAModeling: Bool,
        config: TaxYearConfig
    ) -> [ScenarioWarning]
}
```

**Logic:** engine evaluates each warning category in order, returning all that fire. DataManager exposes `activeScenarioWarnings: [ScenarioWarning]` as a computed property.

**Warning categories:**
- `acaCliff` (timing: currentYear): MAGI ≥ cliff threshold AND any spouse pre-Medicare AND `enableACAModeling`. `dollarImpactPerYear` = previous APTC amount (subsidy lost).
- `acaApproaching` (currentYear): MAGI < cliff threshold AND `dollarsToCliff < 5000`.
- `irmaaTierCrossing` (twoYearsOut): MAGI just crossed an IRMAA tier boundary AND any spouse on Medicare or age 63+. `dollarImpactPerYear` = annualized surcharge increase.
- `irmaaApproaching` (twoYearsOut): within $10K of next IRMAA tier.
- `niitCrossing` (currentYear): existing logic.
- `bracketCrossing` (currentYear): existing logic.
- `widowBracketJump` (currentYear): existing logic in `LegacyImpactView`.

**UI display:**
- **Inline on slider being edited**: top 1-2 most-severe warnings (by `dollarImpactPerYear`) as amber band immediately below the slider control.
- **In Reduce-AGI dashboard**: ALL active warnings as a list, grouped by `Timing` (currentYear, twoYearsOut). Top 3 by impact shown; "+N more" expandable.
- **Inline on Roth conversion slider specifically**: the ACA cliff warning gets pinned position because that lever is the most likely cliff trigger for the 60-64 audience.

**Unit tests (XCTest, ~40-60 new tests):**
- `ACASubsidyEngineTests` — FPL boundary cases, applicable-figure interpolation, cliff detection (cliff-present and cliff-absent scenarios), household-size variations 1-8, AK/HI multipliers, override application
- `MedicareCostEngineTests` — per-spouse cost breakdown, mixed-household joint MAGI handling, IRMAA tier 0-5 crossings, Medigap vs MA distinct paths, override application, pre-Medicare returns zero
- `ContributionLeverTests` — limit computation per age (50, 59, 60, 63, 64, 65, 70), super-catch-up bound (60-63 only), HSA family combined-limit, HSA Medicare-eligibility gating
- `StateTaxHSATests` — CA and NJ HSA add-back; non-CA/NJ no add-back; parameterized across all 50 states
- `AGITypesTests` — `FederalAGI`, `ACAMAGI`, `IRMAAMAGI` distinctness; explicit conversion between them
- `ScenarioWarningEngineTests` — cliff detection, approaching-threshold detection, dual-warning scenarios (Roth conversion triggers both ACA + IRMAA), warning prioritization (top 3 by impact), all 7 categories

**Integration tests (XCTest, ~10-15 new tests):**
- End-to-end: user with `enableACAModeling=true`, age 62, MAGI $83K → ACA subsidy $X, near cliff. Roth conversion of $5K → MAGI $88K → subsidy $0, ACA cliff warning fires, IRMAA 2-year warning fires.

## 7. Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| ACA policy change mid-development (Congress restores enhanced subsidies) | Medium | Data-driven cliff in `tax-2026.json`. Emergency response = JSON update, not code change. |
| 2026 IRS contribution limits change | Low | All limits in `tax-2026.json`. Update JSON, no code change. |
| 2026 Medicare premium values not yet published when 1.9 ships | High | Use 2025 values as placeholders; flag in JSON. Override mechanism lets users correct. CMS publishes typically October. |
| User misinterprets cost-spike chart timing | Low | Two stacked panels with explicit timing labels. |
| User confused by AGI vs ACA-MAGI vs IRMAA-MAGI | Low | Strongly-typed structs prevent silent bugs. UI shows ONE of these per section context. |
| Mixed-household IRMAA computation error (using primary's MAGI instead of joint) | Medium | Engine accepts joint MAGI only; integration tests cover explicitly. |
| HSA family-limit edge case (one spouse >55, other not) | Low | Combined-limit-only approach handles naturally — total enforced, individual limits not. |
| Non-CA/NJ state with HSA quirks I haven't tracked | Medium | Cross-check `hsaContributionsTaxableForState` against authoritative source (FSA Store annual guide or BenefitFocus) before merge. |
| Scenarios with both ACA cliff + IRMAA crossing flood UI with warnings | Low | Top-3 by impact, "+N more" expansion. Avoid noise. |
| User panics when cost-spike chart shows $12K spike, thinks tax bill | Medium | Two-stacked-panel design prevents conflation. Plus warning headlines explicitly carry timing ("this year" vs "in 2028"). |
| `ScenarioWarningEngine` returns warnings on every slider tick, performance hit | Low | Pure-function engine; warnings recomputed only when scenario state changes (existing reactive pattern). Profiling confirms no issue expected at this scale (~7 warning categories × ~10 scenario state changes per second max). |
