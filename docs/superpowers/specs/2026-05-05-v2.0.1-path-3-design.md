# V2.0.1 Path 3 — Tax Planning UI Polish Design

**Goal:** Polish the V2.0 Tax Planning view from "powerful expert tool with kitchen-sink right pane" to "competitively-positioned consumer product with crystal-clear cause/effect framing." Restore the spec's delta framing in the hero stat, eliminate alarm fatigue in the year list, kill the embedded V1.x app on the right side, and re-home long-horizon analytics behind contextual triggers.

**Architecture:** Twelve incremental changes layered on top of the merged 2.0 branch. No engine changes — the optimizer is stable at 951 tests. All work is in the SwiftUI presentation layer plus one additive computation in `MultiYearStrategyManager` (the engine baseline). New compact gauge views replace the kitchen-sink embedded `ScenarioBuilderView` + `DashboardView` in the Year 1 right pane. Long-horizon Legacy/Wealth charts move into a dedicated Heir Tax pill sheet. Conversion Opportunity Window is promoted from buried text to a first-class insight callout banner.

**Tech Stack:** SwiftUI on macOS / iOS / iPadOS. XCTest for tests. Build target via `xcodebuild -project RetireSmartIRA.xcodeproj -scheme RetireSmartIRA`.

---

## Why this work, and why now

V2.0 shipped with three usability problems that emerged in real-device testing on macOS:

1. **Hero stat regression.** The original spec called for delta framing ("$487K → $312K"). The shipped UI shows a bare "$5M LIFETIME TAX" because we couldn't credibly compute a baseline at the time. Without a comparison, the largest font on screen reads as a terrifying bill, not a victory. Every competitor — Boldin, Holistiplan, ProjectionLab, RightCapital — uses delta framing. Our omission makes us look inferior on the headline metric even though our optimization engine is genuinely better.

2. **Right-pane kitchen sink.** Year 1 detail shows `Year1QuickEditor` plus the entire `ScenarioBuilderView` + `DashboardView` embedded below. Users see duplicate controls (Roth slider at top, Roth conversion card below), 30-year content inside the 1-year pane (Family Wealth chart, Legacy Impact), and a scrolling junk drawer. The "telescope/microscope" architecture works conceptually but is broken in execution.

3. **Year list alarm fatigue.** Twenty-five rows from 2026–2050 each badge "⚠ IRMAA T4." Users learn to ignore the warning column entirely.

The competitive context matters: a retiree comparing RetireSmart IRA to Boldin sees Boldin's "Save $400K with Roth conversions!" hero next to ours and concludes Boldin is the smarter tool. We have the better engine; we just aren't surfacing it.

## Decision Log

Eight architectural decisions were made via brainstorm dialogue. Each is locked.

### 1. Hero Stat — Real Baseline + Ledger Layout

**Decision:** Restore the delta framing using a real engine-computed baseline. Layout: savings is the hero (large green number), supporting numbers stacked as a clean ledger.

```
[YOUR ENGINE-OPTIMIZED PLAN SAVES]      ✓ Matches engine plan
$400K
in lifetime tax over 30 years

┌──────────────────────────────────┐
│ Baseline Path           $5.4M    │
│ Your Plan               $5.0M    │
└──────────────────────────────────┘

Baseline Path: no Roth conversions, no QCDs.
RMDs and SS still claimed at your planned ages.

Seeks to minimize lifetime tax including a 22% tax on
assets left to heirs. Accepts IRMAA or ACA cliffs only
when they save more than they cost.
```

The "Baseline Path" is computed by calling `ProjectionEngine.project(inputs:assumptions:actionsPerYear: [:])` — a single forward pass with no Roth conversions and no QCDs (RMDs still imposed automatically, SS at planned ages). Cost: ~5ms per recompute. Honest label: "Baseline Path" with explicit definition footnote.

The off-plan badge ("✓ Matches engine plan" or "⚠ Different from engine") moves to the top-right of the hero card, replacing the prior placement.

**Why:** Competitive parity on the headline metric, plus our engine quality becomes legible. The savings figure is honest because it's a real computation, not a 1.5× placeholder. The footnote prevents users from misreading "Baseline" as "if you do absolutely nothing" — it accurately describes what the comparison includes (RMDs + SS at planned ages).

### 2. Year List — Expandable Transition Groups

**Decision:** Show transition years as full rows; collapse identical-tier runs into tap-to-expand summary rows.

```
2026 │ ⚠ Enters IRMAA T4              │ $110K
▸ 2027–2040 · stays in T4 (14 years)  │ $110K–$248K
2041 │ ⚠ Crosses into T5              │ $253K
▸ 2042 · T5                           │ $256K
2043 │ ↓ Drops back to T4             │ $234K
▸ 2044–2050 · stays in T4 (7 years)   │ $172K–$211K
```

Tapping a `▸` row expands to the full per-year list for that range; it collapses again on tap. Year 1 (current year) is always a full row regardless of transition status. Transition badges: orange `⚠` for "enters worse tier", green `↓` for "improves." Tax column shows the year range as `min–max` for collapsed groups.

**Why:** Eliminates the visual noise of 25 identical "IRMAA T4" badges while preserving access to per-year detail when wanted.

### 3. Year 1 Right Pane — Tax Position Panel

**Decision:** Below `Year1QuickEditor` (sliders + "This Year's Impact" strip), add a single "Tax Position" panel containing visual gauges for Federal bracket and IRMAA tier, plus text-only secondary indicators for State and NIIT.

```
┌── YOUR TAX POSITION THIS YEAR ─────────────────────┐
│ Federal bracket          35% · room to convert     │
│  [bracket gradient bar with current marker]        │   $254K more
│                                                     │
│ IRMAA tier               ⚠ Tier 4 · $113K cushion  │
│  [tier strip with current tier outlined]            │   before T5
│                                                     │
│ ─────────────────────────────────────────────────   │
│ State (CA): 9.3%       NIIT: $5.9K/yr              │
└────────────────────────────────────────────────────┘
```

The panel is ~190px tall. Federal and IRMAA get full visual gauges because they drive 90% of conversion decisions. State is continuous (no cliffs) and NIIT is binary — both work fine as text.

**Why:** Replaces the embedded `ScenarioBuilderView` with a focused "microscope" view. Visual gauges only where cliffs/thresholds matter for decision-making.

### 4. Heir Tax Popover — Sheet With Full Detail

**Decision:** Tapping the Heir Tax Rate pill opens a sheet (on both macOS and iPhone for consistency) containing the rate stepper, a Legacy Impact summary, and a compact Family Wealth chart.

```
Heir Tax Rate                                  [Done]
──────────────────────────────────────────────
Higher rate → more aggressive conversions
recommended today.

  [-]      22%      [+]

LEGACY IMPACT
Cost today: +$128K  →  Family gain: +$3.4M
Net family gain: $3,364,386 (a 2,631% return on
$128K paid today).

HOW ROTH CONVERSIONS GROW FAMILY WEALTH
[line chart: Roth solid vs Traditional dashed]
Roth (solid) overtakes Traditional (dashed) at year 13.
```

Sheet on both platforms because the content (chart + impact widget) doesn't fit comfortably in a transient popover.

**Why:** This is where the long-horizon Legacy Impact and Family Wealth charts now live (Gemini's "elegant move"). The pill is the natural anchor — tapping changes the assumption that drives those charts. Cause and effect tied together. Removes the charts from the kitchen-sink right pane.

### 5. Trade-offs Rewrite — Templated WHY + Part B/D Collapse

**Decision:** Two changes to `TradeOffsAcceptedCard`:

**5a. Plain-English language with templated WHY.** Each row gets a one-line title (WHAT happened) and a small italic subtitle (WHY accepted, templated by constraint type).

```
2026 — Crossed into 24% bracket               $14K
   Accepted to fit larger Roth conversions
   before RMDs raise your bracket later.

2027 — Hit IRMAA Tier 4                       $12K
   Accepted because conversion savings exceed
   the extra Medicare cost.
```

The "WHY" templates are not engine-derived — they are constraint-type-specific explanations that hold true for any time the engine accepts that constraint. Templates:
- **Bracket overrun:** "Accepted to fit larger Roth conversions before RMDs raise your bracket later."
- **IRMAA tier hit:** "Accepted because conversion savings exceed the extra Medicare cost."
- **ACA cliff:** "Accepted because long-term tax savings exceed the lost subsidy."

**5b. Collapse IRMAA Part B + Part D.** Today the same year shows two duplicate IRMAA rows (one for each Medicare program). For display, sum them into a single per-year line. Underlying engine data (`ConstraintHit`) is unchanged; the synthesizer does the dedup.

**Why:** The current "12% → 24% bracket overrun" reads like an error. "Crossed into 24% bracket" reads like a strategic choice. Duplicate rows look like a bug. Both fixes are pure presentation — no engine changes.

### 6. Scenario Detail — Overflow Menu + Conversion Banner

**Decision:** Replace the prior plan's "Full scenario detail →" button with a `⋯` overflow menu in the right pane toolbar. Promote the Conversion Opportunity Window from buried text to a first-class insight callout banner.

**Overflow menu (top-trailing of right pane):**
- Export CPA Briefing (`square.and.arrow.up` icon) — opens the existing Export PDF flow
- Tax Strategy Guide (`graduationcap` icon) — opens a new curated sheet containing only the educational cards from `ScenarioBuilderView` (Roth conversion explanation, QCD primer, IRMAA primer)

**Conversion Opportunity Window banner** lives in `CalloutBannersHost` alongside SS Nudge and Widow Penalty. Triggered when `years_before_first_RMD > 0`. Dismiss key: `"conversion-window-{years_before_rmd}"`. Impact tier `.moderate`.

**Why:** A "kitchen-sink" sheet would undermine the polish work. A `⋯` menu is the native pattern for utility actions (Export) and secondary content (Education). The Opportunity Window is the most persuasive trigger for Roth conversions in the entire app — burying it in a drawer would lose the value. Banner placement is consistent with other actionable insights.

### 7. Action Items — Sticky Banner With Dynamic Visibility

**Decision:** Add a sticky banner at the top of the Year 1 right pane (above sliders) summarizing this year's required actions. The banner is hidden entirely when there are no non-zero actions.

```
⏰  2 ACTIONS DUE BY DEC 31, 2026                    [View all →]
    Roth $341K · Stock donation $50K
```

"View all →" opens a sheet listing the full action items (deadlines, dollar amounts, custodian instructions). Each item has a checkbox the user can tick (persisted via `@AppStorage` or similar lightweight state).

**Visibility rule:** Banner appears only when at least one of (Roth conversion, QCD, stock donation, RMD) for the current year has a non-zero amount. If user overrides everything to $0, banner disappears.

**Why:** Action items are deeply tied to this year's slider decisions — adjust Roth, action item amount updates, you see what to do. Sticky placement keeps cause and effect adjacent. Dynamic visibility prevents "0 Actions Due" from cluttering the UI when irrelevant.

### 8. Gauge Implementation — Build New Compact

**Decision:** Build new compact gauge views from scratch in dedicated files. Do not extract or refactor the existing gauges inside `ScenarioBuilderView`.

- `RetireSmartIRA/CompactBracketGauge.swift` — Federal bracket bar with position marker
- `RetireSmartIRA/CompactIRMAAGauge.swift` — Tier strip with current tier outlined
- `RetireSmartIRA/TaxPositionPanel.swift` — Container assembling the gauges + State/NIIT footer

After other decisions land, `ScenarioBuilderView` becomes orphan code (the only reachable paths into it are the educational cards via "Tax Strategy Guide" and the Export PDF action — both of which can be served from a curated subset). The big V1.x gauges become unreachable dead code in V2.0.1.

**Why:** The "duplication" is against effectively-dead code we'll delete in V2.1. Surgery on a stable 4000-line file during a polish release is the wrong risk/reward. New compact gauges match the agreed visual design, are smaller and simpler than the V1.x versions, and let `ScenarioBuilderView` retire gracefully.

## Data Flow

### Baseline computation

`MultiYearStrategyManager.performCompute()` currently produces two results: `engineOptimalResult` (no Year 1 overrides) and `currentResult` (with overrides). It will produce a third:

```swift
@Published private(set) var baselineResult: MultiYearStrategyResult?

// inside performCompute(), after building inputs:
let baselineActions: [Int: [TaxAction]] = [:]  // empty = no Roth, no QCD
let baselineProjection = await Task.detached(priority: .userInitiated) {
    ProjectionEngine.project(
        inputs: optimalInputs,  // same static inputs as the optimal run
        assumptions: assumptions,
        actionsPerYear: baselineActions
    )
}.value
self.baselineResult = baselineProjection
```

Because `ProjectionEngine.project()` is a pure forward pass (~5ms vs the ~6,300 iterations of the optimization loop), this adds negligible time to recompute. It runs once per static-input change (i.e., on `assumptionsChanged` and `appLaunch`, not on `overridesChanged`) — same trigger condition as `engineOptimalResult`.

### Hero stat data flow

```
manager.baselineResult.lifetimeTaxFromRecommendedPath  ← Baseline Path number
manager.engineOptimalResult.lifetimeTaxFromRecommendedPath  ← Your Plan number (when on-plan)
manager.currentResult.lifetimeTaxFromRecommendedPath  ← Your Plan number (when off-plan, shows user's actual path)
                                                       Savings = Baseline - Your Plan
```

When off-plan, "Your Plan" reflects the user's overridden path — so the savings shown is what they're *actually* getting, not what the engine recommends. This is honest. The OffPlanIndicator separately shows "different from engine" with the option to reset.

## Edge Cases

- **No traditional IRA balance.** Baseline path equals optimal path (nothing to convert). Savings = $0. Hero shows "$0" savings — display rule: when savings < $1,000, show "Your plan is already optimal" instead of "$0".
- **All Year 1 overrides at $0.** Action Items banner hidden entirely (decision 7).
- **No SS claim age data.** SS Nudge banner already handles this; no new logic.
- **Locked pane (assumptionsConfirmed = false).** All new components are inside the macro/Year 1 panes that are already covered by the locked-pane overlay. No additional work.
- **Recompute in progress.** Baseline result follows the same `isComputing` lifecycle as optimal — stale-state overlay covers it.
- **User has no QCDs eligible.** Baseline still excludes QCDs (would-be zeros anyway). Footnote text remains accurate.

## Testing Approach

Pure-logic units (TDD strictly):
- Year list transition detection — input: array of YearRecommendation with tier values, expected output: list of `[FullRow | GroupedRow]`. Test cases: no transitions, all transitions, transitions at boundaries, single-year groups, repeated-tier runs.
- Trade-off WHY template selection — input: `ConstraintHit`, expected: correct template string.
- Part B/D dedup — input: list of `ConstraintHit` with duplicate years/tiers, expected: deduped list with summed costs.
- Action Items visibility logic — input: data manager state, expected: visible bool.
- Baseline computation correctness — input: assumptions with known traditional balance, expected: baseline tax > optimal tax.

SwiftUI views (build-and-construct tests, not visual snapshot tests — no snapshot library yet):
- Each new view instantiates without crashing under reasonable inputs.
- Hero stat handles all combinations of (baseline nil, optimal nil, current nil).
- Action Items banner respects visibility rule.

Integration:
- Full flow: open Tax Planning, verify Tax Position panel renders, tap Heir Tax pill → sheet opens with stepper + chart, change rate → recompute fires, sheet dismisses → hero updates.

Performance:
- Baseline computation adds ≤ 50ms to total recompute time. Test in `MultiYearStrategyManagerPerfTests`.
