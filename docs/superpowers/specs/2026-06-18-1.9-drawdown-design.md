## 1. Overview

A new **Retirement Drawdown Projection** in the RMD Calculator tab. Today the app grows IRA balances and only reduces them once RMDs begin at 73/75, so a pre-RMD retiree sees balances that climb unrealistically (the gap Tim Lucas identified). This feature lets the user model actually living off their accounts before and through RMD age, shows the balance drawn down year by year to ~life expectancy, and flags when projected income heads into IRMAA territory.

**Scope is deliberately contained.** This is the single-account-class drawdown *visualization*, not the multi-bucket tax-aware decumulation engine (that is 2.0/2.1). It must not entangle with the single-year Scenarios/Tax Summary tabs.

### In scope (V1.9 drawdown)
- Two withdrawal input modes (B: spending gap; C: withdrawal rate), user-toggled.
- Year-by-year balance drawdown for Traditional IRA + 401(k) aggregate, to a 40-year horizon.
- Balance-over-time chart (clean line default; expandable cash-source bars).
- Medium IRMAA overlay: an approximate projected-income line flagged against the inflated tier-1 threshold.
- Single inflation input; nominal display with an optional today's-dollars toggle.
- Propagation of drawdowns into the Legacy/estate projection (decision (b), 2026-06-18).

### Out of scope (deferred to 2.0 / 2.1)
- Multi-bucket withdrawal sequencing (Roth, brokerage), pro-rata-vs-sequential ordering, withdrawal-order optimization, Roth-conversion optimization — **2.0/2.1**.
- Brokerage / HSA full-account modeling — **2.0/2.1**.
- Full per-year tax computation, SS taxability mechanics, AGI assembly behind the IRMAA flag — **2.0** (V1.9 uses an explicit approximation).
- Any reuse of the stranded `2.0/*` `ProjectionEngine` — **forbidden** (those branches are ~190 commits behind main; see 2.0 audit). V1.9 builds its own small standalone projector on current main.
- Writing drawdown state into Scenarios or Tax Summary — **never**; this feature is display-only there.

## 2. Decisions log (from the 2026-06-18 brainstorm)

1. **Input modes:** ship **both B (spending gap) and C (withdrawal rate)** with a toggle.
2. **Mode B guaranteed income:** pulled **time-varying** from the Social Security tab (claiming age + amount) and Income & Deductions (pension), at **gross/face value** (no SS-taxability or AGI mechanics in V1.9).
3. **Chart:** **A+** — clean balance line as default, cash-source stacked bars as an expand.
4. **Inflation:** one inflation-rate input (default 2.5%), separate from the growth-rate slider; inflates Mode-B spending + guaranteed income + the IRMAA threshold. Display **nominal by default**, optional **today's-dollars** toggle.
5. **IRMAA overlay:** **Medium** — projected-income line + inflated tier-1 threshold flag, labeled approximate. (Lite = none; Full = per-year tier+surcharge = 2.0.)
6. **Accounts in scope:** **Traditional IRA + 401(k) aggregate only**; Roth/brokerage sequencing is 2.0/2.1. (Keeps every modeled withdrawal taxable → clean projected-income proxy.)
7. **Household split:** **pro-rata** across primary's and spouse's Traditional/401(k) balances; each spouse's RMD at their own RMD age.
8. **Legacy:** **propagate drawdowns into the Legacy/estate projection** (decision (b)).
9. **Architecture:** standalone tiny projector on current main; do NOT build atop the 2.0 engine.

They will consume the same drawdown assumptions (mode, withdrawal/gap, inflation) so the projected estate reflects spending. This is the one intentional cross-tab dependency in V1.9.

## 4. Projection engine

**New type: `DrawdownProjectionEngine`** (pure, standalone, current main). Per owner, per year, from `currentAge` to `currentAge + horizon` (max 40):

```
state: balance per owner (primary, spouse), seeded from Traditional+401(k) aggregate
for each year y (age = startAge + y):
    desiredHouseholdWithdrawal =
        mode == .spendingGap
            ? max(0, inflate(spendingTarget, y) - guaranteedIncome(year))   // gap
            : rate * householdBalance                                        // mode == .rate
    split desiredHouseholdWithdrawal pro-rata by each owner's balance
    for each owner:
        rmd = (age_owner >= rmdAge_owner) ? RMDCalculationEngine.calculateRMD(age_owner, balance_owner) : 0
        actualWithdrawal_owner = max(ownerShareOfDesired, rmd)   // RMD floor; no double count
        balance_owner -= actualWithdrawal_owner
        balance_owner *= (1 + growthRate_owner)
    record { year, age, householdBalance, householdWithdrawal, rmdFloor, guaranteedIncome, projectedIncome }
```

- **`guaranteedIncome(year)`** = Social Security (turns on at the user's chosen claiming age — that timing data exists in the SS planner) + pension, each inflated by the single rate. **Open implementation question for planning:** confirm whether the income model carries a per-source *start year* for pensions ([IncomeModels.swift](RetireSmartIRA/IncomeModels.swift)); if it does, ramp the pension at its start; if not, treat the pension as active from projection start and note the limitation. SS ramping is the load-bearing case (it drives the large early-60s gap that shrinks once SS begins — the feature's core insight), so it must be time-varying regardless.
- **`projectedIncome`** (for the IRMAA flag) = `householdWithdrawal + guaranteedIncome(year)` (all gross; Traditional withdrawals are fully taxable). Explicitly approximate.
- **Nominal vs real:** engine computes nominal; the view deflates by `(1+infl)^-y` when the today's-dollars toggle is on.
- **RMD divisors:** reuse `RMDCalculationEngine.lifeExpectancyFactor` ([RMDCalculationEngine.swift:22](RetireSmartIRA/RMDCalculationEngine.swift:22)). Do not duplicate the table.

## 5. UI

**Inputs (new section header "Retirement Drawdown"):**
- Mode toggle: `[Spending gap | Withdrawal rate]`.
- Mode B: currency field "Target annual spending" (household). A read-only line shows the current-year guaranteed income pulled from the app, with a note it ramps as SS/pension start.
- Mode C: percent stepper "Withdrawal rate" (e.g., 4.0%), range ~2–8%.
- Inflation rate field (default 2.5%).
- Today's-dollars toggle.
- Horizon picker extended to include 30 and 40 years.

**Chart (A+):**
- Default: balance-over-time line; vertical markers at each spouse's SS-start and RMD-start; thin annual-withdrawal line beneath. Brand-token colors per the 1.8 color contract (`Color.Chart.*`, never `Color.Semantic.*`).
- Expandable disclosure: per-year cash-source stacked bars (guaranteed income / gap draw / RMD-forced).

**IRMAA Medium overlay:**
- A `projectedIncome` line and shading on years where `projectedIncome >= inflate(irmaaTier1Threshold, y)`.
- Caption: "Approximate. Flags when projected income (withdrawals + Social Security/pension) reaches the first IRMAA tier; does not compute exact tax."

## 6. Test strategy (Swift Testing `@Test`)

- **`DrawdownProjectionEngineTests`:** flat balance trajectory (mode C); `max(planned, RMD)` netting at/after RMD age (planned>RMD and planned<RMD); pro-rata split across two owners with different RMD ages (73 vs 75); mode-B gap with SS starting mid-projection (gap large pre-SS, shrinks after); inflation applied to spending + guaranteed income; nominal vs today's-dollars equivalence; 40-year horizon bound; zero-balance floor.
- **IRMAA flag:** threshold-crossing detection with inflated tier-1; approximate-proxy income assembly.
- **Legacy integration:** estate projection reflects drawdowns (regression vs prior no-drawdown behavior behind the feature being active).
- **NJ phaseout** (release scope item B): $100K/$125K/$150K boundary tests, single + MFJ (per decision-log 2026-06-14).
- Full suite (1,271+) green before ship.
