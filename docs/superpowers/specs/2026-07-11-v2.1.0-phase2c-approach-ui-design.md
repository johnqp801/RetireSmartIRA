# V2.1.0 Phase 2c — Conversion-Approach UI (Multi-Year tab) — Design

**Date:** 2026-07-11
**Phase:** 2c (the SwiftUI phase) of V2.1.0 selectable conversion approaches.
**Branch:** `2.1/selectable-conversion-approaches`.
**Depends on:** the frozen spec `docs/superpowers/specs/2026-07-10-selectable-conversion-approaches-charitable-modeling-design.md` (§6 Phase 2) and the Phase 2b comparison model (`ApproachComparison` / `ApproachColumn` / `ConsequenceDeltas` / `ConsequenceFlags` / `ApproachComparisonCoordinator` / `MultiYearCPABriefing.approachDeltaSummary`), all built and green. This document refines the frozen spec's UI section with the concrete Phase 2c decisions taken 2026-07-11; it does not change the engine or the comparison model.

## Goal

Give the Multi-Year tab a user-facing "Conversion approach" control that lets the user select among the three built approaches, render the three-way comparison + path-level consequences already computed by the engine, and make the selected approach the **active plan** across the tab (charts, balances, Year-1 levers, CPA briefing). Add a light multi-year giving refinement surfacing the already-seeded charitable plan. No engine changes.

## Locked decisions (2026-07-11)

1. **Scope:** approach picker + three-way comparison + consequence display + a *light* giving refinement (surface the seeded charitable amount, funding = QCD-first, cash-not-deducted disclosure). No new charitable data model.
2. **Comparison IA:** the three-way comparison **replaces** the two-way `PlanComparisonView` as the tab's single comparison block. The `PlanComparison` *model* stays (still consumed by the CPA briefing and `TaxImpactChart`).
3. **Comparison layout:** three-column metric table (anchor / selected / no-added-conversions) + consequence strip + triggered-effect flags; collapses to two columns when selected == the anchor.
4. **Approach picker:** segmented 3-way control; when a deterministic approach is active, **inline chips** (config-derived bracket rates / IRMAA tiers) with baseline-above-target status; IRMAA adds a buffer field (reuses `cliffBuffer`).
5. **Reach:** the selected approach becomes the **active plan everywhere** (charts, balances, Year-1 levers, CPA briefing), not just a comparison lens.
6. **Year-1 rule:** the deterministic rule and a manual Year-1 override are **mutually exclusive** (§4 below).
7. **No "Recommended" anywhere — the app does not make recommendations.** The anchor column and the picker's first segment are labeled by the objective the engine actually optimized, dynamically on the Legacy Planning setting: **"Minimize lifetime tax"** when Legacy Planning is off (heir weight 0), **"Optimize tax + legacy"** when it is on. This supersedes the frozen spec's "Recommended plan" label while preserving that spec's reasoning (never imply heir weight = 0 / never call the legacy-on anchor "tax-minimizing").

## Label scheme (single source of truth for copy)

| Element | Label |
|---|---|
| Anchor column + picker segment 1 (`.recommendedTaxMin`) | Legacy OFF → **"Minimize lifetime tax"**; Legacy ON → **"Optimize tax + legacy"** |
| Picker segment 2 (`.fillToBracket(rate)`) | **"Fill to bracket"** → when active, column reads **"Fill to N% bracket"** |
| Picker segment 3 (`.limitToIRMAA(tier,buffer)`) | **"Limit to IRMAA"** → when active, column reads **"Limit to IRMAA tier N"** |
| Third comparison column | **"No added Roth conversions"** (RMDs, QCDs, withdrawals, giving still occur) |
| Year-1 override active on the objective optimizer | existing off-plan / **"Custom"** badge |

A single helper (e.g. `ConversionApproach.displayLabel(legacyPlanningOn:)` in the view layer) produces the anchor/segment-1 label so the picker segment, the comparison column header, and the CPA briefing all read identically. **"Legacy-on" is defined precisely as the anchor's own heir weight being greater than zero** — i.e. `selectedHeirWeight > 0`, the exact weight passed to the anchor's `optimize(heirWeight:)`. When `selectedHeirWeight == 0` the anchor IS the pure lifetime-tax minimizer, so the label reads "Minimize lifetime tax"; when `> 0` the anchor optimizes the blended objective, so it reads "Optimize tax + legacy". This keeps the label a faithful description of what the anchor column actually computed, with no second signal to disagree with it.

## §1 · State & plumbing (mirror the heir-frontier pattern)

`MultiYearStrategyManager` gains a selection + computed comparison structured exactly like the existing `selectedHeirWeight` / `heirFrontier` / `computeHeirFrontier()` / `isComputingFrontier` group:

- `@Published var selectedApproach: ConversionApproach` (default `.recommendedTaxMin`).
- `@Published private(set) var approachComparison: ApproachComparison?`
- `@Published private(set) var isComputingComparison: Bool`
- `computeApproachComparison()` — runs `ApproachComparisonCoordinator().compare(inputs:assumptions:selectedApproach:heirWeight:configProvider:)` off the main actor (same detached-task + cancellation discipline as `computeHeirFrontier`), publishing on the main actor. `heirWeight` = the tab's current `selectedHeirWeight` (so the anchor is heir-aware exactly when Legacy Planning is on).

**Persistence:** `selectedApproach` persists in `MultiYearAssumptions` (the tab's persisted, codable surface). Store a small codable representation — an approach kind enum plus optional `rate` / `tier` / `buffer` — and rehydrate into a `ConversionApproach`. Round-trips across relaunch like the other assumptions. `cliffBuffer` already exists and is reused for the IRMAA buffer (no new persisted field for the buffer).

**Recompute triggers:** `selectedApproach` changes recompute the active plan + the comparison (same debounce pipeline as assumption edits). Selecting an approach is a static-input-class change (not a slider-only tweak), so it recomputes the optimal/active path, not just the comparison.

## §2 · The two controls (top of the section)

**Approach picker.** A segmented control with the three segments (labels per the scheme above). When `.fillToBracket` or `.limitToIRMAA` is active, a dependent **inline-chip** control appears directly beneath:

- **Fill to bracket:** one chip per ordinary bracket rate **derived from the config bracket table** for the filing status (never hardcoded). A bracket already below the year's unavoidable baseline ordinary income is rendered struck-through / disabled with the status line: *"Baseline income already exceeds the N% ceiling, so no conversion is added this year — the policy still applies later."* Brackets are never filtered out solely because year one can't reach them.
- **Limit to IRMAA:** one chip per IRMAA tier (config-derived) + a buffer field bound to `cliffBuffer`. Same baseline-above-target status treatment ("Baseline MAGI already exceeds this tier…").

Baseline-above-target statuses distinguish room-available / at-target / target-exceeded-by-baseline / not-applicable (spec §6), with the exact messaging above.

**Giving refinement.** A compact block that **surfaces** (does not re-capture) the charitable plan already seeded by `MultiYearInputAdapter` from `dataManager.scenarioTotalCharitable`: shows the annual charitable amount, funding = **QCD-first**, and carries the disclosure that **non-QCD cash charitable is not tax-deducted in this projection** (rides 2.1.1 itemized). No new giving inputs, per-spouse controls, or percent-of-RMD UI in 2.1.0 (those are the 2.1.1 line the frozen spec drew). If a lightweight funding display is all that fits cleanly, that is acceptable — the disclosure is the required element.

## §3 · The three-way comparison view (replaces the two-way)

New `ApproachComparisonView` (SwiftUI) renders the `ApproachComparison` model:

- **Three-column metric table** — columns: anchor (dynamic label) / selected (e.g. "Fill to 24% bracket") / "No added Roth conversions". Rows: lifetime tax, ending traditional, ending Roth, what heirs keep, peak forced RMD (reuse the metric orientation/formatting `PlanComparisonView` uses today; each column is an `ApproachColumn`). Respect the display-units (nominal / present-value) toggle already in the tab, using `ApproachColumn.lifetimeTaxNominal` vs `lifetimeTaxPV` and `terminalPVFactor` for terminal balances.
- **Consequence strip** — path-level dollar deltas of the selected approach vs "No added Roth conversions" from `ConsequenceDeltas`: federal, state, IRMAA, ACA, NIIT, and the total. IRMAA shows the premium-year basis. Selected-vs-anchor deltas (lifetime tax / peak conversion / Medicare) come from `approachDeltaSummary` and can headline the strip.
- **Triggered-effect flags** — from `ConsequenceFlags`: SS taxation increased, IRMAA tier crossed, ACA cliff crossed, ordinary bracket crossed, cap-gain bracket affected, NIIT increased. Rendered as small tags. **NIIT wording:** "NIIT increased because MAGI crossed the threshold and the household has net investment income" — never implying the Roth conversion itself is net investment income.
- **Duplicate-column collapse** — when `approachComparison.collapsesToTwoColumns` (selected == `.recommendedTaxMin`), render two columns (anchor + "No added Roth conversions"); the consequence strip/flags then compare anchor vs no-conversions.

This view **replaces** `PlanComparisonView` in the tab. `PlanComparison` (the model), `MultiYearCPABriefing`'s use of it, and `TaxImpactChart` are untouched.

## §4 · "Active plan everywhere" + the Year-1 rule

Making the selection the active plan: the manager's active/current path is produced by `optimize(approach: selectedApproach, heirWeight: selectedHeirWeight)` when a deterministic approach is selected; the tab's charts (balances, conversion ladder, tax impact, threshold map), balances, and the CPA briefing all read that active path. When `.recommendedTaxMin` is selected, behavior is exactly today's (plus any Year-1 override).

**Year-1 lever interaction — mutual exclusion (locked):** a deterministic per-year rule and a manual Year-1 override cannot both hold.

- Selecting `.fillToBracket` / `.limitToIRMAA` sets the plan; the editable **Year-1 Roth field reflects** the approach's computed Year-1 conversion (shown, read-derived — not blanked). Withdrawal/QCD Year-1 fields behave as today (the engine emits no withdrawal/QCD candidates; deterministic approaches only set conversions).
- If the user **edits the Year-1 Roth field** while a deterministic approach is active, the approach reverts to the **objective optimizer** (`.recommendedTaxMin`) with the manual Year-1 override applied — i.e. the existing off-plan / "Custom" state and badge. A manual override and a deterministic rule never fight.
- The picker reflects the revert (segment 1 becomes active) so the UI never shows a deterministic approach selected while a conflicting manual Year-1 value is in force.

This keeps the tab's single-source-of-truth Year-1 model intact and mirrors the existing off-plan-badge pattern.

## §5 · CPA briefing + copy rules

- The briefing **leads with the selected approach** and its Δ-vs-anchor (lifetime tax / peak conversion / Medicare), wiring the Phase-2b `MultiYearCPABriefing.approachDeltaSummary` into the rendered HTML body. When collapsed (selected == anchor), the deltas are zero and the briefing reads as today's single-plan briefing.
- Copy guardrails: the **anchor is never called "Recommended"** (use the dynamic objective label); the **NIIT** line uses the careful wording above; surface the **heir-frontier-applies-only-to the objective optimizer** precedence (the deterministic approaches ignore heir weight — no frontier for them); include the **cash-charitable-not-deducted** disclosure.

## §6 · Testing

- **Model / manager logic (engine test loop):** `selectedApproach` → `computeApproachComparison` wiring; `MultiYearAssumptions` persistence round-trip of the approach (kind + rate/tier/buffer); collapse flag drives two-vs-three columns; the label helper returns the right string for legacy-on vs legacy-off; the Year-1-edit-reverts-to-`.recommendedTaxMin` rule; baseline-above-target status classification for a bracket/tier below the year's baseline.
- **SwiftUI (preview / visual verification in the running app):** the picker + inline chips + status, the three-column table (and its two-column collapse), the consequence strip + flags, the giving disclosure, and the Year-1 reflect/revert behavior — verified via the browser-preview workflow (build, drive the tab, read the rendered state, screenshot), since visual correctness is the modality this phase adds.
- Full suite stays green; no rebaselines (this phase is UI + manager wiring on top of the finished engine/model).

## Out of scope (2.1.1 or later)

Per-spouse / percent-of-RMD / funding-method giving controls; full itemized cash-charitable deduction; per-conversion incremental consequence attribution; N-way (fill-22 vs fill-24) comparison; making the deterministic approaches swappable as comparison columns; any change to the engine or the Phase-2b model.

## Open implementation questions (resolve in the plan)

- Exact placement of the approach section within the current Multi-Year tab section order (relative to Year-1 levers, the heir-frontier control, and the charts).
- The codable shape for persisting `ConversionApproach` in `MultiYearAssumptions` (enum + associated scalars vs a flat struct) and its migration/default for existing persisted assumptions (must default to `.recommendedTaxMin`).
- Whether the giving refinement is an always-visible block or a disclosure attached to the giving amount already shown elsewhere in the tab.
