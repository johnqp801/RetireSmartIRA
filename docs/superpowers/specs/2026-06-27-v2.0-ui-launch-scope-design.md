## 1. Purpose

Take the Multi-Year Plan tab from a thin, read-only MVP to a **launch-ready V2.0** product: interactive, charted, exportable, and honestly scoped. The engine is locked and validated (1083 tests; CPA-reviewed; PV-correct on a real $5.6M profile). This is overwhelmingly a **rendering + interaction** effort — most "new" capability already exists as engine output or manager scaffolding — plus exactly **one narrow engine change** (heir step-up, §7).

## 2. Product principle & positioning (hard constraints)

1. **Additive only.** The single-year **Scenarios** and **Tax Summary** tabs are CORE and MUST NOT be modified, replaced, or removed. The Multi-Year Plan tab (ContentView tag 10) is purely additive. A test asserts the tab is wired without touching those views.
2. **Focused tool, not a full planner.** V2.0 is positioned as a Roth-conversion / RMD / IRMAA / ACA / survivor / heir-tax optimizer with transparent assumptions. Copy (in-app + marketing + PDF) MUST avoid "complete / full retirement income optimization." Positioning language is itself a launch deliverable, not an afterthought.
3. **Trust comes from not over-claiming.** Disclaimers mark the *edge of scope*; what is inside scope is CPA-reviewed and rigorous. The Assumptions & Limitations surface (§6.6) is a launch requirement, not a slip-able polish item.
4. **Engine stays locked.** The only intentional engine change in V2.0 is §7 (4a, heir step-up). Anything that reopens the optimizer's per-year loop is out of scope (see §9).

## 3. Current built state (verified in code, 2026-06-27)

The tab and its manager are further along than "thin MVP" implies. Already live on `2.0/heir-objective`:

- **`MultiYearPlanView`** — tab root, single scrolling column, `@StateObject MultiYearStrategyManager`, Future-$/PV segmented toggle, `attach(...)` + `recomputeAll()` on `.task`. Renders AssumptionsStrip, PlanSummary, PlanComparison, LadderList, HeirFrontier with loading / setup-incomplete states.
- **`MultiYearStrategyManager`** (`@MainActor ObservableObject`) — two-cache (`engineOptimalResult` vs `currentResult`), 500ms + 50ms two-stage debounce, cooperative cancellation, off-main `Task.detached` compute, `baselineProjection`, `heirFrontier`, `selectedHeirWeight`, `isComputing` / `isComputingFrontier` / `computeFailed`, `firstOffPlanShown`/`markFirstOffPlanShown`, `dismissInsight`/`restoreDismissedInsights`/`dismissedInsightKeys`, `yearsBeforeFirstRMD`, and **`resetYear1ToEngineOptimal()`** which already reads/writes six `DataManager` levers (`yourRothConversion`, `spouseRothConversion`, `yourExtraWithdrawal`, `spouseExtraWithdrawal`, `yourQCDAmount`, `spouseQCDAmount`).
- **Adapter** supports `excludeYear1Overrides` (optimal vs as-edited compute).
- **Value types / presentation models:** `PlanSummary`, `PlanComparison` (plan vs doing-nothing, PV-consistent), `LadderRow` (IRMAA attributed to conversions), `HeirFrontierResult`/`FrontierPoint` (6 weights, each carrying its own `recommendedPath`), `SensitivityBands` (opt/avg/pess growth), `widowStressDelta`, `ssClaimNudge`, `ConstraintHit`.

**The documented gap:** `observeUpstreamChanges()` tracks only `dataManager.scenario` + `scenarioStateManager.yourRothConversion`, with an inline note that full field parity is "the deferred UI milestone." That gap is launch item §6.1.

## 4. Architecture approach

**Chosen: extend the existing tab in place; keep all presentation logic in pure value types.** Views stay thin renderers over `manager` state and `*View`-specific value-type models (the established pattern — `PlanSummary`/`LadderRow`/etc.). No master-detail, no migration/locked-pane UX (void), no replacement of single-year tabs.

Two approaches considered and rejected:
- *Port the old plan-b-ui tab wholesale.* Rejected: it serves the replace-Scenarios architecture we killed; cherry-picking components is lower-risk than inheriting its frame.
- *New parallel "advanced" tab.* Rejected: violates the additive-but-coherent goal; one Multi-Year Plan tab is the product.

Per-increment work attaches to the existing single scrolling column (phone/iPad/Mac identical), matching `RMDCalculatorView` for chart/toggle conventions and MetricCard for stat presentation.

## 6. Launch-bar increments

Each is independently shippable behind the existing tab. Order = recommended build order (see §10).

### 6.1 Editable Year-1 levers + full observation tracking
- **New view `Year1EditorView`** surfacing the six `DataManager` levers (Roth conversion primary/spouse, extra withdrawal primary/spouse, QCD primary/spouse) as labeled inputs/sliders. Edits flow through `DataManager` (the existing source of truth), which the manager already observes.
- **Off-plan indicator:** compare as-edited (`currentResult`) vs `engineOptimalResult`; show a 4-state badge (On plan / Near optimal / Off plan / Significantly off plan) with the plan-b thresholds (<$1K, $1K–$25K, >$25K lifetime-objective delta). Drive copy from a pure `OffPlanStatus` value type. Wire the existing `resetYear1ToEngineOptimal()` to a "Reset to optimal" button and `firstOffPlanShown` to neutral first-impression copy.
- **Full observation tracking:** expand `observeUpstreamChanges()` to track every upstream field that affects the plan (the six levers + relevant scenario/profile inputs), replacing the current 2-field read. Add a manager-integration test asserting each tracked field triggers a debounced recompute (closes review finding #6).
- **Tests:** `OffPlanStatus` thresholds; observation-tracking integration (per-field); `Year1EditorView` constructs from fixture.

### 6.2 Charts / visualizations (Swift Charts)
- **Conversion ladder chart** — recommended conversion by year (bars), bracket/cliff context.
- **Account balances over time** — traditional / Roth / taxable (and HSA) stacked or multi-line from `endOfYearBalances` across `recommendedPath`.
- **Heir-frontier curve** — owner lifetime tax (x) vs what heirs keep (y) across the 6 `FrontierPoint`s, selected weight highlighted; respects the $/PV toggle.
- **Growth sensitivity band** — opt/avg/pess from `SensitivityBands` as a shaded band on the balances or lifetime-tax chart. **Label strictly as growth sensitivity; never "risk" or "odds of success"** (enforced by the `SensitivityBands` doc contract).
- Each chart backed by a pure `*ChartModel` value type (data points + axis/labels), view renders only. Match `RMDCalculatorView` chart styling and the color system.
- **Tests:** each chart model from fixtures; sensitivity-label guard.

### 6.3 Advanced assumptions sheet
- **New `AdvancedAssumptionsSheet`** (`.sheet`) exposing `MultiYearAssumptions`: investment growth, CPI, `pvRealDiscountRate`, **`terminalLiquidationTaxRate`** (flagged required — dominant aggressiveness lever), per-spouse `horizonEndAge`, and the **withdrawal-ordering preset** (`WithdrawalOrderingRule`: tax-efficient / deplete-trad-first / preserve-Roth / proportional — surface the existing enum; this is NOT an optimizer).
- Edits mutate `manager.assumptions` then `recompute(.assumptionsChanged)`. Each control has an inline hint (use the inline-hint vocabulary) explaining its effect.
- The basic `AssumptionsStripView` keeps the high-traffic inputs (taxable, HSA, horizon) and gains an "Advanced" affordance opening the sheet.
- **Tests:** sheet binding round-trips into assumptions; changing `terminalLiquidationTaxRate` / withdrawal preset triggers recompute.

### 6.4 CPA-briefing PDF export
- **New `CPABriefingPDF` builder** producing: executive summary, recommended conversion ladder, plan-vs-doing-nothing comparison, **year-by-year table** (year, age, AGI, MAGI(s), taxable income, federal/state/IRMAA/ACA, RMD, conversion, ending Trad/Roth/Taxable balances), heir-frontier readout, **assumptions page**, **Limitations section** (§6.6 content), disclaimers.
- Pure model → renderer split so the table/figures are unit-testable independent of PDF layout. Export entry point on the tab; share sheet on iOS, save panel on macOS.
- **Tests:** PDF model assembles all sections from a fixture result; limitations section present and non-empty; numbers match the on-screen figures.

### 6.5 Engine-output riders (cheap, high-trust)
- **Survivor banner** — surface `widowStressDelta` ("if one spouse passes, the survivor pays ~$X more in lifetime tax under current law"), dismissible, with the documented modeling caveat.
- **SS-claim nudge banner** — surface `ssClaimNudge` (`ClaimAgeFlag`) when present.
- **Conversion-opportunity-window banner** — use `yearsBeforeFirstRMD`.
- All banners use `dismissInsight`/`dismissedInsightKeys`; a "restore insights" control via `restoreDismissedInsights()`.
- **Richer year table** — promote `LadderListView` to a fuller `YearTableView` exposing the per-year `YearRecommendation` fields above, with IRMAA/ACA/bracket cliff flags from `ConstraintHit` + the per-year MAGIs.
- **Tests:** banner presence/dismissal logic; year-table row model from fixtures.

### 6.6 Assumptions & Limitations surface (launch requirement)
- A dedicated, always-available in-tab section + the PDF section listing what V2.0 models and what it simplifies. Must explicitly state: lifetime taxable-account withdrawals use simplified tax treatment (no lot-level cost basis; LTCG/QDI handling per current engine); withdrawal *order* follows the selected preset (not optimized); growth sensitivity is deterministic, not Monte Carlo; muni interest excluded from MAGI ([[multi-year-muni-magi-gap]]); single-filer survivor modeling is a conservative upper bound.
- Backed by a pure `[Limitation]` value type reused by both UI and PDF (single source of truth).

## 7. Engine change (4a only) — heir step-up at death

Both heir-value sites omit the taxable account, which receives a stepped-up basis at death and passes near tax-free:

- `HeirFrontierCoordinator.swift:53` — `let heirKeeps = terminalRoth + (terminalTrad - heirTax)`
- `PlanComparison.heirsKeep` — same omission (documented inline as a known under-credit)

**Change:** add the terminal taxable balance, credited near-tax-free (step-up = no gain to tax at death):
```swift
let heirKeeps = terminalRoth + (terminalTrad - heirTax) + terminalTaxable
```
`endOfYearBalances.taxable` is already tracked per year. This does NOT touch the optimizer, the per-year tax loop, or require any capital-gains math. It only stops *under*-crediting heirs (directionally safe). Apply at both sites for consistency; update `PlanComparison`'s inline limitation note.

- **Tests:** heir-keep includes terminal taxable at both sites; a large-taxable-balance fixture shows the corrected (higher) heir value; frontier and comparison agree.
- **Re-validation:** full suite green after the change (engine-touching → CLAUDE.md rule).

## 8. Engine/manager additions needed (small, contained)

- Expand `observeUpstreamChanges()` field set (§6.1).
- Add pure value types: `OffPlanStatus`, `*ChartModel`s, `Limitation`, `CPABriefing` model, richer `YearTableRow`. No new engine math beyond §7.
- No change to the optimizer, single-year code, or the `MultiYearStrategyResult` shape (existing fields already carry everything except the §7 correction).

## 9. Out of scope (deferred)

**V2.1:** 4b — capital-gains/NIIT tax on *lifetime* taxable withdrawals (requires fixing the LTCG/QDI-taxed-as-ordinary Path A simplification AND touches the optimizer loop / interacts with conversion decisions); full taxable brokerage cost-basis modeling; withdrawal-order **optimizer** (decision-variable search — perf risk on top of the locked cap=2 fixed point). Consistent with the Tim 2.1 commitment.
**Indefinite:** Monte Carlo / multi-factor sensitivity (positioning mismatch; compute-infeasible at ~1.5s/optimize × 1000 paths).
**Void:** any migration / locked-pane / replace-Scenarios UX; master-detail layout.

## 10. Phasing (recommended build order)

1. **§7 heir step-up** (tiny, removes a live trust bug; re-run suite) +
2. **§6.6 Assumptions & Limitations + positioning copy** (cheap, gates honest framing) +
3. **§6.5 riders** (pure rendering of existing outputs).
4. **§6.1 editable Year-1 + observation tracking** (the interactivity core).
5. **§6.3 advanced assumptions sheet** (unlocks the levers power users need).
6. **§6.2 charts.**
7. **§6.4 CPA PDF** (consumes everything above).

Each phase ends green and is independently demoable. The async/off-main dispatch + "Computing…" affordance already exist (manager), so no separate perf phase is needed.

## 11. Testing strategy

- **Pure value types** (Swift Testing) for every presentation model: `OffPlanStatus`, chart models, `YearTableRow`, `Limitation`, `CPABriefing`, plus the §7 heir-keep correction.
- **Manager integration:** observation tracking per-field; reset-to-optimal; frontier weight selection drives summary/ladder.
- **View construction / snapshot:** each new view builds from a fixture without crashing (mirrors existing `*ViewTests`).
- **Product-principle guard:** Scenarios / Tax Summary tab content unchanged.
- **Full suite green** after every engine-touching change (§7) — source of truth, 1083+ tests.

## 12. Definition of done

The Multi-Year Plan tab is interactive (editable Year-1 with off-plan indicator + reset), charted (ladder, balances, heir frontier, growth sensitivity), configurable (advanced assumptions incl. terminalLiquidationTaxRate + withdrawal preset), exportable (CPA PDF with year table + limitations), and honestly framed (Assumptions & Limitations surface; non-full-planner copy). Heir value credits taxable step-up (§7). Scenarios / Tax Summary untouched. All new logic/manager/view tests + full suite green. V2.0 is shippable to the App Store without waiting on V2.1.
