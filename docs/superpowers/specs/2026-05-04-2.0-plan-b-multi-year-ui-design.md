## 1. Goal & relationship to engine spec

This spec covers the SwiftUI surface that consumes `MultiYearTaxStrategyEngine`. It defers all math, optimization behavior, constraint acceptance, sensitivity bands, and engine API definitions to the engine spec. The engine produces a `MultiYearStrategyResult`; this spec defines what we render with it.

**User promise (unchanged from engine spec):** *"Show me how to minimize lifetime tax across my retirement years, with full transparency about which cliffs and tiers the plan accepts as trade-offs."*

**What changed from engine-spec §4:**

- §4 sketched a single-tab "Multi-Year Strategy" view with the 1.9 Reduce-AGI dashboard "embedded verbatim" in Year 1 expansion. That premise turned out to be flawed: the 1.9 surface is two views (3,681-line `TaxPlanningView` for scenario building + 2,628-line `DashboardView` for projection), and §4 only contemplated the latter. Naively replacing the 1.9 tabs would have deleted the scenario-builder UX entirely.
- §4 specified an "override delta panel" (Accept / Revert flow). This flow assumes a submit-style interaction; the V2.0 model is direct manipulation. Delta panel is dropped.
- §4 placed Multi-Year Strategy as the primary tax tab, leaving open whether it replaced one or both of Scenarios + Tax Summary. This spec resolves: it replaces both (consolidated into a single "Tax Planning" tab).
- §4 did not address mobile execution. This spec adds the bottom-sheet pattern for iPhone / iPad compact.

## 2. Information architecture

### 2.1 Tab consolidation

Today's 10 tabs include:

- Tab 5 **Scenarios** — `TaxPlanningView`, the scenario builder (Roth conversion sliders, withdrawal sliders, QCD, charitable, deduction comparison)
- Tab 6 **Tax Summary** — `DashboardView`, the projection display (bracket charts, IRMAA tier, NIIT chart, account balances)

Both fold into a single new tab **Tax Planning**. The new tab IS the Multi-Year Strategy view. No separate "Multi-Year" tab. No segmented control between "This Year" and "Multi-Year." Multi-year is the canonical view; Year 1 (the current year) is its actionable slice.

**Net tab count: 10 → 9.**

### 2.2 Sidebar restructure (macOS / iPad regular)

The existing sidebar has three sections (Setup / Analysis / More). The Analysis section becomes:

```
Section "Analysis":
  RMD Calculator
  Tax Planning      ← new, replaces Scenarios + Tax Summary
```

No new section header. The macro/micro split doesn't need to be expressed in the IA — it's expressed inside the Tax Planning view itself.

### 2.3 iPhone tab order

Current visible-5 (tabs 0–4): Get Started, Profile, SS, Income, Accounts. Tax Planning would land in the More overflow menu.

This is wrong: Tax Planning is the V2.0 hero feature; burying it behind an extra tap undercuts the launch story.

**New visible-5:** `[Profile, Income, Accounts, SS, Tax Planning]`.

- **Profile leftmost** — natural identity-first position; first-stop for new users.
- **Tax Planning rightmost** — destination position. Returning users (the ~95% case) reach the right edge with thumb muscle memory.
- **Get Started removed from visible-5** — onboarding checklists are ephemeral. See §2.4 below.
- More overflow menu retains: RMD Calculator, Quarterly Tax, State Comparison.

Center placement of Tax Planning was considered (see §13 post-launch UX research candidates) but rejected for V2.0 because it interrupts the LTR setup → destination flow.

### 2.4 Get Started migration

`GuideView` (Tab 0 today) becomes a **dismissible banner at the top of `SettingsView` (Profile tab)**.

- Banner shows the existing setup checklist (Profile complete · Accounts added · Income added · SS info entered).
- Banner is **dismissed permanently** when the user taps a "Got it" / × control, OR automatically when all four checks are true and the user has visited Tax Planning at least once.
- Dismissal flag stored on `DataManager` (or wherever first-run state lives today).

The standalone `GuideView` and Tab 0 are removed from ContentView.

### 2.5 Default tab on app launch

After onboarding completes, default selected tab on launch is **Tax Planning** (the V2.0 hero). For a new user with incomplete setup, Tax Planning's empty state deep-links them back to the missing setup tabs.

## 4. Components

Components are listed top-to-bottom by their position in the macro pane, then the right-pane / sheet contents.

### 4.1 Pill bar

Horizontal scrollable strip at top of view. Pills represent current `MultiYearAssumptions` values.

| Pill | Type | Tap behavior | Default |
|---|---|---|---|
| `CPI 2.5%` | Numeric | Popover stepper (range 0–6%, 0.1% step) | 2.5% |
| `Growth 6%` | Numeric | Popover stepper (range 0–10%, 0.5% step) | 6% |
| `End age 95` | Numeric | Popover stepper per-spouse if MFJ (range 80–105) | 95 |
| `SS 67/67` | Enum | Popover picker per spouse (62–70) | 67 each |
| `Rule: Tax-eff.` | Enum | Popover picker (4 options: Tax-efficient · Deplete-trad · Preserve-Roth · Proportional) | Tax-efficient |
| `Heir Tax Rate: 22%` (featured pill — distinct color) | Numeric | Popover stepper (range 0–37%, 1% step) + info hint expandable | 22% |
| `⚡ Stress test ON` (toggle pill) | Boolean | Tap to flip | ON |
| `⋯ Advanced` | Sheet | Opens settings sheet (cliffBuffer, restore dismissed insights, debug) | — |

**Heir Tax Rate info hint:** Tappable (i) icon on the pill expands inline:
> *The tax rate we model your heirs (or you, at end of plan) paying on remaining traditional IRA balances. Higher rate → engine recommends more aggressive Roth conversions now.*

Closes on tap-outside or after 8 seconds.

**Featured pill styling:** `Heir Tax Rate` uses a distinct background color (matching the iPad mockup's `#d8e7ff` accent) to signal it's the dominant lever on engine aggressiveness. Per [memory/v2-status.md](../../../.claude/projects/-Users-johnurban-Projects-RetireSmartIRA/memory/v2-status.md): "Default 0.22 is reasonable but it's the dominant lever … without UI, the engine is a black box."

**Hit-testing:** disabled when `MultiYearStrategyManager.isComputing == true` to prevent queued conflicting state changes (see §6.3).

### 4.2 Optional callout banners

Appear at the top of the macro pane, below the pill bar, only when applicable:

- **SS claim nudge banner** — renders when `result.ssClaimNudge != nil`. Format: `"Delaying spouse's claim from 64 → 67 could save ~$23K lifetime. [Re-run with 67]"`
- **Widow stress callout** — renders when `result.widowStressDelta.delta > $50K` (threshold tunable; not every household needs this surfaced). Format: `"If [higher earner] dies first, surviving spouse pays ~$X more lifetime tax under single-filer rates."`

**Visual weight scales with impact.** Higher-$ insights get larger banner size and bolder color treatment. Implementation: a single `bannerImpactLevel` enum (`.minor` / `.moderate` / `.major`) drives styling.

**Dismissal:**
- Single-tap dismiss control labeled "Dismiss" (not unadorned ×) for major-impact banners — makes act more deliberate. Minor-impact banners use ×.
- Dismissed state stored in `MultiYearAssumptions.dismissedInsightKeys: Set<String>`. Each key is a content hash (e.g., `"ss-nudge-spouse-67-23k"`).
- Inline "Why this matters" expansion on every banner — context without dismissing.
- "Restore dismissed insights" button in Advanced settings (`⋯` pill) clears the dismissal set.
- **No confirmation modal, no friction-delay**, even on $100K+ insights. Users dismiss with reason; we trust.

### 4.3 Hero stat + objective caption + off-plan indicator

Vertically stacked block:

```
LIFETIME TAX                          [✓ On plan]
$487,200 → $312,400
Saves $174,800 over 30 years

Minimizes lifetime tax including a 22% tax on assets
left to heirs. Accepts IRMAA or ACA cliffs only when
they save more than they cost.
```

The "22%" in the caption reads from `MultiYearAssumptions.terminalLiquidationTaxRate` and updates live when the user changes the heir-rate pill.

**Off-plan indicator** is a 4-state badge in the top-right corner of the hero:

| State | Trigger | Treatment | Inline action |
|---|---|---|---|
| ✓ On plan | All Year 1 overrides match engine recommendations | Green check, no delta | — |
| ✓ Near optimal | <$1,000 lifetime deviation | Green check, "Near optimal (-$340)" | — |
| ⚠ Off plan | $1,000–$25,000 deviation | Yellow, "Off plan: -$8,400" | `[Reset]` link |
| ⚠ Significantly off plan | >$25,000 deviation | Red, "Significantly off plan: -$45,000" | `[Reset]` link |

**Threshold constants** (`Year1OverrideThresholds.minor = 1_000`, `.major = 25_000`) live on `MultiYearStrategyManager` for post-launch tuning.

**Computing the delta:** `MultiYearStrategyManager` holds two cached results:
- `engineOptimalResult: MultiYearStrategyResult` — re-run only when `MultiYearAssumptions` changes
- `currentResult: MultiYearStrategyResult` — re-run whenever any input including user overrides changes (debounced, see §6)

Delta = `currentResult.lifetimeTaxFromRecommendedPath - engineOptimalResult.lifetimeTaxFromRecommendedPath`.

### 4.4 Strategy summary card

Plain-English 2–3 sentence narrative of the engine's recommended path, derived from `result.recommendedPath`. Synthesized server-side (in the Manager, not the View) by a helper that examines key actions:

> *Convert $50–70K/yr to Roth through 2030 (pre-SS bracket-fill), claim SS at 67, then RMD-fill the 12% bracket. Accepts 1 IRMAA Tier 2 hit at 2031.*

Helper logic: identify the highest-frequency LeverAction in the path, cluster by year ranges, name SS claim events, count constraint hits. Plan should specify the exact synthesis grammar (sentence templates per cluster type).

**Contents** (curated subset of `TaxPlanningView`):

| Element | Source |

**The "Restore engine optimal" button is the ONLY reset.** No per-slider ↺ icons. Reasoning: the engine produces a synergistic Year 1 lever set (Roth ⊗ withdrawal ⊗ QCD jointly fill brackets without tipping cliffs). Per-slider reset would either revert to historical values that no longer fit the user's current constraint set (mathematically incoherent), or require partial constrained re-optimization (engine doesn't support per-lever locking within the inner loop). All-or-nothing reset is safe, cheap, and intellectually honest. Per-slider reset deferred to v2.1 IF the engine adds partial locking.

**Live impact strip updates synchronously** on every slider tick from DataManager properties (instant, ms-fast). The macro pane debounces (§6.2).

**Charitable / stock donation / cash gift:** NOT in `Year1QuickEditor`. Available in the embedded `TaxPlanningView` below (master-detail right pane / sheet at `.large`). These are less frequently tweaked than the four primary levers.

## 5. State architecture

### 5.1 `MultiYearStrategyManager` — new ObservableObject

Lives parallel to existing `DataManager` and `ScenarioStateManager`. Single instance, injected via `@EnvironmentObject` like the others.

```swift
@MainActor
final class MultiYearStrategyManager: ObservableObject {
    // Persisted assumptions
    @Published var assumptions: MultiYearAssumptions

    // Engine results — two caches
    @Published private(set) var engineOptimalResult: MultiYearStrategyResult?
    @Published private(set) var currentResult: MultiYearStrategyResult?

    // Compute state
    @Published private(set) var isComputing: Bool = false
    @Published private(set) var hasEverComputed: Bool = false  // gates cold-start skeleton

    // Subscribers
    private var dataCancellable: AnyCancellable?       // ScenarioStateManager / DataManager changes
    private var debounceTask: Task<Void, Never>?
    private var inputHash: Int = 0  // hash of (StaticInputs, Assumptions) for cache validation
    private var optimalHash: Int = 0  // separate hash for engine-optimal cache

    // Public API
    func recompute(reason: RecomputeReason)            // request a recompute; debounces internally
    func resetYear1ToEngineOptimal()                   // wipes Year 1 overrides; triggers recompute
    func dismissInsight(_ key: String)                 // adds to dismissedInsightKeys
    func restoreDismissedInsights()                    // clears the set

    enum RecomputeReason {
        case assumptionsChanged    // recompute BOTH currentResult AND engineOptimalResult
        case overridesChanged      // recompute only currentResult
        case appLaunch             // initial compute on appear
    }
}
```

### 5.2 Two-cache strategy

The off-plan indicator (§4.3) needs both an engine-optimal baseline and a current-with-overrides path. Naive approach: re-run optimization twice on every change. Wasteful — the engine-optimal path doesn't change when only Year 1 overrides change.

**Cache validation:**
- `engineOptimalResult` invalidates on `MultiYearAssumptions` or scenario-static-input changes (anything that affects the optimal solution itself)
- `currentResult` invalidates on any of the above + Year 1 override changes

When user moves a slider:
- `currentResult` recomputes (debounced)
- `engineOptimalResult` stays cached (assumptions didn't change)

When user changes a pill (e.g., growth rate):
- `engineOptimalResult` recomputes (assumptions changed)
- `currentResult` recomputes too (its cache is also invalidated)

### 5.3 Hash key composition

```swift
// engineOptimalHash: assumptions + scenario-static-inputs only (NO user overrides)
let optimalKey = MultiYearStaticInputs(...) ⊕ MultiYearAssumptions(...)

// currentHash: same as above PLUS Year 1 override values
let currentKey = optimalKey ⊕ Year1Overrides(roth, withdrawal, qcd, ...)
```

Both implemented via `Hashable` conformance on the input structs. `hashValue` cached per recompute to avoid recomputing inputs hash on every check.

### 5.4 ScenarioStateManager subscription

`MultiYearStrategyManager` subscribes to changes in `DataManager` and `ScenarioStateManager` via Combine:

```swift
init() {
    dataCancellable = Publishers.Merge(
        dataManager.objectWillChange.eraseToAnyPublisher(),
        scenarioStateManager.objectWillChange.eraseToAnyPublisher()
    )
    .sink { [weak self] _ in
        self?.recompute(reason: /* derived from what changed */)
    }
}
```

The `reason` is derived by comparing pre/post hashes — implementation detail in plan.

### 5.5 Persistence

`MultiYearAssumptions` is persisted on `Scenario` as a single optional field (per engine spec §5.2). `PersistenceManager` extends to serialize/deserialize the new struct. Backward compatible: 1.9 scenarios load with `multiYearAssumptions == nil`. On first view of Tax Planning when nil, the onboarding assumptions sheet (§7.3) presents to populate the struct — defaults are *suggested* in the sheet, not silently applied.

Engine results are NOT persisted — recomputed on launch when needed. The compute is fast enough (<2s) that a cold start on app launch is acceptable.

## 6. Async recompute model

### 6.1 Why decouple

The engine runs in ~1.5–2.7s on M1. Tying every UI tick to that is unusable. But Year 1's micro feedback loop (slider tweak → see this-year tax change) must remain instant — that's the V2.0 promise.

**Solution:** decouple recompute timing per pane.

| Pane | Source | Latency |
|---|---|---|
| Right pane / sheet (Year 1 detail, live impact strip) | `DataManager` computed properties (`scenarioFederalTax`, etc.) | Synchronous, ms-scale |
| Macro pane (waterfall, hero, year list, sparkline) | `MultiYearStrategyManager.currentResult` | Async, debounced |

Right pane updates every slider tick. Macro pane updates ~500ms after the user stops moving sliders.

### 6.2 Debounce mechanism

```swift
func recompute(reason: RecomputeReason) {
    debounceTask?.cancel()
    isComputing = true  // immediate visual signal — see §6.3

    debounceTask = Task { [self] in
        try? await Task.sleep(nanoseconds: 500_000_000)  // 500ms
        guard !Task.isCancelled else { return }

        let staticInputs = MultiYearInputAdapter.build(...)
        let result = await Task.detached(priority: .userInitiated) {
            MultiYearTaxStrategyEngine().compute(inputs: staticInputs, assumptions: self.assumptions)
        }.value

        await MainActor.run {
            self.currentResult = result
            // also update engineOptimalResult if reason warrants
            self.isComputing = false
            self.hasEverComputed = true
        }
    }
}
```

Each new recompute cancels the prior debounceTask. Engine call runs on a detached Task to avoid blocking the main actor.

### 6.3 Stale-state visualization

When `isComputing == true`:

- Macro pane drops to **70% opacity** (`opacity(isComputing ? 0.7 : 1.0)` modifier on the macro container).
- **"Recomputing strategy…"** pill appears in the top-right corner of the macro pane (overlay).
- **Hit testing disabled** on macro pane elements that could trigger conflicting state changes:
  - Year list rows (no year switching mid-compute)
  - Pill bar pills (no assumption changes mid-compute)
  - "Restore engine optimal" button (no double-resets)
  - Trade-offs card and waterfall bars are read-only anyway, so they're fine

Implementation: a `.allowsHitTesting(!isComputing)` modifier on the relevant containers.

Sliders in `Year1QuickEditor` remain interactive throughout — they only update DataManager state (instant), not the engine. The user can keep tweaking; the next recompute picks up their final values.

### 6.4 Cold-start skeleton

When `hasEverComputed == false` (first compute after onboarding):

- Macro pane shows shimmer placeholders:
  - Hero: gray shimmer block where numbers would be
  - Waterfall: gray shimmer bars at uniform heights
  - Sparkline: gray shimmer rectangles
  - Year list: 4–5 gray shimmer rows
- No "Recomputing…" pill (the user hasn't moved anything yet — the skeleton itself signals "computing your plan")

Once `currentResult != nil`, skeleton fades out over 200ms, real content fades in.

For subsequent recomputes (`hasEverComputed == true`), use the stale-state visualization from §6.3 instead of the skeleton — preserves continuity.

### 6.5 Error handling

The engine itself doesn't throw, but compute can fail if `MultiYearStaticInputs` is malformed (missing filing status, missing IRA accounts, etc.). The setup-gap empty state (§7.2) prevents most cases. For edge cases:

- Engine returns `nil` or compute Task throws → show an error banner: "Strategy couldn't be computed. [Details]" with a tappable details link to a sheet listing the missing/invalid inputs.
- Macro pane shows the previous valid `currentResult` if one exists, or the skeleton + error banner if cold start.

No retry UI — assumption is that fixing inputs auto-triggers a fresh compute via the Combine subscription.

## 7. Onboarding

### 7.1 First-visit sequence

User opens Tax Planning tab for the first time. Two possible states:

**A. Setup gaps exist** → setup-gap empty state (§7.2). Cannot proceed to assumptions sheet until gaps are filled.

**B. Setup complete, assumptions unset** → assumptions sheet (§7.3). Required to populate Tax Planning content.

### 7.2 Setup-gap empty state

The engine requires three things from existing 1.9 setup:

| Required | Source | Tab to fix |
|---|---|---|
| Filing status (Single / MFJ) | `DataManager.filingStatus` | My Profile |
| At least one IRA account | `DataManager.iraAccounts.isEmpty == false` | Accounts |
| Social Security planned claiming | `DataManager.primarySSBenefit != nil` | Social Security |

If any are missing, Tax Planning shows:

```
┌── Set up your data first ───────────────────────────┐
│                                                     │
│  Multi-year strategy needs:                         │
│   ☑ Filing status (set on Profile)                  │
│   ☐ IRA account balances → [Go to Accounts]         │
│   ☐ Social Security plan → [Go to Social Security]  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

Each checkbox renders as ☑ when satisfied, ☐ + deep-link button when not. Tapping a button switches tabs to the corresponding setup tab.

When all gaps close, the empty state auto-transitions to §7.3 (assumptions sheet).

### 7.3 Assumptions sheet

Modal sheet (iPad/iPhone) or popover (macOS). Six inputs:

1. **Taxable account balance** — currency field. Default $0. Hint: "Total in non-IRA brokerage accounts."
2. **HSA balance** — currency field. Default $0. Hint: "Total in Health Savings Accounts."
3. **Annual living expenses** — currency field. Default `$60,000`. Hint: "Approximate annual spending in retirement (today's dollars)."
4. **Horizon end age** — stepper. Default 95. Per-spouse override checkbox if MFJ ("My spouse plans to..."). Hint: "Plan ends at this age."
5. **Withdrawal-ordering preset** — picker (4 options). Default Tax-efficient.
   - Tax-efficient: "Defer traditional, preserve Roth" (recommended)
   - Deplete trad first: "Drain traditional before other accounts"
   - Preserve Roth: "Protect Roth as long as possible"
   - Proportional: "Withdraw from all accounts pro-rata"
6. **Heir tax rate** — slider 0–37%, step 1%. Default 22%. Hint: "Tax rate on assets remaining at end of plan (typically your heirs' bracket)."

`cliffBuffer` is **not** in onboarding. Hidden in Advanced settings sheet (`⋯` pill) at default $5,000. Power-user knob.

Submit button: **"Build my strategy"**. Disabled until all required fields valid.

On submit:
- `MultiYearAssumptions` saved to active scenario
- Sheet dismisses
- Cold-start skeleton appears (§6.4)
- Engine computes
- Tax Planning populates

### 7.4 Re-entry after onboarding

User dismisses sheet without submitting → empty state shows:

```
┌── Strategy not set up ──────────────────────────────┐
│  Set your assumptions to see your multi-year plan.  │
│         [Set up strategy]                           │
└─────────────────────────────────────────────────────┘
```

Single button re-opens the assumptions sheet.

User can change all six values later via the pill bar (assumptions 1–4 and 6) or Advanced settings (taxable balance, HSA balance — these don't go on the pill bar because they're set-once values, not assumption knobs).

Wait — that's not quite right. Re-reading: taxable balance and HSA balance ARE on `MultiYearAssumptions`, but they're "current state" values (not knobs). They live in Advanced settings sheet. Annual expenses lives there too.

| Value | Where it's edited post-onboarding |

## 8. Persistence

### 8.1 `MultiYearAssumptions` storage

Per engine spec §5.2 — single optional field on `Scenario`. `PersistenceManager` extends serialization. Backward compatible: 1.9 scenarios load with `multiYearAssumptions == nil`, default populated on first Tax Planning visit.

### 8.2 `dismissedInsightKeys`

`Set<String>` field on `MultiYearAssumptions`. Each key is a stable hash of insight content (e.g., `"ss-nudge-spouse-67-23k"`). Persisted with assumptions.

### 8.3 What's NOT persisted

- Engine results (`MultiYearStrategyResult`) — recomputed on launch
- `isComputing`, `hasEverComputed` — session-scoped
- Get Started banner dismissal (separate flag on `DataManager` or per-user defaults)

## 11. Out of scope (deferred to v2.1+)

| Item | Reason |
|---|---|
| Per-slider reset (↺ icon per slider) | Engine doesn't support partial constrained re-optimization within Year 1's inner loop. Reverting one slider while others stay overridden produces mathematically incoherent recommendations. v2.1 candidate IF engine adds per-lever locking. |
| Auto-restore dismissed insights on substantial change | Adds change-detection complexity. V2.0 dismissals persist until manual restore via Advanced. |
| Year-1-as-non-current-year (Dec 31 edge case) | User opens app on Dec 31 wanting "next year as Year 1." V2.0 invariant: Year 1 = current calendar year. v2.1 candidate. |
| LTCG / qualified dividends preferential treatment | Engine V2.0 simplification: all non-wage/non-pension/non-SS taxable income taxed as ordinary. UI surfaces what the engine produces. v2.1 will refactor engine inputs to consume `[IncomeSource]` directly. |
| Joint-owned IRA support | `MultiYearInputAdapter` filters by `.primary` / `.spouse`. `.joint` ownership silently dropped (mislabeling — IRAs are individual). v2.1 candidate: UI validation preventing `.joint` on IRA accounts. |
| SS claim age optimization | Engine takes claim age as input; doesn't optimize over it. SS claim nudge is informational. v2.1 candidate. |
| Off-plan acknowledgment / "intentional override" mode | Suppress the off-plan indicator after user explicitly accepts the deviation. V2.0 keeps the indicator visible until the override is reset. |
| Confidence-weighted insight dismissal friction | "Are you sure?" modal for $100K+ insights rejected as paternalism. Visual weight scaling sufficient for V2.0. |
| `forYear` parameter on engine `cliffCandidates(...)` | Currently unused parameter — separate engine cleanup. |
| Per-spouse SS claim ladder UI | Pill shows aggregate `SS 67/67`. v2.1 may add a more granular per-spouse ladder editor. |

## 13. Post-launch UX research candidates

Decisions made for V2.0 ship that are worth empirical testing post-launch:

1. **iPhone tab placement:** rightmost (current spec) vs. center. Rightmost chosen for V2.0 based on LTR setup→destination flow + thumb reachability. Center placement may better signal "decision engine, not data-entry app." A/B test post-launch.
2. **Heir Tax Rate label:** "Heir Tax Rate" vs alternatives (e.g., "Tax on Remaining Assets," "Terminal Tax Rate," "Future Tax Rate"). Current label optimizes for precision over comprehension. Single-term test post-launch if user research shows comprehension fail.
3. **Off-plan threshold tuning:** $1K minor / $25K major thresholds are intuition, not data. Tune based on actual user override magnitudes once we have telemetry.
4. **Default detent on iPhone Year 1:** `.medium` vs `.15`. `.medium` chosen so the lever editor is immediately visible. If users complain about screen real estate, test `.15` default with hint text.
5. **Debounce duration:** 500ms is industry standard. Tune based on observed user behavior — too long feels laggy; too short causes excessive recomputes.
6. **Visual weight scaling for callout banners:** thresholds for `.minor`/`.moderate`/`.major` impact buckets are intuition. Tune based on actual savings distributions.

These are notes to the team, not blockers for V2.0.
