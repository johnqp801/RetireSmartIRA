## 1. Goal & relationship to engine spec

This spec covers the SwiftUI surface that consumes `MultiYearTaxStrategyEngine`. It defers all math, optimization behavior, constraint acceptance, sensitivity bands, and engine API definitions to the engine spec. The engine produces a `MultiYearStrategyResult`; this spec defines what we render with it.

**User promise (unchanged from engine spec):** *"Show me how to minimize lifetime tax across my retirement years, with full transparency about which cliffs and tiers the plan accepts as trade-offs."*

**What changed from engine-spec ¬ß4:**

- ¬ß4 sketched a single-tab "Multi-Year Strategy" view with the 1.9 Reduce-AGI dashboard "embedded verbatim" in Year 1 expansion. That premise turned out to be flawed: the 1.9 surface is two views (3,681-line `TaxPlanningView` for scenario building + 2,628-line `DashboardView` for projection), and ¬ß4 only contemplated the latter. Naively replacing the 1.9 tabs would have deleted the scenario-builder UX entirely.
- ¬ß4 specified an "override delta panel" (Accept / Revert flow). This flow assumes a submit-style interaction; the V2.0 model is direct manipulation. Delta panel is dropped.
- ¬ß4 placed Multi-Year Strategy as the primary tax tab, leaving open whether it replaced one or both of Scenarios + Tax Summary. This spec resolves: it replaces both (consolidated into a single "Tax Planning" tab).
- ¬ß4 did not address mobile execution. This spec adds the bottom-sheet pattern for iPhone / iPad compact.

## 2. Information architecture

### 2.1 Tab consolidation

Today's 10 tabs include:

- Tab 5 **Scenarios** ‚Äî `TaxPlanningView`, the scenario builder (Roth conversion sliders, withdrawal sliders, QCD, charitable, deduction comparison)
- Tab 6 **Tax Summary** ‚Äî `DashboardView`, the projection display (bracket charts, IRMAA tier, NIIT chart, account balances)

Both fold into a single new tab **Tax Planning**. The new tab IS the Multi-Year Strategy view. No separate "Multi-Year" tab. No segmented control between "This Year" and "Multi-Year." Multi-year is the canonical view; Year 1 (the current year) is its actionable slice.

**Net tab count: 10 ‚Üí 9.**

### 2.2 Sidebar restructure (macOS / iPad regular)

The existing sidebar has three sections (Setup / Analysis / More). The Analysis section becomes:

```
Section "Analysis":
  RMD Calculator
  Tax Planning      ‚Üê new, replaces Scenarios + Tax Summary
```

No new section header. The macro/micro split doesn't need to be expressed in the IA ‚Äî it's expressed inside the Tax Planning view itself.

### 2.3 iPhone tab order

Current visible-5 (tabs 0‚Äì4): Get Started, Profile, SS, Income, Accounts. Tax Planning would land in the More overflow menu.

This is wrong: Tax Planning is the V2.0 hero feature; burying it behind an extra tap undercuts the launch story.

**New visible-5:** `[Profile, Income, Accounts, SS, Tax Planning]`.

- **Profile leftmost** ‚Äî natural identity-first position; first-stop for new users.
- **Tax Planning rightmost** ‚Äî destination position. Returning users (the ~95% case) reach the right edge with thumb muscle memory.
- **Get Started removed from visible-5** ‚Äî onboarding checklists are ephemeral. See ¬ß2.4 below.
- More overflow menu retains: RMD Calculator, Quarterly Tax, State Comparison.

Center placement of Tax Planning was considered (see ¬ß14 post-launch UX research candidates) but rejected for V2.0 because it interrupts the LTR setup ‚Üí destination flow.

### 2.4 Get Started migration

`GuideView` (Tab 0 today) becomes a **dismissible banner at the top of `SettingsView` (Profile tab)**.

- Banner shows the existing setup checklist (Profile complete ¬∑ Accounts added ¬∑ Income added ¬∑ SS info entered).
- Banner is **dismissed permanently** when the user taps a "Got it" / √ó control, OR automatically when all four checks are true and the user has visited Tax Planning at least once.
- Dismissal flag stored on `DataManager` (or wherever first-run state lives today).

The standalone `GuideView` and Tab 0 are removed from ContentView.

### 2.5 Default tab on app launch

After onboarding completes, default selected tab on launch is **Tax Planning** (the V2.0 hero). For a new user with incomplete setup, Tax Planning's empty state deep-links them back to the missing setup tabs.

## 4. Components

Components are listed top-to-bottom by their position in the macro pane, then the right-pane / sheet contents.

### 4.1 Pill bar

Horizontal scrollable strip at top of view. Pills represent current `MultiYearAssumptions` values.

| Pill | Type | Tap behavior | Default |
|---|---|---|---|
| `CPI 2.5%` | Numeric | Popover stepper (range 0‚Äì6%, 0.1% step) | 2.5% |
| `Growth 6%` | Numeric | Popover stepper (range 0‚Äì10%, 0.5% step) | 6% |
| `End age 95` | Numeric | Popover stepper per-spouse if MFJ (range 80‚Äì105) | 95 |
| `SS 67/67` | Enum | Popover picker per spouse (62‚Äì70) | 67 each |
| `Rule: Tax-eff.` | Enum | Popover picker (4 options: Tax-efficient ¬∑ Deplete-trad ¬∑ Preserve-Roth ¬∑ Proportional) | Tax-efficient |
| `Heir Tax Rate: 22%` (featured pill ‚Äî distinct color) | Numeric | Popover stepper (range 0‚Äì37%, 1% step) + info hint expandable | 22% |
| `‚ö° Stress test ON` (toggle pill) | Boolean | Tap to flip | ON |
| `‚ãØ Advanced` | Sheet | Opens settings sheet (cliffBuffer, restore dismissed insights, debug) | ‚Äî |

**Heir Tax Rate info hint:** Tappable (i) icon on the pill expands inline:
> *The tax rate we model your heirs (or you, at end of plan) paying on remaining traditional IRA balances. Higher rate ‚Üí engine recommends more aggressive Roth conversions now.*

Closes on tap-outside or after 8 seconds.

**Featured pill styling:** `Heir Tax Rate` uses a distinct background color (matching the iPad mockup's `#d8e7ff` accent) to signal it's the dominant lever on engine aggressiveness. Per [memory/v2-status.md](../../../.claude/projects/-Users-johnurban-Projects-RetireSmartIRA/memory/v2-status.md): "Default 0.22 is reasonable but it's the dominant lever ‚Ä¶ without UI, the engine is a black box."

**Hit-testing:** remains enabled during recompute. The cancel-restart debounce pattern (¬ß6.2) handles concurrent pill changes safely.

### 4.2 Optional callout banners

Appear at the top of the macro pane, below the pill bar, only when applicable:

- **SS claim nudge banner** ‚Äî renders when `result.ssClaimNudge != nil`. Format: `"Delaying spouse's claim from 64 ‚Üí 67 could save ~$23K lifetime. [Re-run with 67]"`
- **Widow stress callout** ‚Äî renders when `result.widowStressDelta.delta > $50K` (threshold tunable; not every household needs this surfaced). Format: `"If [higher earner] dies first, surviving spouse pays ~$X more lifetime tax under single-filer rates."`

**Visual weight scales with impact.** Higher-$ insights get larger banner size and bolder color treatment. Implementation: a single `bannerImpactLevel` enum (`.minor` / `.moderate` / `.major`) drives styling.

**Dismissal:**
- Single-tap dismiss control labeled "Dismiss" (not unadorned √ó) for major-impact banners ‚Äî makes act more deliberate. Minor-impact banners use √ó.
- Dismissed state stored in `MultiYearAssumptions.dismissedInsightKeys: Set<String>`. Each key is a content hash (e.g., `"ss-nudge-spouse-67-23k"`).
- Inline "Why this matters" expansion on every banner ‚Äî context without dismissing.
- "Restore dismissed insights" button in Advanced settings (`‚ãØ` pill) clears the dismissal set.
- **No confirmation modal, no friction-delay**, even on $100K+ insights. Users dismiss with reason; we trust.

### 4.3 Hero stat + objective caption + off-plan indicator

Vertically stacked block:

```
LIFETIME TAX                          [‚úì On plan]
$487,200 ‚Üí $312,400
Saves $174,800 over 30 years

Seeks to minimize lifetime tax including a 22% tax on assets
left to heirs. Accepts IRMAA or ACA cliffs only when
they save more than they cost.
```

The "22%" in the caption reads from `MultiYearAssumptions.terminalLiquidationTaxRate` and updates live when the user changes the heir-rate pill.

**Off-plan indicator** is a 4-state badge in the top-right corner of the hero:

| State | Trigger | Treatment | Inline action |
|---|---|---|---|
| ‚úì On plan | All Year 1 overrides match engine recommendations | Green check, no delta | ‚Äî |
| ‚úì Near optimal | <$1,000 lifetime deviation | Green check, "Near optimal (-$340)" | ‚Äî |
| ‚öÝ Off plan | $1,000‚Äì$25,000 deviation | Yellow, "Off plan: -$8,400" | `[Reset]` link |
| ‚öÝ Significantly off plan | >$25,000 deviation | Red, "Significantly off plan: -$45,000" | `[Reset]` link |

**Threshold constants** (`Year1OverrideThresholds.minor = 1_000`, `.major = 25_000`) live on `MultiYearStrategyManager` for post-launch tuning.

**Computing the delta:** `MultiYearStrategyManager` holds two cached results:
- `engineOptimalResult: MultiYearStrategyResult` ‚Äî re-run only when `MultiYearAssumptions` changes
- `currentResult: MultiYearStrategyResult` ‚Äî re-run whenever any input including user overrides changes (debounced, see ¬ß6)

Delta = `currentResult.lifetimeTaxFromRecommendedPath - engineOptimalResult.lifetimeTaxFromRecommendedPath`.

### 4.4 Strategy summary card

Plain-English 2‚Äì3 sentence narrative of the engine's recommended path, derived from `result.recommendedPath`. Synthesized server-side (in the Manager, not the View) by a helper that examines key actions:

> *Convert $50‚Äì70K/yr to Roth through 2030 (pre-SS bracket-fill), claim SS at 67, then RMD-fill the 12% bracket. Accepts 1 IRMAA Tier 2 hit at 2031.*

Helper logic: identify the highest-frequency LeverAction in the path, cluster by year ranges, name SS claim events, count constraint hits. Plan should specify the exact synthesis grammar (sentence templates per cluster type).

### 4.5 Waterfall chart

Swift Charts `BarMark` per year, stacked by tax component (`federal`, `state`, `irmaa`, `acaPremiumImpact`). Sensitivity bands (¬±2pp growth, when `stressTestEnabled`) overlaid as a shaded ribbon using `RectangleMark` between `result.sensitivityBands.optimistic` and `.pessimistic` totals.

Tap a bar ‚Üí swap the year detail pane to that year (master-detail) or change the bottom sheet's selected year (mobile).

X-axis: year; Y-axis: total tax in $. Legend: Federal / State / IRMAA / ACA.

Stale-state behavior per ¬ß6.3.

### 4.6 Sparkline 4-up

Four side-by-side cards: Trad IRA, Roth, Taxable, HSA. Each shows:
- Current-year balance (from `result.recommendedPath[0].endOfYearBalances`)
- Tiny SVG/Swift Charts line of the 30-year trajectory (one data point per year)

Read-only. Not interactive.

### 4.7 Trade-offs accepted card

Yellow / warning style. Renders top 3‚Äì5 entries from `result.tradeOffsAccepted` ranked by `cost` descending.

Each row:


If `tradeOffsAccepted.isEmpty`, the card hides entirely (cleaner than rendering "No trade-offs").

### 4.10 `Year1QuickEditor` (new view)

Curated lever editor for the V2.0 surface. ~300‚Äì500 LOC. Hosted in:
- iPad/macOS master-detail right pane: at top, above embedded TaxPlanningView/DashboardView
- iPhone bottom sheet at `.medium`: the entire sheet content
- iPhone bottom sheet at `.large`: at top, above embedded TaxPlanningView/DashboardView

**Contents** (curated subset of `TaxPlanningView`):

| Element | Source |
|---|---|
| **"Restore engine optimal" button** | New ‚Äî single tap wipes all Year 1 overrides, restores engine's coherent set |
| Roth conversion slider (you) | Reuses DataManager binding `dataManager.yourRothConversion` |
| Roth conversion slider (spouse), if MFJ | `dataManager.spouseRothConversion` |
| Withdrawal slider (you) | `dataManager.yourExtraWithdrawal` |
| Withdrawal slider (spouse), if MFJ | `dataManager.spouseExtraWithdrawal` |
| QCD field (you) ‚Äî only if `dataManager.isQCDEligible` | `dataManager.yourQCDAmount` |
| QCD field (spouse) ‚Äî only if MFJ + `dataManager.spouseIsQCDEligible` | `dataManager.spouseQCDAmount` |
| **Live this-year impact strip** (4-up grid) | Federal ¬∑ State ¬∑ IRMAA tier indicator ¬∑ Bracket headroom |

**The "Restore engine optimal" button is the ONLY reset.** No per-slider ‚Ü∫ icons. Reasoning: the engine produces a synergistic Year 1 lever set (Roth ‚äó withdrawal ‚äó QCD jointly fill brackets without tipping cliffs). Per-slider reset would either revert to historical values that no longer fit the user's current constraint set (mathematically incoherent), or require partial constrained re-optimization (engine doesn't support per-lever locking within the inner loop). All-or-nothing reset is safe, cheap, and intellectually honest. Per-slider reset deferred to v2.1 IF the engine adds partial locking.

**Live impact strip updates synchronously** on every slider tick from DataManager properties (instant, ms-fast). The macro pane debounces (¬ß6.2).

**Charitable / stock donation / cash gift:** NOT in `Year1QuickEditor`. Available in the embedded `TaxPlanningView` below (master-detail right pane / sheet at `.large`). These are less frequently tweaked than the four primary levers.

## 5. State architecture

### 5.1 `MultiYearStrategyManager` ‚Äî new ObservableObject

Lives parallel to existing `DataManager` and `ScenarioStateManager`. Single instance, injected via `@EnvironmentObject` like the others.

```swift
@MainActor
final class MultiYearStrategyManager: ObservableObject {
    // Persisted assumptions
    @Published var assumptions: MultiYearAssumptions

    // Engine results ‚Äî two caches
    @Published private(set) var engineOptimalResult: MultiYearStrategyResult?
    @Published private(set) var currentResult: MultiYearStrategyResult?

    // Compute state
    @Published private(set) var isComputing: Bool = false
    @Published private(set) var hasEverComputed: Bool = false  // gates cold-start skeleton

    // Subscribers
    private var dataCancellable: AnyCancellable?       // ScenarioStateManager / DataManager changes
    private var debounceTask: Task<Void, Never>?
    private var inputHash: Int = 0  // hash of (StaticInputs, Assumptions) for cache validation
    private var optimalHash: Int = 0  // separate hash for engine-optimal cache

    // Public API
    func recompute(reason: RecomputeReason)            // request a recompute; debounces internally
    func resetYear1ToEngineOptimal()                   // wipes Year 1 overrides; triggers recompute
    func dismissInsight(_ key: String)                 // adds to dismissedInsightKeys
    func restoreDismissedInsights()                    // clears the set

    enum RecomputeReason {
        case assumptionsChanged    // recompute BOTH currentResult AND engineOptimalResult
        case overridesChanged      // recompute only currentResult
        case appLaunch             // initial compute on appear
    }
}
```

### 5.2 Two-cache strategy

The off-plan indicator (¬ß4.3) needs both an engine-optimal baseline and a current-with-overrides path. Naive approach: re-run optimization twice on every change. Wasteful ‚Äî the engine-optimal path doesn't change when only Year 1 overrides change.

**Cache validation:**
- `engineOptimalResult` invalidates on `MultiYearAssumptions` or scenario-static-input changes (anything that affects the optimal solution itself)
- `currentResult` invalidates on any of the above + Year 1 override changes

When user moves a slider:
- `currentResult` recomputes (debounced)
- `engineOptimalResult` stays cached (assumptions didn't change)

When user changes a pill (e.g., growth rate):
- `engineOptimalResult` recomputes (assumptions changed)
- `currentResult` recomputes too (its cache is also invalidated)

**Dirty-flag pattern (implementation requirement).** A naive cancel-and-restart can lose pending optimal-cache updates. Scenario: user changes a pill (assumptions invalidate; task A starts to recompute both caches) ‚Üí user touches a slider mid-compute (task A cancels; task B starts with `RecomputeReason.overridesChanged`) ‚Üí task B's reason flag does NOT include the pending optimal update from task A ‚Üí optimal cache stays stale, off-plan indicator shows wrong delta.

Fix: a `needsOptimalRecompute: Bool` flag on `MultiYearStrategyManager`, set whenever an assumption or static-input change is detected, cleared only after a successful optimal compute. Each detached compute task checks the flag, computes the optimal baseline if true, and clears it on success ‚Äî independent of the `RecomputeReason` that triggered the task.

```swift
@Published private(set) var needsOptimalRecompute: Bool = false

// In recompute task:
if needsOptimalRecompute {
    let optimal = MultiYearTaxStrategyEngine().compute(inputs: staticInputs, assumptions: assumptions)
    await MainActor.run {
        self.engineOptimalResult = optimal
        self.needsOptimalRecompute = false
    }
}
// Then compute current with overrides
let current = MultiYearTaxStrategyEngine().compute(inputs: inputsWithOverrides, assumptions: assumptions)
```

This guarantees a cancelled task's pending optimal update is picked up by the next task ‚Äî no race window where the optimal cache silently goes stale.

### 5.3 Hash key composition

```swift
// engineOptimalHash: assumptions + scenario-static-inputs only (NO user overrides)
let optimalKey = MultiYearStaticInputs(...) ‚äï MultiYearAssumptions(...)

// currentHash: same as above PLUS Year 1 override values
let currentKey = optimalKey ‚äï Year1Overrides(roth, withdrawal, qcd, ...)
```

Both implemented via `Hashable` conformance on the input structs. `hashValue` cached per recompute to avoid recomputing inputs hash on every check.

**Floating-point hash hardening (implementation requirement).** Slider-bound `Double` values drift by sub-cent amounts (e.g., a slider returning `50_000.000001` after a round-trip drag-release). Naive `Hashable` conformance on raw `Double` values produces different hashes for visually-identical states, causing constant cache misses. Fix: in the `Hashable` implementation of `Year1Overrides` (and any other slider-bound override structs), round each `Double` to the nearest dollar before mixing into the hasher:

```swift
extension Year1Overrides: Hashable {
    func hash(into hasher: inout Hasher) {
        hasher.combine(roth.rounded())
        hasher.combine(withdrawal.rounded())
        hasher.combine(qcd.rounded())
        // ... per-spouse equivalents
    }

    static func == (lhs: Year1Overrides, rhs: Year1Overrides) -> Bool {
        lhs.roth.rounded() == rhs.roth.rounded()
            && lhs.withdrawal.rounded() == rhs.withdrawal.rounded()
            && lhs.qcd.rounded() == rhs.qcd.rounded()
        // ...
    }
}
```

Engine input precision is dollar-level anyway; sub-dollar drift is noise.

### 5.4 ScenarioStateManager subscription

`MultiYearStrategyManager` subscribes to changes in `DataManager` and `ScenarioStateManager` via Combine:

```swift
init() {
    dataCancellable = Publishers.Merge(
        dataManager.objectWillChange.eraseToAnyPublisher(),
        scenarioStateManager.objectWillChange.eraseToAnyPublisher()
    )
    .sink { [weak self] _ in
        self?.recompute(reason: /* derived from what changed */)
    }
}
```

The `reason` is derived by comparing pre/post hashes ‚Äî implementation detail in plan.

### 5.5 Persistence

`MultiYearAssumptions` is persisted on `Scenario` as a single optional field (per engine spec ¬ß5.2). `PersistenceManager` extends to serialize/deserialize the new struct. Backward compatible: 1.9 scenarios load with `multiYearAssumptions == nil`. On first view of Tax Planning when nil, the onboarding assumptions sheet (¬ß7.3) presents to populate the struct ‚Äî defaults are *suggested* in the sheet, not silently applied.

Engine results are NOT persisted ‚Äî recomputed on launch when needed. The compute is fast enough (<2s) that a cold start on app launch is acceptable.

## 6. Async recompute model

### 6.1 Why decouple

The engine runs in ~1.5‚Äì2.7s on M1. Tying every UI tick to that is unusable. But Year 1's micro feedback loop (slider tweak ‚Üí see this-year tax change) must remain instant ‚Äî that's the V2.0 promise.

**Solution:** decouple recompute timing per pane.

| Pane | Source | Latency |
|---|---|---|
| Right pane / sheet (Year 1 detail, live impact strip) | `DataManager` computed properties (`scenarioFederalTax`, etc.) | Synchronous, ms-scale |
| Macro pane (waterfall, hero, year list, sparkline) | `MultiYearStrategyManager.currentResult` | Async, debounced |

Right pane updates every slider tick. Macro pane updates ~500ms after the user stops moving sliders.

### 6.2 Debounce mechanism

```swift
func recompute(reason: RecomputeReason) {
    debounceTask?.cancel()
    isComputing = true  // immediate visual signal ‚Äî see ¬ß6.3

    debounceTask = Task { [self] in
        try? await Task.sleep(nanoseconds: 500_000_000)  // 500ms
        guard !Task.isCancelled else { return }

        let staticInputs = MultiYearInputAdapter.build(...)
        let result = await Task.detached(priority: .userInitiated) {
            MultiYearTaxStrategyEngine().compute(inputs: staticInputs, assumptions: self.assumptions)
        }.value

        await MainActor.run {
            self.currentResult = result
            // also update engineOptimalResult if reason warrants
            self.isComputing = false
            self.hasEverComputed = true
        }
    }
}
```

Each new recompute cancels the prior debounceTask. Engine call runs on a detached Task to avoid blocking the main actor.

### 6.3 Stale-state visualization

When `isComputing == true`:

- Macro pane drops to **70% opacity** (`opacity(isComputing ? 0.7 : 1.0)` modifier on the macro container).
- **"Recomputing strategy‚Ä¶"** pill appears in the top-right corner of the macro pane (overlay).

**Hit-testing is NOT disabled.** The cancel-restart debounce pattern (¬ß6.2) handles all concurrent interactions safely:

- Pill change mid-compute ‚Üí cancels current task, restarts with new assumption. Same pattern as the initial recompute.
- Year list tap mid-compute ‚Üí updates `selectedYear` view state (no recompute trigger). User sees fresh year detail in the right pane / sheet immediately; macro pane finishes its current compute and updates regardless.
- Slider tweak mid-compute ‚Üí updates DataManager (instant), restarts the debounce timer. Same pattern.
- Reset button tap mid-compute ‚Üí mutations replace overrides, restarts debounce. Idempotent on repeat taps (no overrides to clear after first tap).

Disabling hit-testing during compute would frustrate users mid-iteration without preventing any actual race condition. Visual stale-state (opacity + "Recomputing‚Ä¶" pill) is sufficient signal that a recompute is pending.

Sliders in `Year1QuickEditor` are always interactive ‚Äî they update DataManager state (instant), and the next debounce cycle picks up their final values.

### 6.4 Cold-start skeleton

When `hasEverComputed == false` (first compute after onboarding):

- Macro pane shows shimmer placeholders:
  - Hero: gray shimmer block where numbers would be
  - Waterfall: gray shimmer bars at uniform heights
  - Sparkline: gray shimmer rectangles
  - Year list: 4‚Äì5 gray shimmer rows
- No "Recomputing‚Ä¶" pill (the user hasn't moved anything yet ‚Äî the skeleton itself signals "computing your plan")

Once `currentResult != nil`, skeleton fades out over 200ms, real content fades in.

For subsequent recomputes (`hasEverComputed == true`), use the stale-state visualization from ¬ß6.3 instead of the skeleton ‚Äî preserves continuity.

### 6.5 Error handling

The engine itself doesn't throw, but compute can fail if `MultiYearStaticInputs` is malformed (missing filing status, missing IRA accounts, etc.). The setup-gap empty state (¬ß7.2) prevents most cases. For edge cases:

- Engine returns `nil` or compute Task throws ‚Üí show an error banner: "Strategy couldn't be computed. [Details]" with a tappable details link to a sheet listing the missing/invalid inputs.
- Macro pane shows the previous valid `currentResult` if one exists, or the skeleton + error banner if cold start.

No retry UI ‚Äî assumption is that fixing inputs auto-triggers a fresh compute via the Combine subscription.

## 11. Migration from 1.8

The 2.0 App Store release bundles all 1.9 internal-checkpoint work plus 2.0 multi-year. For public users this is a single 1.8 ‚Üí 2.0 update ‚Äî a substantial visual + functional jump (1.9's design refresh + correctness fixes + 2.0's multi-year surface, all at once).

### 11.1 Default-tab branching on app launch

Determines whether a user lands on My Profile (with Get Started banner) or Tax Planning. Computed once in `ContentView` at launch:

```swift
let setupComplete = dataManager.filingStatus != nil
                 && !dataManager.iraAccounts.isEmpty
                 && dataManager.primarySSBenefit != nil

selectedTab = setupComplete ? .taxPlanning : .myProfile
```

| User type | Setup complete? | MultiYearAssumptions | Default landing | First experience |
|---|---|---|---|---|
| 1.8 migrating, common case | ‚úì | nil | Tax Planning | Year 1 sliders visible + locked-pane overlay |
| 1.8 migrating, partial setup | ‚úó | nil | My Profile | Get Started banner with checklist |
| Brand new install | ‚úó | nil | My Profile | Get Started banner ‚Äî full checklist guides through Profile ‚Üí Income ‚Üí Accounts ‚Üí SS, then nudges to Tax Planning |
| Returning V2.0 user | ‚úì | set | Tax Planning | Standard view (macro + detail) |

### 11.2 Data preservation invariants

All 1.8 user data flows automatically into 2.0 with no re-entry:

- **Profile** (`SettingsView` data): filing status, ages, state, ACA enrollment year, Medicare enrollment age ‚Äî preserved.
- **IRA accounts**: balances, types (Traditional / Roth / Inherited), owner attribution ‚Äî preserved.
- **Income sources**: consulting, pension, dividends, qualified dividends, interest, capital gains (short / long), state tax refund, other ‚Äî all preserved. The 2.0 engine adapter (`MultiYearInputAdapter`) reads ALL these income types into `primaryOtherOrdinaryIncome` / `spouseOtherOrdinaryIncome` per the engine's Round 6 fix (see [memory/v2-status.md](../../../.claude/projects/-Users-johnurban-Projects-RetireSmartIRA/memory/v2-status.md) bug #11). Pre-existing 1.8 income data flows correctly without user intervention.
- **Social Security plans**: PIA, planned claiming age, COLA assumptions ‚Äî preserved.
- **Existing scenario state**: Roth conversion amounts, withdrawal amounts, QCD amounts, charitable settings, deduction overrides ‚Äî all preserved on `ScenarioStateManager`. Flow into Year 1 of the multi-year plan automatically.
- **Saved scenarios**: persisted scenarios continue to load. Schema delta is a single optional field (`multiYearAssumptions`); older scenarios load with `nil`.

### 11.3 Locked-pane macro states

When a user has complete setup but `MultiYearAssumptions == nil`, the macro pane has three render states. The Year 1 detail pane / sheet remains fully usable in all three.

**State A ‚Äî Overlay (initial post-setup state):**

Macro pane content (waterfall, hero, sparkline, year list) renders blurred behind a centered overlay card:

> **Unlock your 30-year strategy**
> RetireSmart IRA now projects your tax plan decades into the future. Add a few details to see your lifetime savings.
> `[ Set up multi-year plan ]`  `[√ó]`

Tap primary button ‚Üí assumptions sheet (¬ß7.3). Submit ‚Üí State C.

Tap √ó ‚Üí State B.

**State B ‚Äî Slim banner (overlay dismissed, still locked):**

Compact dismissible banner at the top of the macro pane area:

> üìà Multi-year strategy locked ¬∑ `[Set up]`

Macro pane content remains blurred behind. Tap banner ‚Üí assumptions sheet ‚Üí State C. Banner persists across sessions until assumptions are set; never re-promotes to overlay (State A) once dismissed.

**State C ‚Äî Unlocked:**

Standard macro pane render. Locked-state machinery removed entirely. Cold-start skeleton (¬ß6.4) shown briefly during initial compute, then real content fades in.

**Transition:** State A or B ‚Üí State C: blur cross-fades out over ~400ms after assumptions submit; cold-start skeleton replaces blur for engine compute (~1.5s); real content fades in.

**Self-explanatory by design:** the locked-pane overlay alone communicates "new feature here, opt in when curious." No separate "What's New" modal needed. The user's familiar Year 1 sliders being immediately usable in the detail pane / sheet communicates "your existing scenarios live here now." No tab-muscle-memory banner needed.

### 11.4 Onboarding inputs (1.8-aware framing)

The six-field assumptions sheet (¬ß7.3) applies to migrating users. Migration-specific defaults and hints:

| Field | Default | Hint copy |
|---|---|---|
| Taxable balance | $0 | "Total in non-retirement brokerage accounts." |
| HSA balance | $0 | "Skip if you don't have an HSA." |
| Annual expenses | $60,000 | "Approximate annual spending in retirement (today's dollars). You can refine later." |
| Horizon end age | 95 | per ¬ß7.3 |
| Withdrawal preset | Tax-efficient | per ¬ß7.3 |
| Heir tax rate | 22% | per ¬ß7.3 |

**No reverse-engineered estimates.** Taxable balance is asked plainly without dividend-yield-derived suggestions. The engine is precision-sensitive: a $20K dividend stream could imply $300K (dividend-focused funds) or $1.4M (broad-market index, ~1.4% yield) ‚Äî a 4‚Äì5√ó range. Anchoring on a hallucinated default is materially worse than asking for the actual number.

### 11.5 First-impression: neutral off-plan copy

After the user submits assumptions and the macro pane unlocks, the engine compares the user's existing scenario (carried forward from 1.8) against the engine-optimal multi-year plan. Deltas are usually material ‚Äî single-year tuning differs from multi-year optimum.

**On the first off-plan moment** (first time the indicator would render in a non-on-plan state for a migrating user), use neutral framing:

> **Your plan: $321K ¬∑ Engine recommends: $312K**
> `[See engine's plan]`

Information, not warning. No yellow color. No "Off plan" verb.

After the user has interacted with the indicator (moved a slider, dismissed it explicitly, or tapped "See engine's plan" and either accepted or returned), the indicator switches permanently to the standard 4-state warning behavior (¬ß4.3).

A `firstOffPlanShown: Bool` flag on `MultiYearStrategyManager` (persisted) tracks the one-time transition.

### 11.6 What's NOT in the migration UX

Explicitly omitted to avoid popup fatigue:

- **No "What's New in 2.0" modal** ‚Äî App Store release notes communicate version-level changes; the locked-pane overlay self-explains the new feature in context.
- **No "Tab moved" banner** ‚Äî the Tax Planning tab name + the user's familiar Year 1 sliders being immediately visible self-explain that Scenarios + Tax Summary consolidated here.
- **No "Welcome back" splash** ‚Äî opening to a working app is the warmest welcome.

The 1.8 user's first 2.0 launch from app store update: open app ‚Üí land on Tax Planning ‚Üí see familiar Year 1 sliders behind / beside a blurred macro with one overlay. Dismiss-actions required before usable: zero (Year 1 sliders are immediately interactive; multi-year is opt-in).

### 11.7 Failure modes (cross-references)

- **Setup-gap empty state** (¬ß7.2) ‚Äî defensive fallback when a user manually navigates to Tax Planning before completing setup. Not the primary new-user path; the Get Started banner gates that.
- **Engine compute failure** (¬ß6.5) ‚Äî malformed inputs surface an error banner.
- **Corrupted scenario data** ‚Äî existing 1.8 fallback behavior preserved (load-fresh-scenario flow); no V2.0-specific logic.

### 11.8 Backward compatibility

- **2.0 reads 1.8-saved data:** required, handled via the optional `multiYearAssumptions` field on `Scenario` and the existing fields on `DataManager` / `ScenarioStateManager`.
- **1.8 reads 2.0-saved data:** not supported. Standard App Store policy ‚Äî users update forward only.

### 11.9 Migration scope explicitly NOT in V2.0

Deferred to v2.1+:

- Migration analytics (assumptions-completion rate, time-to-unlock, dismissal rates) ‚Äî instruments the migration's success but not required for ship.
- In-app changelog / "What changed in 2.0" detail page ‚Äî App Store release notes carry this for V2.0.
- Animated transition from 1.8 tab structure to 2.0 ‚Äî pure polish; cold open is fine.

## 14. Post-launch UX research candidates

Decisions made for V2.0 ship that are worth empirical testing post-launch:

1. **iPhone tab placement:** rightmost (current spec) vs. center. Rightmost chosen for V2.0 based on LTR setup‚Üídestination flow + thumb reachability. Center placement may better signal "decision engine, not data-entry app." A/B test post-launch.

3. **Off-plan threshold tuning:** $1K minor / $25K major thresholds are intuition, not data. Tune based on actual user override magnitudes once we have telemetry.
4. **Default detent on iPhone Year 1:** `.medium` vs `.15`. `.medium` chosen so the lever editor is immediately visible. If users complain about screen real estate, test `.15` default with hint text.
5. **Debounce duration:** 500ms is industry standard. Tune based on observed user behavior ‚Äî too long feels laggy; too short causes excessive recomputes.
6. **Visual weight scaling for callout banners:** thresholds for `.minor`/`.moderate`/`.major` impact buckets are intuition. Tune based on actual savings distributions.

These are notes to the team, not blockers for V2.0.
