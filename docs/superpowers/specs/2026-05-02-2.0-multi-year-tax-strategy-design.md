## 1. Goal & user promise

Multi-Year Tax Strategy is the headline 2.0 feature. It extends the 1.9 Reduce-AGI dashboard from a single-year optimizer to a multi-year retirement-tax planner.

**User promise:** *"Show me how to minimize lifetime tax across my retirement years, with full transparency about which cliffs and tiers the plan accepts as trade-offs."*

**Competitive positioning.** Boldin, MaxiFi, Pralana, and Income Lab all lead with cash flow or balances. RetireSmartIRA 2.0 leads with the **tax-savings story** — and runs heavier optimization on Apple Silicon than any web tool can match (a strategic differentiator we exploit deliberately).

**1.9 + 2.0 ship together as a single 2.0 App Store release.** The 1.9 work is an internal checkpoint; the public release is 2.0.

## 2. Architecture overview

Three layers, mirroring the 1.9 pattern:

1. **Engine layer** — `MultiYearTaxStrategyEngine`, a pure-calculation module. Takes static scenario data + assumption inputs, returns a `MultiYearStrategyResult`. No SwiftUI dependencies; fully testable in isolation.

2. **State layer** — extends `ScenarioStateManager` (existing) with a new `MultiYearAssumptions` sibling. Static scenario data (accounts, income, ages, filing status, state) stays in `ScenarioStateManager`. New horizon/CPI/growth/preset inputs live in `MultiYearAssumptions`. Year-1 lever overrides write back to `ScenarioStateManager` when the user accepts them.

3. **UI layer** — `MultiYearStrategyView` becomes the **primary tax-planning surface**. The 1.9 `ReduceAGIDashboard` becomes the Year-1 detail expandable section *within* `MultiYearStrategyView`. Existing 1.9 ACA / Medicare / Tax-Calculator tabs remain as supporting calculators.

## 3. Engine design

### 3.1 Inputs

**From `ScenarioStateManager` (existing, static per scenario):**
- Account balances by type: Traditional IRA, Roth IRA, Taxable, HSA
- Income sources: wages (if still working), pensions, expected SS amounts at FRA
- Ages, filing status, state, ACA enrollment year, Medicare enrollment age
- SS claim age per spouse (input, not optimized)

**From new `MultiYearAssumptions`:**
- Horizon end age — default `95`, override per spouse
- CPI rate — default `2.5%`
- Investment growth rate — default `6%` nominal, single rate
- Withdrawal-ordering preset — default `Tax-efficient`; alternatives `Deplete-trad-first`, `Preserve-Roth`, `Proportional`
- Stress-test toggle — default ON
- Per-year living-expense overrides — keyed by year, default empty (smart-default expenses derived from existing 1.9 data)

### 3.2 Outputs

```swift
struct MultiYearStrategyResult {
    let recommendedPath: [YearRecommendation]
    let tradeOffsAccepted: [ConstraintHit]
    let sensitivityBands: SensitivityBands  // { optimistic, average, pessimistic }
    let widowStressDelta: TaxImpact
    let ssClaimNudge: ClaimAgeFlag?
}

struct YearRecommendation {
    let year: Int
    let agi: FederalAGI
    let acaMagi: ACAMAGI
    let irmaaMagi: IRMAAMAGI
    let taxableIncome: Money
    let taxBreakdown: TaxBreakdown   // { federal, state, irmaa, acaPremiumImpact }
    let accountBalances: AccountSnapshot  // post-year balances
    let recommendedActions: [LeverAction]  // Roth conversion, withdrawals, contributions, HSA
}

struct ConstraintHit {
    let year: Int
    let type: ConstraintType  // .irmaaTier(level), .acaCliff, .bracketOverrun(from:to:)
    let cost: Money
    let acceptanceRationale: String  // "Lifetime savings $18.4K > IRMAA cost $2.1K"
}

struct ClaimAgeFlag {
    let spouse: SpouseID
    let currentClaimAge: Int
    let suggestedClaimAge: Int
    let estimatedLifetimeTaxDelta: Money  // negative = savings
}
```

### 3.3 Optimization scope (Phase 0 → commit decision)

**Phase 0 perf prototype (Week 1, before any architecture commitment):**

Build a 1–2 day spike on M1 base model. Try full dynamic programming over `(year × bracket-bucket × IRMAA-tier × Roth-balance-bucket)` state space for a 30-year horizon at realistic state size (~10⁵ states). Measure wall-clock runtime end-to-end.

**Phase 0 decision tree:**
| Measured runtime | Commitment | Architecture |
|---|---|---|
| <5 sec | **Scope E** — full DP | Multi-year optimal across all dimensions |
| 5–15 sec | **Scope C+D** — greedy with 2-year lookahead heuristic | ~80% of optimal per published retirement-planning research |
| >15 sec | **Scope B** — greedy year-by-year with constraint awareness | Document the limitation; revisit in 2.1 |

The engine signature (inputs/outputs above) stays the same regardless of internal scope chosen. Downstream UI work proceeds identically.

### 3.4 Soft constraints — the trade-off model

The engine **does not block** actions that hit constraints. It accepts them when the lifetime trade is positive. Constraints surfaced as `ConstraintHit`:

- **IRMAA tier hit** — 2-year lookback (e.g., 2027 MAGI affects 2029 Medicare premium). Cost = additional Part B + Part D IRMAA in affected year(s).
- **ACA cliff** (ages <65 with ACA enrollment) — cost = lost subsidy if MAGI exceeds 400% FPL.
- **Bracket overrun** — flagged when crossing a marginal-rate jump (e.g., 12% → 22%) where the engine couldn't avoid it. Not technically a "cliff" but worth surfacing.

UI surfaces top 3–5 by impact in the trade-offs card.

### 3.5 Stress-test bands

Three full engine runs at growth rate = `avg`, `avg − 2pp`, `avg + 2pp`. Bands overlaid on lifetime tax total, portfolio trajectory, and bracket-fill chart. Single recommendation under `avg`; bands show the range. Toggle in the assumption pill bar.

### 3.6 Widow stress test

Separate single computation: lifetime tax delta if higher-earning spouse dies at age 75 (or scenario-configured average widowhood age). Surfaced as a discrete callout, not blended into the bands. Reason: widow penalty is a discrete event, not continuous variance.

### 3.7 SS claim-age nudge

After producing the main recommendation, the engine runs claim-age perturbations at `−2yr`, `−1yr`, `+1yr`, `+2yr` per spouse (4 extra runs per spouse, 8 total for couples). If any perturbation produces a lifetime tax difference >$5K, surface a `ClaimAgeFlag` with the largest-impact perturbation. UI shows it as an optional banner: *"Delaying spouse 1's claim from 64 to 67 could save ~$23K. [Re-run with 67]"*

The engine does **not** optimize over claim age — that's deferred to a future release. This is an information-only flag.

## 4. UX architecture

### 4.1 Page structure (top to bottom)

1. **Top-strip pill bar** — tappable pills for `CPI 2.5%`, `Growth 6%`, `Mortality 95`, `SS 67/67`, `Rule: Tax-efficient`, plus `⚡ Stress test ON` toggle. Inline popover on tap (no modal navigation).
2. **Hero stat** — `$487,200 → $312,400` with subtitle "Engine recommendation saves $174,800 lifetime."
3. **Strategy summary card** — 2–3 sentences in plain English. Example: *"Convert $50K–$70K/yr to Roth through 2030 (pre-SS bracket-fill), claim SS at 67, then RMD-fill the 12% bracket. Saves $174,800 lifetime."*
4. **Tax-savings waterfall** (hero chart) — stacked bars by year. Stack = Federal / State / IRMAA / ACA. Stress-test bands shown as a shaded ribbon. Click a year bar → expands per-year detail below.
5. **Account-balance sparkline** (secondary) — 4 lines (Trad IRA / Roth / Taxable / HSA), full horizon. Compact — ~20% the height of the waterfall.
6. **Trade-offs accepted card** — yellow/warning style. Top 3–5 `ConstraintHit`s ranked by cost.
7. **Per-year detail (expandable)** — clicking a waterfall bar expands a row inline. Shows AGI breakdown, recommended actions, account-balance delta, applicable warnings. **Year 1's expansion embeds the 1.9 Reduce-AGI dashboard verbatim** (preserves all 1.9 work as the year-1 implementation).
8. **Optional banners** (top of view, only when applicable):
   - SS claim nudge (if engine produced one)
   - Widow stress callout

### 4.2 Override flow

When the user changes a Year-1 lever (in the embedded Reduce-AGI dashboard, or via the per-year detail panel):

1. Engine recomputes immediately.
2. **Delta panel slides in** at the top of the page: side-by-side comparison.
   - *Original:* lifetime tax $312K, strategy summary text
   - *With your change:* lifetime tax $337K (+$25K), updated strategy summary
3. User clicks `[Accept revised plan]` → new path becomes canonical, writes back to `ScenarioStateManager`.
   Or `[Revert to original]` → discard the override.
4. While the delta panel is open, the chart shows both paths overlaid (ghosted original + solid new).

### 4.3 Navigation impact

- TaxPlanningView tab order: **Multi-Year Strategy** (new primary) | ACA Subsidy | Medicare Costs | Tax Calculator | (other 1.9 tabs)
- The standalone "Reduce AGI" tab is removed — its functionality lives inside Year-1 detail of Multi-Year Strategy.
- Lever overrides accepted in Year-1 detail update `ScenarioStateManager`, so the ACA / Medicare tabs see the new figures without manual sync.

### 4.4 Responsive behavior

- **Mac / iPad full width:** full layout as described.
- **iPad split / iPhone:** pill bar wraps to 2 rows; waterfall stays full width; sparkline stacks below; trade-offs card stacks below sparkline. Per-year detail expands to full width.

## 5. Data model & state changes

### 5.1 New types

- `MultiYearAssumptions` (struct, persisted per scenario) — horizon, CPI, growth, preset, stress toggle, per-year expense overrides
- `MultiYearStrategyResult` and nested types as in §3.2
- `WithdrawalOrderingRule` (enum) — `.taxEfficient`, `.depleteTradFirst`, `.preserveRoth`, `.proportional`
- `ConstraintType` (enum) — `.irmaaTier(level: Int)`, `.acaCliff`, `.bracketOverrun(from: Bracket, to: Bracket)`
- `LeverAction` (enum) — `.rothConversion(Money)`, `.traditionalWithdrawal(Money)`, `.taxableWithdrawal(Money)`, `.rothWithdrawal(Money)`, `.hsaContribution(Money)`, `.deferSocialSecurity`, etc.

### 5.2 Existing types — extensions

- `ScenarioStateManager` — adds optional `multiYearAssumptions: MultiYearAssumptions` field; persisted alongside existing scenario data
- `PersistenceManager` — extended to serialize/deserialize `MultiYearAssumptions`
- `Scenario` model gains a single new optional field; backwards-compatible with 1.9 saved scenarios (loads as `nil`, defaults populated lazily on first view of Multi-Year Strategy)

### 5.3 Caching

Engine output is cached on the hash of `(static-inputs ⊕ MultiYearAssumptions)`. Recomputed only when inputs change. Cache lives in memory per session; not persisted (compute time is short enough that a cold start on app launch is acceptable).

## 6. Out of scope — deferred to 2.1+

| Deferred | Why | What 2.0 does instead |
|---|---|---|
| Brokerage as a separate tax-drag bucket | Adds dimension; needs cost-basis tracking | Single investment growth rate applied to all accounts |
| Full SS claim-age optimization | Different problem class (one-time decision, mortality-driven) | Take SS claim age as input + flag if any ±1yr / ±2yr perturbation saves >$5K |
| Multi-session "actuals vs plan" tracking | Needs year-rollover workflow we don't have | Plan recomputed-on-demand; no historical tracking yet |
| Per-year per-account override granularity | Muddles "single recommendation" framing | User picks a withdrawal-ordering preset; engine optimizes within it |
| Monte Carlo / probabilistic mortality | High complexity | Single-point mortality + widow stress test as separate callout |
| Charitable / QCD strategies | Significant scope; small fraction of target audience | Future "advanced strategies" release |
| Inherited IRA optimization (T.D. 10001 10-yr rule) | Specialized; only applies to inheritors mid-rule | Separate "inherited IRA wizard" workstream |

## 8. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Apple Silicon perf can't support full DP | Medium | Forces fallback to greedy (less differentiation) | Phase 0 prototype answers this in Week 1 before commitment |
| Soft-constraint "accept" logic produces counterintuitive recommendations | Medium | Trust loss | Reference-case validation against published optimal scenarios; surface acceptance rationale in trade-offs card |
| Tax law changes between now and ship | Low (OBBBA recently passed, made TCJA permanent) | Hardcoded rules go stale | Tax constants centralized; documented update process |
| SwiftUI type-checker timeouts (1.9 recurrence) | High (large new view) | Build delays | Pre-emptive sub-view extraction; size budget per view file (<300 LOC) |
| Removing Reduce-AGI tab disrupts existing users | Low (1.9 hasn't shipped publicly yet) | UX confusion | 1.9 + 2.0 ship together; no installed base to migrate |
| Engine recomputation on every input change feels sluggish | Medium | Perceived poor responsiveness | Output cached on input hash; only recompute on dirty |

## 9. Testing strategy

- **Unit tests:** boundary conditions for IRMAA tiers (each threshold), ACA cliff (400% FPL exact), bracket transitions (every marginal-rate edge), RMD calculations, Roth conversion math.
- **Reference scenarios:** 5–10 hand-validated cases with known good answers, drawn from published retirement-planning research (Pfau, Kitces, BogleHeads). Engine must reproduce within 1% of expected lifetime tax.
- **Round-trip invariants:** every accepted `ConstraintHit` must satisfy `lifetime savings ≥ constraint cost` — encoded as a property test.
- **Snapshot tests:** key SwiftUI views (full plan rendered, override delta panel, stress-test bands), extending the 1.9 snapshot pattern.
- **Perf budget:** engine produces a recommendation in <2 sec on M1 base for 30-year horizon. Phase 0 measures whether this is achievable with full DP or requires medium scope.
