# 2.0 Optimizer Correctness Fixes — Design Spec

**Status:** Approved (2026-05-03)
**Branch target:** `2.0/multi-year-engine`
**Predecessor:** [2026-05-02-2.0-multi-year-tax-strategy-design.md](./2026-05-02-2.0-multi-year-tax-strategy-design.md)

## Origin

External code review by Gemini (web) on 2026-05-03 of `OptimizationEngine.swift` identified three correctness flaws in the Scope C+D forward-greedy optimizer. All three are confirmed by code inspection. Without these fixes, the optimizer systematically biases toward "do nothing" and cannot discover the multi-year Roth conversion ladders that are the engine's core value proposition.

## The three bugs

### Bug 1: Terminal Tax Illusion

**Location:** [OptimizationEngine.swift:147](../../RetireSmartIRA/OptimizationEngine.swift) and again at line 196.

**Problem:** The objective function is `lifetimeTax = sum of taxBreakdown.total over horizon years`. End-of-horizon traditional balance is invisible — leaving $2M in trad at year 30 records $0 tax, so the optimizer treats deferral as free.

**Impact:** Roth conversions trade present tax for tax-free future growth. With deferred tax counted as $0, the trade always looks bad. Optimizer recommends $0 conversions in nearly all scenarios.

### Bug 2: Greedy Trap

**Location:** [OptimizationEngine.swift:137](../../RetireSmartIRA/OptimizationEngine.swift).

**Problem:** When evaluating candidate $X for year Y, the inner loop fills all undecided future years with `[]` (i.e., $0 conversions). A single-year conversion of $X cannot meaningfully reduce a year-6 RMD bracket — multi-year ladder synergies are invisible to this enumeration.

**Impact:** Even with Bug 1 fixed, hand-computed sustained ladders (e.g., $100K/yr for 5 years) are rejected because each individual year's standalone benefit doesn't justify the standalone cost.

### Bug 3: Bracket-Edge Blindness

**Location:** [OptimizationEngine.swift:89-97](../../RetireSmartIRA/OptimizationEngine.swift) (the static candidate set).

**Problem:** Fixed candidates `[0, 25K, 50K, 75K, 100K, 150K, 200K]` are not aligned to any tax-law boundary. A user just under an IRMAA tier may have the optimizer jump them to $50K conversion, blowing past the cliff for $500-$2,000+/yr/person of penalty.

**Impact:** Single-cliff trips can erase thousands in annual savings. Already documented as v2.1 limitation #5 but more user-visible than originally appreciated.

## Scope

**In scope:**
- Fix Bugs 1, 2, and 3 in `OptimizationEngine.swift`
- Add two fields to `MultiYearAssumptions`: `terminalLiquidationTaxRate` (default 0.22) and `cliffBuffer` (default 5,000)
- Add cliff-aware candidate generator with filing-status-specific IRMAA / ACA / bracket thresholds
- New constants file `MultiYearTaxThresholds.swift`
- Three new scenario tests + one performance regression guard
- Update existing tests whose expected values change because the optimizer now finds ladders it previously rejected

**Out of scope:**
- Full Scope-E DP migration (v2.1)
- Per-hit constraint attribution rationale (v2.1 limitation #5; partially mitigated by per-year cliff candidates)
- Heir-bracket inheritance modeling (v2.1)
- SS taxation phase-in (Tax Torpedo) as a cliff candidate (v2.1)
- Capital gains 0%/15% bracket as a cliff candidate (v2.1; requires separate-buckets work)
- NIIT 3.8% threshold

## Success criteria

1. Hand-computed 7-year Roth ladder scenario (ages 60-66 pre-SS): engine produces a recommended path with cumulative conversions in the ladder years matching hand-math within the band specified in Test 1
2. ACA cliff scenario: engine does not propose a conversion that crosses the (FPL × 4 − $5K) threshold when avoidance is the higher-savings path
3. IRMAA buffer scenario: engine does not land MAGI in any (cliff − $5K, cliff) dead zone
4. All 933 pre-existing tests either still pass OR have updated expected values with documented justification (no test deleted)
5. 30-year horizon with full feature set runs in **< 3 seconds** on M1 base hardware (relaxed from 2.0's original < 2s budget per design decision)

## Architecture changes

### `MultiYearAssumptions` extension

```swift
extension MultiYearAssumptions {
    let terminalLiquidationTaxRate: Double  // default 0.22
    let cliffBuffer: Double                 // default 5_000
}
```

Defaults make all existing callers continue to work without source changes. UI surfacing (Plan B onboarding strip) deferred to Plan B work.

### `OptimizationEngine.swift` changes

1. **Outer fixed-point iteration loop** wrapping the existing inner forward-greedy loop. Cap = 5 iterations. Exit early on exact-equality convergence of `[Int: [LeverAction]]` locked dictionary.
2. **Inner-loop "fill futures"** changed from `[]` to "previous-iteration locked decisions for years Y+1..endYear". This is what fixes the Greedy Trap — the optimizer now sees what it intends to do in subsequent years when evaluating year Y.
3. **Objective function** changed from `sum(taxBreakdown.total)` to `sum(taxBreakdown.total) + terminalLiquidationTax(path)`. Applied at both inner-loop and post-loop acceptance comparison sites.
4. **Per-year cliff candidates** generated at the start of each outer iteration using a baseline projection's MAGI. Unioned with the static candidate set.
5. **Convergence telemetry:** if iteration 5 completes without convergence, log a debug warning. Accept the iteration-5 result.

### New file: `MultiYearTaxThresholds.swift`

Holds the base-year-2026 threshold tables. CPI-projected per call.

```swift
enum MultiYearTaxThresholds {
    // MFJ and Single tables for IRMAA tiers (5 each)
    static let irmaaThresholdsBaseYear2026: [FilingStatus: [Double]]

    // ACA 400% FPL by household size (1-8)
    static let acaFPLBaseYear2026: [Int: Double]

    // Ordinary tax bracket TOP boundaries by filing status
    // (12, 22, 24, 32, 35 — 37 has no top, omitted)
    static let ordinaryBracketTopsBaseYear2026: [FilingStatus: [Double]]

    static func project(_ value: Double, fromBaseYear: Int, toYear: Int, cpi: Double) -> Double
}
```

## Algorithm

```
optimize(inputs, assumptions) -> Result:
    baseYear = current year
    horizonYears = assumptions.horizonEndAge - inputs.primaryCurrentAge + 1
    endYear = baseYear + horizonYears - 1
    if horizonYears <= 0: return Result([], [])

    # Initial state: all years $0 (matches current code as starting point)
    locked = {y: [] for y in baseYear...endYear}

    # ──── Outer fixed-point iteration ────
    for iteration in 1...5:
        previousLocked = locked

        # ──── Inner forward greedy ────
        for Y in baseYear...endYear:
            # Compute baseline MAGI for year Y using locked decisions AS OF NOW.
            # Critical: must be inside the Y loop, NOT once per outer iteration —
            # within an iteration, locked[Y'] for Y' < Y has been updated in this
            # pass, and those updates change year Y's pre-conversion trad balance
            # and therefore year Y's MAGI. Using a per-iteration baseline would
            # generate stale cliff candidates and lock in wrong amounts.
            currentBaselinePath = ProjectionEngine.project(inputs, assumptions, locked)
            baselineMAGI = currentBaselinePath[Y - baseYear].magi
            cliffCands = cliffCandidates(
                forYear: Y, baseYear: baseYear,
                baselineMAGI: baselineMAGI,
                filingStatus: inputs.filingStatus,
                householdSize: inputs.acaHouseholdSize,
                assumptions: assumptions
            )
            candidates = (staticCandidates ∪ cliffCands).deduped(tolerance: 1_000)

            bestAmount = 0
            bestObjective = +∞
            for amount in candidates:
                trial = locked
                trial[Y] = amount > 0 ? [.rothConversion(amount)] : []
                # Future years keep their PRIOR-iteration locked values (greedy trap fix)
                path = ProjectionEngine.project(inputs, assumptions, trial)
                objective = sum(path[y].taxBreakdown.total)
                          + terminalLiquidationTax(path, rate: assumptions.terminalLiquidationTaxRate)
                if objective < bestObjective:
                    bestObjective, bestAmount = objective, amount

            locked[Y] = bestAmount > 0 ? [.rothConversion(bestAmount)] : []

        if locked == previousLocked: break  # converged

    # ──── Final projection + acceptance rationale ────
    finalPath = ProjectionEngine.project(inputs, assumptions, locked)
    hits = ConstraintAcceptor.detect(...)

    baselineActions = {y: [] for y in baseYear...endYear}
    baselinePath = ProjectionEngine.project(inputs, assumptions, baselineActions)
    baselineTotal = sum(baselinePath[y].taxBreakdown.total)
                  + terminalLiquidationTax(baselinePath, rate: ...)
    currentTotal  = sum(finalPath[y].taxBreakdown.total)
                  + terminalLiquidationTax(finalPath, rate: ...)
    lifetimeSavings = baselineTotal - currentTotal

    acceptedHits = []
    for hit in hits:
        if lifetimeSavings >= hit.cost:
            acceptedHits.append(formatRationale(hit, lifetimeSavings))

    return Result(finalPath, acceptedHits)
```

### Cliff candidate generator

```swift
func cliffCandidates(
    forYear year: Int,
    baseYear: Int,
    baselineMAGI: Double,
    filingStatus: FilingStatus,
    householdSize: Int,
    assumptions: MultiYearAssumptions
) -> [Double] {
    var candidates: [Double] = []
    let inflation = pow(1 + assumptions.cpi, Double(year - baseYear))

    // IRMAA tiers (with buffer)
    let irmaaTiers = MultiYearTaxThresholds.irmaaThresholdsBaseYear2026[filingStatus]!
    for tier in irmaaTiers {
        let projected = tier * inflation
        let target = projected - assumptions.cliffBuffer
        let delta = target - baselineMAGI
        if delta > 0 && delta < 500_000 { candidates.append(delta) }
    }

    // ACA 400% FPL (with buffer)
    if let fpl = MultiYearTaxThresholds.acaFPLBaseYear2026[householdSize] {
        let projected = fpl * inflation
        let target = projected - assumptions.cliffBuffer
        let delta = target - baselineMAGI
        if delta > 0 && delta < 500_000 { candidates.append(delta) }
    }

    // Ordinary bracket tops (NO buffer — these aren't cliffs, they're fill targets)
    let bracketTops = MultiYearTaxThresholds.ordinaryBracketTopsBaseYear2026[filingStatus]!
    for top in bracketTops {
        let projected = top * inflation
        let delta = projected - baselineMAGI
        if delta > 0 && delta < 500_000 { candidates.append(delta) }
    }

    return candidates
}
```

### Convergence and edge cases

- **Discrete candidates → exact equality** convergence test on `[Int: [LeverAction]]` works without FP drift concerns
- **`LeverAction` Equatable conformance** required for the convergence check. Plan must verify conformance exists; if not, add it (single-line synthesis if cases are simple)
- **Oscillation guard:** the 5-iteration cap is the safety net. Telemetry logs cap-hit-without-convergence cases for diagnosis
- **Non-convergent edge case:** accept iteration-5 result as best-effort; do not error out
- **Empty horizon** (`horizonYears <= 0`): return early with empty Result (preserved from current code)

## Performance estimate

Current code: 30 years × 7 candidates × 30-year projection = **6,300 year-iterations** (~300ms M1)

New design (worst case, 5 iterations × 12 candidates per year, with per-year baseline projection per inner-loop fix):

- Per year per iteration: 1 baseline projection + 12 candidate projections = 13 full-horizon projections
- Per outer iteration: 30 × 13 = 390 projections
- Each projection is 30-year: 390 × 30 = 11,700 year-iterations per outer iteration
- 5 iterations worst-case: 58,500 year-iterations (~2.7s estimated)
- 3 iterations typical convergence: ~35,000 year-iterations (~1.6s estimated)

Decision: accept the 5-iteration cap and the potential ~3s wall clock on hard scenarios in exchange for true fixed-point accuracy and correct per-year cliff candidate generation. Performance test asserts `< 3s` ceiling.

**Main-thread caveat (Plan B implementation requirement):** The engine is a pure value-in/value-out function with no `@MainActor` constraints, but a 2-3s synchronous call on the SwiftUI main thread will freeze the UI and may trigger the watchdog on older iPads. Plan B's UI layer MUST dispatch `MultiYearTaxStrategyEngine().compute(...)` off the main actor — either by marking the calling method `async` and using `Task.detached`, or by wrapping in an explicit background queue. A "Calculating strategy…" loading state is required.

## Test plan

### New tests

**Test 1 — `RothLadderHandComputedTests.swift` (new file)**

Profile:
- Couple, both age 60, retiring early
- Buckets: $1M trad, $200K Roth, $300K taxable, $50K HSA
- Expenses: $50K/yr (inflation-adjusted)
- SS at 67 (FRA): $30K/yr each
- No pension, no other income
- Horizon to age 90 (30 years)
- Filing MFJ, household size 2

Hand-computed optimal:
- Convert $80-100K/yr in years ages 60-66 (pre-SS), filling 12% bracket and draining trad before RMDs
- $0 once SS starts (would push into 22%+)
- Cumulative conversions years 0-6: ~$560K-$700K

Assertion: `sum(recommendedPath[0..7].rothConversion) in 560_000...700_000`

**Test 2 — `IRMAABufferRespectTests.swift` (new file)**

Profile:
- Couple, both age 67, Medicare-enrolled
- Baseline year-1 MAGI = $200K (just under MFJ Tier 1 cliff at $212K)
- Trad balance large enough that engine wants to convert

Assertion (MAGI-space): the resulting year-1 MAGI must NOT fall in any (T − $5K, T) range where T is an IRMAA threshold. For 2026 MFJ that means avoiding all of: ($207K, $212K), ($261K, $266K), ($329K, $334K), ($395K, $400K), ($745K, $750K).

In conversion-amount terms (relative to the $200K baseline), the engine may recommend $0, or any amount whose resulting MAGI lands at or below $207K, or any amount that lands at or above the next cliff threshold (i.e., already inside Tier 1+). It must never land MAGI in the dead zone strictly between $207K and $212K.

The test should ideally check ALL years where the engine recommends a conversion (not just year 1) — for each such year, compute final MAGI and assert it's not in any dead zone for that year's projected (CPI-adjusted) thresholds.

**Test 3 — `CliffCandidateGeneratorTests.swift` (new file)**

Unit tests on `cliffCandidates(...)`:
- Inputs: baseline MAGI = $150K MFJ, year = 2026, household size = 2, cpi = 0.03
- Expected to include: ~$57K (IRMAA Tier 1 fill: $212K − $5K − $150K)
- Expected to include: ~$109K (IRMAA Tier 2 fill: $266K − $5K − $150K)
- Expected to include: ~$56.7K (24% bracket top fill, no buffer)
- Expected to NOT include: anything for cliffs already passed (IRMAA Tier 5 if MAGI > $750K)

**Test 4 — `OptimizerPerformanceRegressionTests.swift` (new file or extend existing perf test)**

Profile: 30-year horizon, full feature set including SS, RMDs, IRMAA, ACA gating
Assertion: `optimize(...)` completes in `< 3.0` seconds wall-clock

### Existing test updates

Strategy:
1. Run full suite against new optimizer
2. For each failure, classify:
   - **(a) Optimizer now correct** — update expected values, add comment in test or commit message documenting "previously $0 due to Bug 1+2; new value $X reflects discovered ladder"
   - **(b) Genuine regression** — fix code, do not update test
3. No test deletions. Even tests that passed for the wrong reason document scenarios we care about
4. Each (a)-class update individually documented in a commit message

Expected scope: 5-15 of 933 tests need updates. Confirm during execution.

## Plan B requirements (not optional)

Two items that this spec defers to Plan B but MUST be addressed there before V2.0 ships:

1. **Async dispatch:** `MultiYearTaxStrategyEngine().compute(...)` must be called off the main actor. See "Main-thread caveat" above.

2. **`terminalLiquidationTaxRate` UI surfacing:** This single assumption controls engine aggressiveness end-to-end. At rate=0 the engine recommends nothing; at rate=37% it ruthlessly drains traditional. The 22% default is defensible but only optimal for "average MFJ heir." Power users with high-earning heirs (children inheriting under SECURE Act 10-year drain in their peak earning years) will need to dial up to 24-32%. Plan B's assumptions strip MUST include this field with a clear label like "Estimated heir tax bracket" and a tooltip explaining the leverage. Without it, the engine is a black box that can't be tuned.

3. **`cliffBuffer` UI surfacing:** Less urgent than #2 but desirable. Default $5K is planner-defensible; a "Safety margin to IRMAA/ACA cliffs" picker (options: $2K aggressive / $5K standard / $10K conservative) would give users meaningful control over the optimizer's risk tolerance.

## Future work (v2.1+)

- **Scope-E DP migration** — solves the greedy-trap class of problems natively
- **Per-hit constraint attribution rationale** — current global-savings comparison can mask localized waste; v2.1 limitation #5
- **SS taxation phase-in (Tax Torpedo) as a cliff candidate** — significant for 12%/22% bracket users
- **Capital gains 0%/15% bracket as a cliff candidate** — requires separate-buckets work
- **NIIT 3.8% threshold cliff** — currently invisible to optimizer
- **Threshold table maintenance:** IRMAA / FPL / bracket boundaries are published annually. We need a yearly update process for `MultiYearTaxThresholds.swift` constants — track as a release-checklist item starting 2027
