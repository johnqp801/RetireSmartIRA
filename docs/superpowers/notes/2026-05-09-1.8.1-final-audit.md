## Findings worth flagging to John

1. **Item #12 is documentation-only.** The spec asked for "child/children grammar" but the codebase is a single-heir model with consistent "Primary Heir's" labeling. The commit `95ab35e` is a comment update only. If Ron intended a stronger model change (e.g. multi-heir support), that did NOT ship — it's correctly out of scope, but worth confirming Ron's expectation aligns with "we already say 'Primary Heir' so this is fine."
2. **VA Disability scope expansion** added a new public-API enum case (`IncomeType.vaDisability`). Any persisted user data referencing income types serializes the enum's raw value `"VA Disability"`. New install / new income source path is exercised by tests; existing serialized profiles without this case are unaffected (additive enum). No migration needed, but it's a permanent surface area increase.
3. **Backport provenance is preserved.** The V2.0-ACA and V2.0-Medicare fixes carry inline comments citing V2.0 commit `f323da8`. This is the right pattern from the V2.0 archaeology lessons — when V2.0 ships its own fix path, future maintainers will know these were synced.
4. **Item #11 chip wiring:** Commit message lists 5 input + 5 analysis surfaces but the diff stat shows only 3 modified files (`AccountsView`, `DashboardView`, `IncomeSourcesView`) plus the new `TabPurposeChip.swift`. This is consistent if Settings/Social-Security/Tax-Planning/Quarterly-Tax/State-Comparison/RMD-Calculator chip-host points are inside the same files or a shared toolbar — worth a 30-second visual smoke check on a real device that all 10 named surfaces actually display the chip, since the test file is only 28 lines (purpose-mapping unit test, not an integration test).
