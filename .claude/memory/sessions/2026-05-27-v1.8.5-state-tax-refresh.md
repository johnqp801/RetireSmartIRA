## TriSTAR multi-LLM review — the MO catch

The Bucket 2 multi-LLM review (TriSTAR source #4) sent the diff to ChatGPT and Gemini independently. Results:

- **Gemini:** 10/10 MATCH
- **ChatGPT:** 9/10 MATCH + flagged Missouri threshold mismatch

ChatGPT's flag was correct. The Bucket 2 research agent had pulled MO thresholds from the MO DOR "year-changes" page, which publishes the prior tax year's schedule (TY 2025, for returns filed April 2026). The actual TY 2026 thresholds are inflation-bumped +2.67% per MO § 143.011 CPI indexing. Direct verification against the MO DOR 2026 Withholding Tax Formula PDF (page 13, Annual Payroll column) confirmed the flag. Engine corrected before ship in commit `897d7ef`.

This is exactly the failure mode TriSTAR source #4 is designed to catch — a primary source that *looks* current but isn't.

The methodology brief features this story as evidence that the verification process works.

## Lessons learned (for next release cycle)

### Engineering
- **Primary-source pages can be year-mislabeled.** The MO DOR "year-changes" page shows the schedule for returns filed in the current calendar year, which is the PRIOR tax year. Always verify against the withholding formula PDF for TY 2026 numbers.
- **TriSTAR source #4 (multi-LLM independent review) earns its keep.** ChatGPT's MO catch would not have been found by a single reviewer or single agent. The independence is the whole point.
- **Vermont-style "TY 2026 PDF not machine-readable" is a known recurring problem.** TY 2025 primary-source values as a conservative under-bracket floor is the Path 1 fallback.
- **Test expectations tied to bracket math need to be hunted down explicitly.** Two NY MFJ scenarios (B and D) needed test-expected updates after the NY bottom-5 rate cuts. Scenario D was missed in the initial sweep and caught only when ⌘U flagged failures.

### Process
- **Path 1 (TY 2026 actuals where published, latest TY 2025 elsewhere, all primary-sourced) is the right framework.** It's defensible, honest, and produces a brief like the one we shipped today.
- **Multi-LLM review needs a structured payload format.** The bucket2 payload format (intro + per-state code + primary source URL + expected MATCH/MISMATCH grid) is the template going forward.

### App Store Connect
- **"Developer Rejected" hides the + button for that platform.** Edit the rejected version's version number directly to re-purpose the slot for the next submission.
- **"Newer Build Available" warning means the old build is still attached.** Remove it from the Build section and attach the new one.
- **macOS reviews are running slower than iOS** as of 2026-05.

### Methodology brief
- **Don't lead with "Accuracy."** Use "Methodology." Accuracy invites a standard that planning software shouldn't pretend to meet.
- **Drop "100%" everywhere.** Even with primary sources, modeling decisions exist.
- **De-emphasize LLM review in press-facing materials.** It's a powerful internal tool but reads as "an AI checks our tax law" to skeptical external readers.
- **Inventory math must tie cleanly.** A skeptical reader will count.
