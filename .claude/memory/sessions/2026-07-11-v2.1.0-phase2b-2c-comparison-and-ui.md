## Phase 2c — the SwiftUI UI (brainstorm → spec → build → visual pass)
Brainstormed the UI (visual companion used for layout decisions), wrote + committed a frozen design spec, then built via SDD (7 tasks) + hardened one layout issue found in the live visual pass.

**Brainstorm decisions (locked with the user):**
1. Scope = approach picker + three-way comparison + consequence display + a LIGHT giving refinement (surface the already-seeded charitable amount; no new giving data model). Discovered the giving is ALREADY wired end-to-end via `MultiYearInputAdapter` from `dataManager.scenarioTotalCharitable`.
2. IA: the three-way comparison REPLACES the two-way `PlanComparisonView` (model kept — CPA briefing + TaxImpactChart still use it).
3. Comparison layout: three-column metric table + consequence strip + flags (user chose "A · table-centric" over a "hero" layout).
4. Approach picker: segmented control + INLINE bracket/tier chips with baseline-above-target status (user chose "A · inline chips" over dropdown/stepper).
5. Reach: the selected approach becomes the **ACTIVE PLAN everywhere** (charts/summary/ladder/CPA), not just a comparison lens.
6. Year-1 rule: mutual exclusion — editing Year-1 reverts a deterministic approach to the objective optimizer.

**THE naming/positioning decision (also in decisions/log.md 2026-07-11):** **NO "Recommended" anywhere — the app does not make recommendations.** The anchor column + picker segment 1 are labeled by the objective the engine actually optimized, dynamically on `selectedHeirWeight > 0`: **"Minimize lifetime tax"** (legacy off) / **"Optimize tax + legacy"** (legacy on). User rejected "Recommended" (advice framing) AND plain "minimize lifetime tax" (inaccurate when the legacy-aware objective is in force). This supersedes the frozen 2b spec's "Recommended plan" label while preserving its reasoning.
