## TY2023 federal validation unlock

- New `tools/taxsim-refresh/` Swift package that POSTs 20 retirement scenarios to NBER's TAXSIM-35 endpoint and writes responses to `RetireSmartIRATests/Fixtures/taxsim-expected.json`.
- New `RetireSmartIRATests/TaxsimOracleTests.swift` reads fixtures offline, evaluates each scenario under TY2023 constants (via `TaxCalculationEngine.withConfig(forYear: 2023)`), and asserts federal liability within $200 + state directional consistency.
- `tax-2023.json` resource: all values verified from IRS Rev. Proc. 2022-38 (read the actual PDF, transcribed Section 1(j) brackets, Section 3.11 AMT, Section 3.15 std deduction, etc.).
- **Latent bug found and fixed during unlock**: `DataManager` had `static let default2026Brackets = TaxCalculationEngine.default2026Brackets` and 15 similar lines ŌĆö class-load-time snapshots. That meant `loadConfig(forYear:)` was effectively a no-op for anything reading through DataManager. Converted all 16 to computed `static var`. This was a real production-affecting bug, not just test infrastructure.

Result: federal validation went from 0/20 (pre-unlock) to 20/20 within $200 (post-unlock).

## LLM-audit follow-up ŌĆö the part that mattered most

After the 5 primary fixes landed, dispatched an independent review through ChatGPT + Gemini using a structured prompt at `.claude/memory/drafts/2026-05-20-v1.8.4-engine-review-payload.md`. Same prompt to both, separately. Each returned MATCH or MISMATCH per state against the quoted primary source.

**Bugs both models flagged:**

1. **NY missing age 59┬Į gate** ŌĆö statute requires it; engine had no age gate on NY's exclusion. Pre-fix any age could claim the $20K exclusion.
2. **GA separate caps** ŌĆö statute is a single combined retirement-income cap; engine applied $65K to pension AND $65K to IRA separately. Manifests when user has both income types.

**Bug Gemini caught alone:**

3. **Per-individual multiplier used hardcoded `>= 59`** ŌĆö for states with higher age thresholds (GA 62-64 or 65+), under-62 spouses still enabled cap doubling. Example: GA MFJ primary 60 + spouse 65 was wrongly doubling the $65K cap to $130K.

**One Gemini false positive** correctly identified and rejected: it claimed NY also needs a shared cap. NY IT-201 has separate pension and IRA worksheets ŌĆö the existing separate-cap encoding is correct.

All three real bugs fixed in commit `0f2e6b1`. 4 new regression tests pin them.

**Lesson**: The TAXSIM oracle, the property tests, and the 1,200-test suite all passed BEFORE the LLM review. Five independent verification sources are not five layers when only one source type can probe a given bug class. The LLM-audit step earned its keep ŌĆö and TriSTAR was written specifically because of this experience.

## Engineering decisions made today

1. **TY2023 brackets via test-only config override** (`TaxCalculationEngine.withConfig(forYear:)`). Did NOT add user-facing year selection.
2. **`exemptionAppliesPerIndividual` flag** rather than separate per-status caps. Lighter schema; covers NY and GA today.
3. **`regularExemptionMinAge` + `earlyAgeTier`** to model age-banded exemptions (GA 62-64/65+, CO 55-64/65+).
4. **`pensionAndIRAShareSingleCap` flag** for CO and GA where one cap covers both income types.
5. **Per-individual multiplier qualifies each spouse independently against the state's age threshold**, not a hardcoded 59. Gemini caught this.
6. **NJ filing-status-aware caps and AGI phaseout deferred** to v1.8.5+ with explicit TODOs in code. The age-62 gate is the structurally most important piece.
7. **`stateTaxBreakdown` mirror updated in lockstep** with `applyRetirementExemptions`. `StateTaxBreakdownTests` pins consistency.

## Process notes ŌĆö what worked

1. **Test-first discipline.** Each state finding: write failing test citing the statute ŌåÆ engine/data change ŌåÆ make it pass ŌåÆ run state suite ŌåÆ run full suite. Cleaner audit trail than "fix and add a test later."
2. **Primary-source verification BEFORE coding.** Read the actual IRS Rev. Proc. PDF, the MD Comptroller TB-51 PDF, the GA Code, the NJ statute, the CO regulations. Pasted screenshots into the conversation (user did this manually when WebFetch couldn't parse PDFs). Cited every value back to the source in commit messages and code comments.
3. **LLM-audit-of-LLM-code surfaces real bugs.** This is the main empirical finding of the day. Both ChatGPT and Gemini converging on the same MISMATCH is a high-confidence signal.
4. **Filed regression tests that document why the test exists** ŌĆö every test has the statute citation and the bug it pins, so future readers (Claude or human) know what's protected.
5. **Updated `What's New` flagged a meta-error** ŌĆö user's initial draft inverted the Maryland line ("included in" vs "excluded from" the pension subtraction). That would have shipped wrong copy to the App Store. Caught by re-reading against the source.

## Process notes ŌĆö what would improve

1. **Mirror-helper drift is a recurring trap.** Every time we update `applyRetirementExemptions`, we have to remember to update the mirror in `DataManager.stateTaxBreakdown`. Today it tripped us once (StateTaxBreakdownTests caught it). Plan: unify the two paths in a future release.
2. **TAXSIM-35 state coverage is 2020-base with inflation.** Means its state numbers are approximate. The TAXSIM oracle is a federal cross-check first, state directional check second. The LLM-audit step does heavier lifting on state correctness.
3. **The `static let` snapshot bug** existed for a long time and would have prevented future year-config switching in production too. Found only because TAXSIM forced a year-config swap in tests. Lesson: don't snapshot delegating constants at class-load time.
4. **NJ's AGI phaseout and filing-status caps remain TODOs.** Real user-facing inaccuracy for NJ retirees with $100K-$150K AGI. Should be in v1.8.5.

### Roth withholding feature ŌĆö design (locked in with the user)

Six design decisions before coding:
1. **UI placement:** on the Roth Conversion view, next to the slider (a).
2. **UI form:** Picker (b) ŌĆö "Pay conversion tax from: [Outside Money / The Conversion Itself]" ŌĆö more explicit than a toggle.
3. **Tax computation:** user enters gross, engine shows net (a). Simpler than iterative gross-to-net solving.
4. **PA Ans 274 interaction:** honor the "full balance" rule ŌĆö withheld portion becomes PA-taxable as distribution (a).
5. **Heir-comparison transparency:** show a note in the heir card explaining the net Roth amount (rather than silent input change).
6. **Under-59┬Į:** warning banner, do NOT compute the 10% additional tax (b). The IRC ┬¦ 72(t) interaction is documented but not modeled.
