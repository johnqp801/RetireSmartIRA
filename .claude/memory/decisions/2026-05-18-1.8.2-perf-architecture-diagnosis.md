# 1.8.2 Scenarios Tab Hang тАФ Architectural Perf Diagnosis

**Date:** 2026-05-18
**Branch under analysis:** `1.8.2/incremental` (HEAD `ff36d22`), worktree `/Users/johnurban/Projects/RetireSmartIRA/.worktrees/1.8.2-incremental`
**Symptom:** 2.27тАУ3.29s hang switching to the Scenarios tab on macOS and iPadOS; white-screen on iPad scene restore.

---

## 1. Executive summary

The Scenarios-tab hang is **architectural**, not a hot-spot bug. Three reinforcing causes:

1. **DataManager fans out its `objectWillChange` to every observer of `@EnvironmentObject`**, and aggressively re-broadcasts changes from all 8 child managers (87 `@Published` properties total) via Combine sinks. Any state mutation invalidates all 22 `@EnvironmentObject`-using views in the app, including the entire 4,477-line `TaxPlanningView`.
2. **`TaxPlanningView` is a single struct with one giant `body` graph** (27 view-builder properties, 277 `dataManager.X` reads, 42 `if dataManager.X` conditionals, 11 `AnyView(...)` wrappers across the wide/compact body sub-groups). `AnyView` defeats SwiftUI's structural diffing тАФ when DataManager invalidates, SwiftUI cannot prune subtrees and re-evaluates the whole body, which fans out into a `CA::Transaction::commit` layout cascade (~520ms per pass per the trace).
3. **Memoization at the engine layer (today's `EngineMemoCache`) addresses arithmetic cost only, not the layout / view-invalidation cost.** It cut Swift-side compute. The 2.27s residual is QuartzCore/AppKit layout, which is upstream of memoization.

**Confidence: HIGH** on causes (1) and (2) тАФ direct evidence in source. **MEDIUM** on causes (3) тАФ interpretation of the Time Profiler ratio (520ms layout vs ~80ms Swift) plus the diagnostic-branch finding that disabling Phase 1-3 mounts did not move the needle.

**Recommended path:** Migrate the observation layer to the `@Observable` macro (iOS 17 / macOS 14 Observation framework) **and** decompose `TaxPlanningView` into structurally-typed child Views (drop the `AnyView(Group {...})` shim pattern). These are independent and stackable. Detail in ┬з5.

## 2. Current architecture (what was found)

### 2.1 Observation framework

- **Still on legacy `ObservableObject` + `@Published`.** Zero uses of `@Observable`.
- 87 `@Published` properties across 12 files; 9 distinct `ObservableObject` classes.
- 33 `@EnvironmentObject` declarations across 22 view files. 2 `@StateObject`, 1 `@ObservedObject` тАФ i.e. environment injection is the dominant pattern.

## 3. Root cause analysis

### 3.1 Why 2.27s hangs (residual after memoization)

Tab switch flow:
1. SwiftUI tears down the previous tab's view tree and mounts `TaxPlanningView`.
2. `TaxPlanningView.body` is evaluated. Because the top of the body is `Group { if isWideLayout { wideBody } else { compactBody } }` and `wideBody` / `compactBody` immediately compose `AnyView(Group {...})` blocks, SwiftUI loses type structure at those seams. It cannot diff against a previous render (there isn't one yet тАФ this is a fresh mount), but every subsequent invalidation in this session forces re-eval of the full subtree under the `AnyView`.
3. The body reads ~277 properties off `dataManager`, fanning into the engine pipelines.
4. SwiftUI runs measurement / layout passes for the resulting tree. The Time Profiler shows ~520ms in `CA::Transaction::commit` and `NSView._layoutSubtree`, ~80ms in Swift body code тАФ meaning the AppKit/QuartzCore layout commit is the bottleneck, not the Swift evaluation.
5. SwiftUI does this **multiple times** during a tab transition (Apple's tab-switch animation runs at least one extra layout pass; `availableWidth` env mutations and `onAppear` writes back to `@State` queue another). Each pass repeats the full layout commit.

Total: ~3-4 full layout commits ├Ч ~500ms each = the observed 2.27s.

### 3.2 Why memoization helped ~30% but not ~100%

`EngineMemoCache` (DataManager+Memo.swift) caches 8 engine outputs against a comprehensive `engineInputsHash`. This pays off on the **second through Nth body invocation in the same transition** тАФ the engine pipelines (TaxCalculationEngine, ACA, IRMAA) get short-circuited. But:

1. The **first** read of each cached property in the transition still pays full cost (cache cold).
2. **Layout cost is independent of compute cost.** Even if every `dataManager.X` read returned in 0ns, SwiftUI would still execute `CA::Transaction::commit` for the entire view tree because the tree itself is large and AnyView-wrapped.
3. Hashing 70+ properties to compute `engineInputsHash` costs ~tens of microseconds per call, and it's called on every memoized access. Not free.

The 30% gain is consistent with eliminating Swift-side recompute on repeat reads while leaving the layout cost intact.

### 3.3 Why iPad and Mac both hang

Both platforms use AppKit-like layout under SwiftUI on iPadOS via `_UIHostingView` and on macOS via `NSHostingView`. The bottleneck is the **measurement and commit pass**, which behaves equivalently on both. There's nothing platform-specific in the trace; this rules out an AppKit-only quirk and points at SwiftUI itself.

### 3.4 Why disabling Phase 1-3 mounts didn't fix it (commit `e85c20c`)

Because the cost is in the **pre-existing** body тАФ specifically `scenarioSummaryCard`, `legacyImpactCard` (тЖТ `LegacyImpactView.familyWealthSection` at 29ms per body тАФ confirmed in trace), `rateBreakdownSection`, `taxImpactWaterfallChart`, `perDecisionImpact`, `strategyTipsSection`. The 7 new cards added maybe 1-3% each. Removing them stays well within measurement noise.

## 5. Recommended re-architecture plan

Three options, ordered by leverage. They can stack but are **independent** тАФ you don't need #1 to do #2.

### Option A тАФ Migrate to `@Observable` macro (Observation framework)

**Highest expected impact.** Apple designed this framework explicitly to fix the "ObservableObject invalidates every observer on every change" problem you're hitting. Views only re-render when they read a property that actually changed. The big `@Published`-fan-in tax in `DataManager` largely disappears: the explicit `objectWillChange.sink` forwarding at lines 884-905 becomes unnecessary because the macro tracks reads at the property level across object boundaries automatically.

- **Effort:** 2-4 days. Mechanical for the manager classes (drop `: ObservableObject`, drop `@Published`, add `@Observable`). Slightly more work at the view layer: `@EnvironmentObject var x: T` тЖТ `@Environment(T.self) var x` and `var x: T` becomes a property; `Binding` from a `let` requires `@Bindable`. Tests should not need changes тАФ engine math is unaffected.
- **Risk:** **Medium.** The two-way bindings (sliders, text fields) need `@Bindable` shims. Some `objectWillChange.send()` callers may exist (search showed `objectWillChange` referenced in DataManager init only тАФ but worth re-auditing). Edge case: Combine pipelines on `profile.$birthDate` (DataManager.swift:911-925) тАФ under `@Observable`, the synthesized `$birthDate` Publisher no longer exists; you'd use `withObservationTracking` or a different sync trigger.
- **Blast radius:** Touches all 9 managers + all 22 EnvironmentObject-using views. Big PR but very mechanical.
- **Rollback:** Hard once committed because the `@Published` annotations are gone. Mitigate by doing one manager at a time on a branch.
- **Can it be incremental?** Partially. The framework supports mixing; you can migrate `DataManager` first while children stay `ObservableObject`, but the win is largest when the whole graph is on it.
- **Target platforms:** Requires iOS 17 / macOS 14 minimums. Verify deployment target.

**Expected outcome:** Eliminates the "all 22 views re-render on every state change" problem. Likely to halve or better the hang on its own. The Apple docs and avanderlee benchmarks describe view-update-frequency improvements; no published number for cold tab-switch latency, but the mechanism directly maps to your symptom.

### Option B тАФ Decompose `TaxPlanningView` into typed child Views; remove `AnyView`

**Second-highest impact.** Each major section (`summaryCard`, `scenarioSummaryCard`, `rateBreakdownSection`, `legacyImpactCard`, `wideLeftGroupA`, etc.) becomes a `struct XxxSection: View` that takes the data it needs as init args (or just an `@EnvironmentObject` if you accept that cost). Replace `AnyView(Group {...})` with `@ViewBuilder` properties or extracted structs. Structural typing restores SwiftUI's ability to diff subtrees independently.

- **Effort:** 3-5 days. Largely lift-and-shift; risk of accidentally changing behavior is low because the structure already exists as `private var` view-builders. The complication: shared `@State` (`stockPurchasePriceText`, sheet flags) needs to be lifted into the parent or moved into bindings into the children.
- **Risk:** **LowтАУmedium.** Visual diffing not affected. Refactor risk is loose-binding sheet presentations and `.onChange` handlers that currently live at the body root.
- **Blast radius:** TaxPlanningView only. Other views are unaffected.
- **Rollback:** Easy тАФ revert.
- **Incremental?** Yes. Extract one section at a time; each extraction makes that subtree diffable in isolation.
- **Combines with A?** Yes; doing B after A is even better because extracted views with `@Environment` will only re-render on their narrowed reads.

**Expected outcome:** Smaller per-pass layout commits because SwiftUI can prune unchanged subtrees. Probably less impact than Option A in steady-state but more impact on **cold tab-switch** (the actual reported symptom) because the initial measurement pass shrinks per-section. Estimate 20-40% off the residual.

### Option C тАФ Push DataManager computed-property memoization further + collapse the dependency graph

**Lowest leverage now that the obvious 8 are cached.** Look for opportunities to:
- Cache `scenarioGrossIncome`, `standardDeductionAmount`, `effectiveDeductionAmount`, `scenarioTaxableIncome` (the cycle-shaped subgraph at the heart of every section).
- Replace transitive computed-property chains with a single "compute scenario result" struct produced once per inputs-hash and read N times (move from many memos to one memo of a struct).
- Particularly: precompute a `ScenarioAnalysisSnapshot` value type once at the top of `body` (or as a `@State`-stored object updated on `onChange`) and pass it through; sub-views read fields off the struct, not off DataManager.

- **Effort:** 1-2 days.
- **Risk:** Low тАФ pure refactor of compute. Test coverage on the engine math is solid (1100+ tests).
- **Blast radius:** DataManager + sub-views that consume the snapshot.
- **Incremental?** Yes; coarsen one chain at a time.
- **Limit:** Even if all engine compute is zero, the layout cost (the dominant share per the trace) doesn't move. So this stacks usefully on A/B but is not a fix on its own.

### Suggested ordering

1. **Spike Option A on a throwaway branch first** (4-8h spike): migrate ONLY `ScenarioStateManager` and `ProfileManager` (the two biggest publishers) to `@Observable`, leave others alone, verify it compiles, measure hang. This is the highest-leverage hypothesis test before committing to a full migration.
2. If the spike shows large wins тЖТ do the full Option A migration as the primary fix.
3. Layer Option B on top тАФ independent of A, cleans up TaxPlanningView for long-term maintainability and adds incremental gains.
4. Option C only if A+B aren't enough.

## 6. Things to verify before committing to a plan

1. **Deployment target.** `@Observable` requires iOS 17 / macOS 14 / iPadOS 17. App Store positioning has been "macOS + iOS"; check Info.plist / `MARKETING_VERSION`. If the floor is iOS 16, Option A is blocked.
2. **Combine pipelines on profile dates.** `DataManager.swift:911-925` uses `profile.$birthDate`. Migration to `@Observable` removes the synthesized publisher. Need to confirm whether `syncSSToIncomeSources()` can be triggered by `withObservationTracking { ... } onChange: { ... }` or moved to a different lifecycle hook.
3. **The actual `LegacyImpactView` and other heavy sub-views** also use `@EnvironmentObject` тАФ the win on Option A depends on them also migrating. Plan should include their view-layer migration.
4. **Is `TaxPlanningView` re-mounted from scratch on every tab switch, or is the state cached?** The white-screen-on-iPad scene restore symptom suggests fresh mount. If true, lazy-load of sub-sections matters more (Option B + use real `LazyVStack` without `AnyView` wrapper).
5. **Are the 11 `AnyView`s actually load-bearing?** Most of them wrap `Group` for what looks like a stylistic choice; a few wrap `VStack(spacing: 0)`. The Group ones can be replaced trivially with `@ViewBuilder` properties. The `VStack` ones need a closer look тАФ they're at lines 1871/2148/2201/2260/2922 which I did not inspect. Some may have been added to work around the compiler's "the compiler is unable to type-check this expression in reasonable time" error on giant bodies, which itself is a symptom of the same single-struct problem.
6. **Does `engineInputsHash` itself dominate?** It hashes ~70 properties on every memoized read. For body calls that hit 20+ memoized accessors, that's 1,400+ Hasher.combine calls per body invocation. Worth measuring in isolation; if non-trivial, switching to a generation counter (`@Published var stateVersion: Int`, incremented on any change) is cheaper than a content hash.
